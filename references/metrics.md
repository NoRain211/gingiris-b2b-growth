# 第五部分：数据驱动增长闭环

## 1. 北极星指标

**定义：** 最能衡量产品为客户创造核心价值的那一个指标。

**要求：**
- 全公司（产品、研发、市场、销售）围绕它对齐
- 直接反映客户价值——不是收入，不是虚荣指标
- 可按周或按天衡量
- 产品改动后能在合理时间内看到变化
- 能预测长期业务健康度（收入、留存）

### 如何选择北极星指标

**流程：**
1. 列出客户从产品中获得真实价值的所有时刻。
2. 找到最能代表这种价值交换的动作或结果。
3. 验证该指标的变化与过去 6-12 个月的收入和留存变化相关。
4. 压力测试："如果这个指标翻倍但收入没动，你能接受吗？"如果能，换一个指标。
5. 争取每个团队负责人的认同。如果研发不信这个指标，埋点就不会做好。

**常见误区：**
- 拿收入当北极星。收入是结果。北极星应该是驱动收入的输入。
- 选了一个你动不了的指标。"总注册用户数"只增不减——它不反映产品健康度。
- 选了太抽象的东西。"客户满意度"没法衡量。"每周完成的任务数"可以。
- 每个季度换一次。选定一个，至少坚持一年。总换就永远不会建立起改进它的肌肉记忆。
- 让 CEO 一个人拍板。最懂产品和数据的人需要参与决策。

### 各 SaaS 类目的北极星指标示例

| 类目 | 产品示例 | 北极星指标 | 为什么有效 |
|:---|:---|:---|:---|
| AI 视频工具 | Synthesia, Loom AI | 每周导出的视频数 | 导出 = 用户获得了足够价值并愿意分享输出 |
| AI 会议助手 | Otter, Fireflies | 每周生成的会议纪要数 | 生成纪要 = 交付了核心价值 |
| 协作平台 | Notion, Slack | 每周有活跃协作的团队数 | 团队而非个人——协作才是价值 |
| CRM | HubSpot, Salesforce | 每周更新的商机数 | 商机活动 = 销售团队在获取价值 |
| 开发工具 / API | Stripe, Twilio | 每周生产环境 API 调用量 | 生产环境用量 = 真实集成，真实价值 |
| 项目管理 | Linear, Asana | 每周完成的任务数 | 是"完成"而非"创建"——完成才是价值 |
| 分析 / BI | Amplitude, Metabase | 每周非分析师用户执行的查询数 | 数据分析民主化 = 平台粘性 |
| 安全 | Snyk, Datadog | 每周解决的漏洞/问题数 | 重在"解决"而非"发现"——修复才是价值 |
| 电子签名 / 法务 | DocuSign, PandaDoc | 每周签署的文件数 | 签署 = 交易完成 |
| 客户支持 | Intercom, Zendesk | 每周解决的对话数 | 解决 = 客户问题被处理 |
| 设计工具 | Figma, Canva | 每周分享给协作者的设计数 | 分享 = 设计好到可以给别人看 |
| 邮件营销 | Mailchimp, Brevo | 每周发送的打开率 >20% 的邮件活动数 | 有质量门槛：单纯发送不代表价值 |
| 数据集成 | Fivetran, Airbyte | 每周成功完成的同步数 | 同步成功 = 数据管线在交付价值 |

### 先行指标 vs 滞后指标

北极星通常是一个**滞后**指标——它告诉你已经发生了什么。你需要**先行**指标来预测和驱动它。

| 类型 | 定义 | 示例（以"每周导出的视频数"为例） |
|:---|:---|:---|
| **滞后** | 已经发生的结果 | 本周导出的视频数 |
| **先行** | 能预测滞后指标的早期信号 | 新建项目数、模板浏览量、首次渲染完成数 |

建立模型：先行指标 → 北极星 → 业务结果（收入、留存）。

### 将北极星分解为输入指标

把北极星拆解为一棵**指标树**。每个团队负责一条分支。

示例：北极星 = "每周有活跃协作的团队数"

```
每周有活跃协作的团队数
├── 本周新激活的团队（增长/市场）
│   ├── 注册 → 建团率
│   ├── 每个新团队发出的邀请数
│   └── 邀请接受率
├── 本周留存的现有团队（产品）
│   ├── D7 团队留存率
│   ├── 每团队每周使用的功能数
│   └── 跨团队协作事件数
└── 本周被唤回的团队（生命周期/CRM）
    ├── 召回邮件打开率
    ├── 活动召回率
    └── 被唤回团队的上次活跃时间
```

每个团队负责自己的分支。每周复盘指标树，定位哪条分支表现不达标。

## 2. AARRR 漏斗指标

| 阶段 | 核心问题 | 关键指标 |
|:---|:---|:---|
| **获客 (Acquisition)** | 用户从哪来？ | 各渠道 UV、注册数、分渠道 CAC |
| **激活 (Activation)** | 用户是否体验到核心价值？ | "Aha Moment"转化率、引导完成率、Time to Value |
| **留存 (Retention)** | 用户会回来吗？ | D1/D7/D30 留存率、功能留存率、周活跃率 |
| **收入 (Revenue)** | 用户会付费吗？ | 免费转付费率、ARPU、LTV、扩展收入 |
| **推荐 (Referral)** | 用户会推荐别人吗？ | NPS、K-factor、推荐转化率 |

### B2B SaaS 各阶段基准值

以下是 B2B SaaS 的中位数和上四分位数。具体数值因客群、价格带和 GTM 模式而异。

#### 获客

| 指标 | 中位数 | 上四分位 | 备注 |
|:---|:---|:---|:---|
| 官网 → 注册转化率 | 2-5% | 8-12% | PLG 模式；Sales-led 会更低 |
| 综合 CAC（SMB） | $200-500 | <$150 | 全口径：人力 + 工具 + 广告 |
| 综合 CAC（中型客户） | $2,000-5,000 | <$1,500 | |
| 综合 CAC（企业客户） | $10,000-30,000 | <$8,000 | 长销售周期会推高此值 |
| 自然流量占注册比例 | 30-40% | >60% | 高自然流量 = 复利增长 |

#### 激活

| 指标 | 中位数 | 上四分位 | 备注 |
|:---|:---|:---|:---|
| 注册 → 激活 | 20-30% | 40-60% | "激活" = 完成 Aha Moment |
| Time to Value | 1-3 天 | <1 小时 | 最好的 PLG 产品几分钟内交付价值 |
| 引导完成率 | 40-60% | >80% | 每多一步流失 20% |

#### 留存

| 指标 | 中位数 | 上四分位 | 备注 |
|:---|:---|:---|:---|
| 第 1 个月留存 | 40-50% | >60% | 是用户留存，不是客户 logo 留存——账户级别不同 |
| 第 3 个月留存 | 25-35% | >45% | 多数产品在这里流失最严重 |
| 第 12 个月留存 | 15-25% | >35% | 低于 15% 说明是产品问题，不是增长问题 |
| Logo 留存（年度） | 85-90% | >95% | B2B 合同有助于此 |
| Net Dollar Retention | 100-110% | >130% | 最优秀的公司扩展速度快于流失速度 |

#### 收入

| 指标 | 中位数 | 上四分位 | 备注 |
|:---|:---|:---|:---|
| 免费 → 付费转化 | 2-5% | 7-12% | Freemium 模式；Free Trial 转化更高（15-25%） |
| 试用 → 付费（无需绑卡） | 10-15% | 20-30% | 要求绑卡可以推高到 40-60%，但注册量会降低 |
| 月付 → 年付转化 | 15-25% | >40% | 年付折扣 15-20% 以改善现金流 |
| ARPU 年增长 | 5-10% | >20% | 定价优化和扩展收入 |

#### 推荐

| 指标 | 中位数 | 上四分位 | 备注 |
|:---|:---|:---|:---|
| NPS | 30-40 | >50 | B2B SaaS 均值低于 B2C |
| K-factor | 0.1-0.3 | >0.5 | K > 1 才算病毒传播；几乎没有 B2B 产品能做到 |
| 来自推荐的新用户占比 | 10-20% | >35% | 低 CAC 渠道；值得投入降低推荐门槛 |

### 识别和修复瓶颈

漏斗中最大的瓶颈是你损失绝对用户数最多的环节（不是比例）。10,000 用户中损失 50% 比 100 用户中损失 70% 严重得多。

**获客瓶颈的表现：**
- CAC 相对于 LTV 偏高
- 超过 60% 的注册依赖付费渠道
- 目标市场中品牌知名度低

**解法：** 投入内容/SEO（6-12 个月见效），搭建推荐循环，优化落地页，尝试渠道多元化。不要只是在同一个付费渠道加预算。

**激活瓶颈的表现：**
- 注册 → Aha Moment 转化率低于 25%
- Time to Value 超过 24 小时
- 用户注册后从未完成引导

**解法：** 缩短引导流程（删步骤、延后账号设置），预填示例数据，基于触发条件发送引导邮件，通过对比留存用户和流失用户的首次会话行为来找到真正的 Aha Moment。

**留存瓶颈的表现：**
- 第 1 个月到第 3 个月之间的急剧下降
- 各 cohort 的功能使用量逐周递减
- 用户有登录但没有执行核心动作

**解法：** 生命周期邮件促活、App 内提示低使用率功能、建立习惯（通知、摘要）、和流失用户对话——真的打电话给他们。

**收入瓶颈的表现：**
- 免费用户永远不转付费
- 扩展收入低
- 对涨价高度敏感

**解法：** 收紧免费版限制（用量限制比功能门控更有效），引入按用量计费，在自然摩擦点设置升级提示，提供年付折扣。

**推荐瓶颈的表现：**
- 留存不错但 NPS 低
- 用户不知道推荐计划的存在
- 推荐流程步骤太多

**解法：** 把分享内嵌到产品中（共享链接、带品牌水印的导出），双向激励（推荐人和被推荐人都有奖励），让推荐动作出现在自然时机（成功操作之后，而不是在设置页面）。

### 各阶段看板示例

每个阶段都应该有自己的看板，包含 3-5 个指标。以下是起点参考。

**获客看板：**
- 每日/每周注册数（总计及分渠道）
- 分渠道 CAC（按月更新）
- 官网流量 → 注册转化率
- 转化率 Top 10 落地页
- 渠道结构趋势（自然 vs 付费 vs 推荐 vs 直接访问）

**激活看板：**
- 注册 → Aha Moment 漏斗（逐步流失情况）
- Time to Value 分布（直方图）
- 分步骤引导完成率
- 分获客渠道的激活率（哪些渠道带来高意向用户？）
- 分注册 cohort 的激活率（周环比趋势）

**留存看板：**
- Cohort 留存表（行 = 注册周，列 = 注册后周数）
- 周活跃用户趋势
- 功能留存率（每周使用各核心功能的活跃用户占比）
- 流失率趋势（按月）
- 复活率（曾流失又回来的用户）

**收入看板：**
- MRR 瀑布图（新增 + 扩展 - 缩减 - 流失）
- ARPU 趋势（按月）
- 免费转付费率趋势
- 分渠道 LTV/CAC 比
- 分套餐收入分布

**推荐看板：**
- 每周推荐邀请数
- 推荐转化率（被邀请 → 注册 → 激活）
- K-factor 趋势
- NPS 趋势（按季度）
- 来自推荐的新注册占比

## 3. 关键指标定义

### 获客指标

| 指标 | 定义 | 公式 | 健康范围 |
|:---|:---|:---|:---|
| **CAC** | 客户获取成本 | 销售与市场总支出 / 新增客户数 | 取决于 ACV |
| **LTV** | 客户生命周期价值 | ARPU × 毛利率 × (1 / 月流失率) | >3× CAC |
| **LTV/CAC** | 获客效率 | LTV / CAC | 3:1 到 5:1 |
| **CAC Payback** | 收回获客成本需要的月数 | CAC / (ARPU × 毛利率) | <18 个月（SMB：<12 个月） |

**关于 LTV 计算的说明：** 简单公式（ARPU × 生命周期）会高估 LTV，因为它忽略了毛利率。如果你的 ARPU 是 $100/月，但毛利率只有 70%，那你每月的贡献是 $70 而不是 $100。务必使用毛利率调整后的 LTV。

**关于 LTV/CAC 的说明：** 比值超过 5:1 通常意味着你在增长上投入不足，而不是效率高。你在浪费机会。

### 激活指标

| 指标 | 定义 | 如何找到它 |
|:---|:---|:---|
| **Aha Moment 转化率** | 完成核心价值体验的用户占比 | 对比首次会话中留存用户和流失用户的行为差异 |
| **Time to Value** | 从注册到首次获得价值的时间 | 从创建账户到 Aha Moment 事件的中位时间 |
| **引导完成率** | 完成所有引导步骤的用户占比 | 对引导流程做漏斗分析 |

### 留存指标

| 指标 | 公式 | 备注 |
|:---|:---|:---|
| **D1 留存** | 第 2 天活跃用户 / 第 1 天注册用户 | 更适用于日频使用的产品 |
| **D7 留存** | 第 8 天活跃用户 / 第 1 天注册用户 | 大多数 SaaS 的关键拐点 |
| **D30 留存** | 第 31 天活跃用户 / 第 1 天注册用户 | 长期留存的强预测指标 |
| **Logo 留存** | (期初客户数 - 流失客户数) / 期初客户数 | 按月或按年 |
| **Net Dollar Retention** | (期初 MRR + 扩展 - 缩减 - 流失) / 期初 MRR | 衡量 PMF 最好的单一指标 |

### 收入指标

| 指标 | 定义 | 公式 |
|:---|:---|:---|
| **MRR** | 月度经常性收入 | 所有经常性订阅收入之和（年合同折算为月） |
| **ARR** | 年度经常性收入 | MRR × 12 |
| **ARPU** | 每用户平均收入 | MRR / 付费客户总数 |
| **ACV** | 年度合同价值 | 新签合同的平均年度价值 |
| **NRR / NDR** | 净收入留存 / 净金额留存 | 见下文详述 |

### 进阶指标

#### 回本周期

**是什么：** 收回一个客户获取成本所需的月数。

**公式：** CAC / (ARPU × 毛利率)

**为什么比 LTV/CAC 更重要：**
- LTV 是预估值，假设未来和过去一样。回本周期用的是当前的真实数据。
- 现金流是关键。LTV/CAC 5:1 没意义，如果回本要 36 个月而你 18 个月后就没钱了。
- 可操作。你可以通过提高 ARPU（定价）、降低 CAC（效率）或改善毛利率（基础设施成本）来缩短回本周期。

**基准值：**
| 客群 | 良好 | 优秀 |
|:---|:---|:---|
| SMB SaaS | <12 个月 | <6 个月 |
| 中型客户 SaaS | <18 个月 | <12 个月 |
| 企业客户 SaaS | <24 个月 | <18 个月 |

如果回本超过 24 个月，你就是在烧钱换增长。没有风投支撑的话行不通。

#### Quick Ratio

**是什么：** 衡量 MRR 增长效率。每损失一美元收入，你能产生多少新收入？

**公式：** (New MRR + Expansion MRR) / (Churned MRR + Contraction MRR)

**基准值：**
| Quick Ratio | 解读 |
|:---|:---|
| <1 | 在萎缩。停下来修流失。 |
| 1-2 | 增长缓慢。流失吃掉了大部分新收入。 |
| 2-4 | 健康增长。你在弥补损失的同时还有盈余。 |
| >4 | 优秀。增长效率很高。 |

Quick Ratio 为 4 意味着每流失 $1，你能带来 $4 的新增 + 扩展收入。最优秀的 B2B SaaS 公司运营在 4 以上。

**注意：** 如果高 Quick Ratio 主要靠 New MRR（大量新客户）支撑，不如靠 Expansion MRR（老客户加购）来得持久。扩展收入更便宜、更具粘性。

#### 毛利率

**是什么：** 收入减去交付服务成本后，占收入的百分比。

**公式：** (收入 - COGS) / 收入

**SaaS 的 COGS 包括哪些：**
- 托管/基础设施成本（AWS, GCP, Azure）
- 第三方 API 成本（OpenAI, Twilio 等）
- 客服团队薪资（直接支持，不含客户成功经理）
- DevOps / SRE 薪资（有争议——有些公司算，有些不算）
- 支付处理费（Stripe 的 2.9% + $0.30）

**不算 COGS 的：**
- 研发薪资（R&D）
- 销售和市场费用
- 行政管理费（房租、法务、财务）

**基准值：**
| 类型 | 毛利率 |
|:---|:---|
| 传统 SaaS | 75-85% |
| AI/ML 重度 SaaS | 50-70%（GPU 成本不低） |
| 按用量计费的 SaaS | 60-75% |
| 头部水平 | >80% |

**为什么重要：** 毛利率决定了每一元收入中有多少可以用于增长、研发和利润。80% 毛利率的公司可以承受 12 个月的回本周期，50% 毛利率的公司不行——需要 6 个月以内回本。

AI 重度 SaaS 公司需要密切关注。每次 LLM API 调用都在侵蚀你的利润。如果 AI 功能的运行成本超过了它带来的增量收入，你的问题是毛利率问题——只是被包装成了增长故事。

#### 40 法则

**是什么：** 增长与盈利的综合衡量。核心思路：一家 SaaS 公司要么增长快，要么盈利，最好两者兼具。

**公式：** 收入增长率（YoY %）+ 自由现金流利润率（%）

**示例：**
| 公司 | 增长率 | FCF 利润率 | 40 法则得分 |
|:---|:---|:---|:---|
| 早期创业公司 | 150% | -80% | 70 ✓ |
| 成长期 | 60% | -10% | 50 ✓ |
| 成熟 SaaS | 20% | 25% | 45 ✓ |
| 困境公司 | 15% | 5% | 20 ✗ |

**基准值：**
- 低于 20：表现不佳。要么加速增长，要么削减成本。
- 20-40：尚可但不突出。
- 40+：健康。这是高质量 SaaS 的最低门槛。
- 60+：顶尖。极少数公司能持续达到。

**注意：** 有些人用 EBITDA 利润率代替 FCF 利润率。FCF 更诚实，因为它包含了资本支出和股权激励。

#### Net Dollar Retention (NDR)

**是什么：** 不算新客户，仅从现有客户身上留存和扩展了多少收入。

**公式：** (期初 MRR + Expansion MRR - Contraction MRR - Churned MRR) / 期初 MRR

**示例：** 月初有 $100K 的老客 MRR，扩展 $15K，缩减 $3K，流失 $5K。NDR = ($100K + $15K - $3K - $5K) / $100K = 107%。

**基准值：**
| NDR | 解读 |
|:---|:---|
| <90% | 严重问题。产品粘性不够或定价无法随用量增长。 |
| 90-100% | 在弥补流失但没有扩展。 |
| 100-110% | 不错。老客户在小幅增长。 |
| 110-130% | 很好。扩展收入强劲。 |
| >130% | 头部水平。Snowflake, Datadog, Twilio 级别。 |

NDR > 100% 意味着即使零新客也能增长。这是 SaaS 最强的增长引擎。

**如何提升 NDR：**
- 按用量计费（客户自然增长时自然付更多）
- 按席位计费且方便加购
- 分层功能随客户规模解锁
- 交叉销售相邻产品
- 配备专属 CSM 团队推动扩展对话

#### Magic Number

**是什么：** 衡量销售效率——每在销售市场上花一美元，能产生多少新收入。

**公式：** (本季度 ARR - 上季度 ARR) / 上季度销售与市场支出

**基准值：**
| Magic Number | 解读 |
|:---|:---|
| <0.5 | 效率低。先解决销售流程或 PMF 问题，再扩预算。 |
| 0.5-0.75 | 还行但有优化空间。 |
| 0.75-1.0 | 健康。可以开始加大投入。 |
| >1.0 | 非常高效。全力加速。 |

Magic Number 0.75 意味着在销售市场上每花 $1 就在当季带来 $0.75 的新增 ARR。由于 ARR 是持续性收入，这笔投入的长期回报远高于此。

#### Burn Multiple

**是什么：** 每产生一美元新增 ARR 需要烧掉多少现金。

**公式：** 净现金消耗 / 净新增 ARR

**基准值：**
| Burn Multiple | 解读 |
|:---|:---|
| <1x | 极好。产生的 ARR 超过消耗的现金。罕见。 |
| 1-1.5x | 很好。高效增长。 |
| 1.5-2x | 早期阶段可以接受。 |
| 2-3x | 开始贵了。投资人会开始问问题。 |
| >3x | 不可持续。你在用钱买增长。 |

Burn Multiple 是 2022 年以后 VC 越来越看重的指标。高效增长比不计代价增长重要。

#### 分渠道 CAC 回本

不要只算综合 CAC。分渠道拆，因为综合数据掩盖了巨大的差异。

**示例拆解：**

| 渠道 | CAC | 回本周期 | LTV/CAC | 结论 |
|:---|:---|:---|:---|:---|
| 自然搜索 | $80 | 2 个月 | 15:1 | 大力投入 |
| 内容营销 | $150 | 4 个月 | 8:1 | 加大投入 |
| Google Ads（品牌词） | $200 | 5 个月 | 6:1 | 保持现状 |
| Google Ads（非品牌词） | $600 | 15 个月 | 2:1 | 优化或砍掉 |
| LinkedIn Ads | $900 | 22 个月 | 1.5:1 | 除非做企业客户，否则砍掉 |
| 外呼 SDR | $1,200 | 18 个月 | 2.5:1 | 仅用于中型客户以上 |
| 会议赞助 | $2,500 | 30+ 个月 | 1:1 | 通常不值得 |

重点是：你所谓的"综合 CAC $400"毫无意义。有的渠道 $80，有的 $2,500。加倍投入便宜的，砍掉或优化贵的。

## 5. 增长会议最佳实践

### 频率
每周一次。两周一次太慢——你会失去节奏。如果每周没有足够内容讨论，说明你的实验节奏太慢了。

### 参与者
- 产品经理（掌控议程）
- 增长/市场负责人
- 研发负责人（不是整个团队——一个能代表可行性和时间评估的人）
- 数据分析师/数据负责人
- 可选：设计负责人、销售负责人（适用于 Sales-assisted PLG）

控制在 4-6 人。超过这个数就做不了决策了。

### 会议议程模板（45 分钟）

```
增长会议 — [日期]

1. 计分卡 (5 min)
   复盘北极星指标及输入指标 vs 目标
   ┌──────────────────────┬─────────┬─────────┬────────┐
   │ 指标                 │ 目标    │ 实际    │ 趋势   │
   ├──────────────────────┼─────────┼─────────┼────────┤
   │ 北极星               │         │         │ ↑↓→    │
   │ 输入指标 1            │         │         │ ↑↓→    │
   │ 输入指标 2            │         │         │ ↑↓→    │
   │ 输入指标 3            │         │         │ ↑↓→    │
   └──────────────────────┴─────────┴─────────┴────────┘
   红/黄/绿标注。绿的不讨论。

2. 实验结果 (15 min)
   每个已完成的实验：
   - 假设
   - 结果（含置信区间）
   - 决策：上线 / 终止 / 迭代
   - 下一步和负责人

3. 流水线评审 (10 min)
   正在运行的实验：
   - 状态（样本量完成百分比）
   - 预计完成日期
   - 问题（数据污染、Bug、流量不足）

4. 新实验 (10 min)
   评审 backlog，选出本周上线的 1-3 个。
   每个需要：
   - ICE 评分
   - 负责人
   - 上线日期
   - 预计运行时长

5. 阻塞与决策 (5 min)
   需要当场做决策的事项。
   如果讨论超过 2 分钟，线下处理。
```

### 如何呈现数据

**原则：**
1. 展示趋势，不要只给数字。"激活率 32%"没有意义。"激活率在过去 4 周从 38% 降到了 32%"才是有行动指导的。
2. 实验结果必须带置信区间。"变体 B 转化率提升 12%"不完整。"变体 B 转化率提升 12%（95% CI: 3% 到 21%，p=0.01）"才能告诉你是否该上线。
3. 和上周/上月/去年同期对比。季节性是真实的，B2B 也一样。
4. 分维度拆解。"整体留存 45%"掩盖了企业客户 80% 和 SMB 20% 的事实。
5. 做标注。在每张图表上标记产品发布、定价变更、宕机、市场活动。

### 决策框架

| 信号 | 决策 |
|:---|:---|
| 实验达到统计显著，效果正向且有意义 | 上线 |
| 实验达到显著，效果正向但很小（<2% 相对提升） | 大概不值得维护成本。除非零维护，否则终止。 |
| 实验达到显著，效果负向 | 终止。从中学习。 |
| 实验跑完全部时长，未达到显著 | 无定论。效果（如果有）小于你能检测到的范围。终止或用更多流量重新设计。 |
| 本周指标急剧下降 | 立即调查。是数据问题、Bug，还是真实的产品变化？ |

### 升级标准

在以下情况需要在增长会议之外升级：
- 北极星指标周环比下降 >10% 且无已知原因
- 已上线的实验导致核心指标回退
- CAC 回本周期连续两个月超过 18 个月（SMB）或 24 个月（中型客户）
- Net Dollar Retention 连续两个月低于 95%
- 激活率降至 20% 以下

## 6. 增长实验框架

### ICE 评分

| 维度 | 说明 | 评分 (1-10) |
|:---|:---|:---|
| **Impact** | 对北极星指标的潜在影响 | 1 = 微小，10 = 改变格局 |
| **Confidence** | 对假设的信心（基于数据、用户调研或历史实验） | 1 = 直觉，10 = 强数据支撑 |
| **Ease** | 实施难度 | 1 = 需要数月，10 = 今天就能部署 |

**优先级 = Impact × Confidence × Ease**

独立评分，然后讨论分歧。讨论本身比分数更有价值。

**替代方案：RICE 评分**（当 backlog 中条目触达范围差异很大时使用）

| 维度 | 说明 |
|:---|:---|
| **Reach** | 每季度影响多少用户？ |
| **Impact** | 对每个用户的影响程度？（0.25 = 极小，0.5 = 低，1 = 中，2 = 高，3 = 巨大） |
| **Confidence** | 你有多确定？（100% = 高，80% = 中，50% = 低） |
| **Effort** | 构建所需的人周 |

**RICE 分数 = (Reach × Impact × Confidence) / Effort**

### 实验文档模板

```
实验名称：[名称]

负责人：[姓名]
开始日期：[日期]
结束日期：[日期 或 "达到显著性时"]

假设
如果我们 [具体变更]，那么 [具体指标] 将 [上升/下降] [预期幅度]，
因为 [基于数据/调研的推理]。

指标
主要指标：[这个实验要推动的核心指标]
次要指标：[监控非预期副作用的指标]
护栏指标：[不能变差的指标——如页面加载时间、错误率]

设计
类型：[A/B / A/B/C / 多变量 / 前后对比]
对照组：[未做变更时用户看到的]

分流比例：[50/50 或其他]
定向：[全部用户 / 某细分 / 仅新用户 / 等]

样本量计算
基线转化率：[X%]
最小可检测效应：[Y% 相对变化]
统计功效：[80% 标准，高风险用 90%]
显著性水平：[α = 0.05 标准]
每组所需样本量：[计算值——见下文]
预计达到样本量的天数：[基于当前流量]

结果
[实验完成后填写]
变体 | 转化率 | 相对提升 | 95% CI | p 值
对照组 |                 |               |        |
实验组 |               |               |        |

决策
[ ] 上线
[ ] 终止
[ ] 迭代 → [下次迭代的变更内容]

总结
[我们学到了什么，即使实验失败了？]
```

### 统计显著性

**基础概念：**
- **显著性水平（α）：** 假阳性的概率。标准是 α = 0.05（在没有真实差异时有 5% 的概率误判为有效）。
- **统计功效（1-β）：** 检测到真实效果的概率。标准是 80%，高风险实验用 90%。
- **最小可检测效应（MDE）：** 你关心的最小改进幅度。诚实一点——1% 的相对提升大概不值得投入工程资源。

**如何计算样本量：**

比较两个比例的双侧检验：

```
n = (Z_α/2 + Z_β)² × (p₁(1-p₁) + p₂(1-p₂)) / (p₂ - p₁)²

其中：
  n = 每组样本量
  Z_α/2 = 1.96（α = 0.05 时）
  Z_β = 0.84（功效 80%），1.28（功效 90%）
  p₁ = 基线转化率
  p₂ = 实验组预期转化率
```

**速查表（α = 0.05，功效 80%）：**

| 基线转化率 | MDE（相对） | 每组样本量 |
|:---|:---|:---|
| 2% | 20% (2.0% → 2.4%) | 47,000 |
| 2% | 50% (2.0% → 3.0%) | 6,300 |
| 5% | 10% (5.0% → 5.5%) | 58,000 |
| 5% | 20% (5.0% → 6.0%) | 13,000 |
| 10% | 10% (10% → 11%) | 26,000 |
| 10% | 20% (10% → 12%) | 5,800 |
| 20% | 10% (20% → 22%) | 10,000 |
| 20% | 20% (20% → 24%) | 2,400 |
| 50% | 10% (50% → 55%) | 3,100 |

如果你的流量不足以在 2-4 周内达到这些数字，要么提高 MDE（只测大胆的改动），要么换方法。

没有统计学学位？用这些计算器：
- Evan Miller 的样本量计算器（免费、简单）
- PostHog 内置的实验计算器
- GrowthBook 的贝叶斯分析（处理样本量的方式不同但统计上有效）

### A/B 测试常见陷阱

1. **偷看结果。** 如果你每天看一次，看到好结果就停，你的假阳性率可能超过 30%。要么事先承诺固定样本量，要么使用序贯检验方法（如永远有效的 p 值或贝叶斯方法）。

2. **测试时间太短。** 至少跑满一整个工作周（周一到周日）以消除星期效应。理想是两周。如果测试 3 天就"赢了"，保持怀疑。

3. **测试时间太长。** 如果测试跑了 4-6 周还没有达到显著性，效果大概率太小不值得关注。判定为无定论然后继续。长期运行的测试会因外部事件积累偏差。

4. **没有考虑多重比较。** 如果你看 5 个指标，纯粹靠随机其中一个就会在 α = 0.05 时达到显著。用 Bonferroni 校正（α / 指标数量）或在测试开始前指定一个主指标。

5. **样本比例不匹配（SRM）。** 你的 A/B 分流应该精确是 50/50（或你设定的比例）。如果实际是 48/52，说明分流逻辑有问题。在分析结果前先对样本量做卡方检验。

6. **不做分层分析。** 一个整体持平的实验可能是新用户 +20% 而老用户 -15%。务必检查关键细分。

7. **幸存者偏差。** 如果你的实验改变了第 1 天的行为，但指标是 30 天后才衡量的，早期流失的用户就被排除在分析之外了。使用意向性分析。

8. **测试太多变体。** 每增加一个变体都需要和对照组相同的样本量。A/B/C/D/E 测试需要 A/B 测试 5 倍的流量。控制在 2-3 个变体。

### 多变量测试

当你想同时测试多个改动并理解它们之间的交互时，使用多变量测试。

**示例：** 你想同时测试落地页的标题和 CTA 按钮。
- 标题：A1（原版），A2（新版）
- CTA：B1（原版），B2（新版）
- 组合：A1B1, A1B2, A2B1, A2B2（4 个变体）

**适合使用多变量测试的场景：**
- 流量足够（每个组合都需要完整样本量）
- 改动之间可能有交互效应（标题 + CTA 配合起来效果不同）
- 你想整体优化某个页面/流程

**不适合使用的场景：**
- 流量低——永远达不到显著性
- 改动之间相互独立不存在交互
- 你需要快速拿到结果

### 什么时候不需要 A/B 测试

不是所有东西都需要 A/B 测试。以下情况不用测：

1. **改动显然是对的。** 修复坏掉的注册表单不需要测试。直接修。
2. **流量不够。** 如果达到显著性需要 3 个月，改用定性方法（用户访谈、会话录制）。
3. **改动不可逆。** 定价变更、重大架构调整或品牌变更不容易回滚。用其他验证方法（问卷、支付意愿调研、Beta 测试）。
4. **风险太低。** 改页脚链接颜色不值得一个测试的开销。
5. **还在找 PMF。** 在寻找 PMF 阶段，你应该基于用户反馈快速做大胆改动，而不是花两周测试按钮颜色。学习速度 > 统计严谨性。

## 7. Cohort 分析

Cohort 分析按用户注册时间分组（按周或月），追踪他们随时间的行为。这是理解留存、收入和产品健康度最重要的分析方法。

### 如何构建 Cohort 留存表

行 = cohort（注册周或月）。列 = 注册后的时间。值 = cohort 中仍活跃的比例。

**示例：每周用户留存 Cohort**

| Cohort | 第 0 周 | 第 1 周 | 第 2 周 | 第 3 周 | 第 4 周 | 第 8 周 | 第 12 周 |
|:---|:---|:---|:---|:---|:---|:---|:---|
| 1月1日 (n=500) | 100% | 45% | 35% | 30% | 28% | 22% | 20% |
| 1月8日 (n=520) | 100% | 48% | 38% | 33% | 31% | 25% | 22% |
| 1月15日 (n=480) | 100% | 52% | 42% | 38% | 35% | 29% | — |
| 1月22日 (n=550) | 100% | 55% | 44% | 40% | 37% | — | — |
| 1月29日 (n=600) | 100% | 58% | 47% | 42% | — | — | — |

### 怎么看

1. **曲线变平。** 健康的留存曲线在初期下降后趋于平坦。如果曲线以相同速率持续下降，你没有留存——你有慢性流失问题。

2. **新 cohort 越来越好。** 上面的例子中，每个 cohort 的留存都好于前一个（1月29日第 1 周 = 58% vs 1月1日第 1 周 = 45%）。说明产品改进有效果。

3. **断崖。** 最大的单期跌幅在哪？如果第 0 周到第 1 周损失 55%，那是激活问题。如果第 3 个月到第 6 个月损失 30%，可能是"新鲜感消退"问题。

4. **微笑曲线。** 有时留存曲线先降后升。说明用户离开后意识到还是离不开你的产品。这是积极信号，但要搞清楚他们最初为什么离开。

### 收入 Cohort 表

结构和留存表一样，但值是收入而非活跃用户。更有洞察力。

| Cohort | 第 0 月 | 第 3 月 | 第 6 月 | 第 12 月 | 收入留存 |
|:---|:---|:---|:---|:---|:---|
| 2024 Q1 (n=100) | $10K | $9K | $11K | $13K | 第 12 月 130% |
| 2024 Q2 (n=120) | $14K | $13K | $15K | — | — |
| 2024 Q3 (n=150) | $20K | $19K | — | — | — |

第 12 月收入留存超过 100% 意味着你的老客户付费比刚开始时更多——扩展收入超过了流失。这是目标。

### 如何用 SQL 构建 Cohort 表

```sql
-- 用户留存 cohort（PostgreSQL）
WITH cohort AS (
  SELECT
    user_id,
    DATE_TRUNC('week', created_at) AS cohort_week
  FROM users
),
activity AS (
  SELECT
    user_id,
    DATE_TRUNC('week', event_timestamp) AS activity_week
  FROM events
  WHERE event_name = 'core_action'  -- 用核心动作定义"活跃"，不要只算登录
)
SELECT
  c.cohort_week,
  COUNT(DISTINCT c.user_id) AS cohort_size,
  COUNT(DISTINCT CASE WHEN a.activity_week = c.cohort_week + INTERVAL '1 week' THEN a.user_id END)::float
    / COUNT(DISTINCT c.user_id) AS week_1_retention,
  COUNT(DISTINCT CASE WHEN a.activity_week = c.cohort_week + INTERVAL '2 weeks' THEN a.user_id END)::float
    / COUNT(DISTINCT c.user_id) AS week_2_retention,
  COUNT(DISTINCT CASE WHEN a.activity_week = c.cohort_week + INTERVAL '4 weeks' THEN a.user_id END)::float
    / COUNT(DISTINCT c.user_id) AS week_4_retention
FROM cohort c
LEFT JOIN activity a ON c.user_id = a.user_id
GROUP BY c.cohort_week
ORDER BY c.cohort_week;
```

### Cohort 分析解读指南

| 规律 | 含义 | 怎么做 |
|:---|:---|:---|
| 新 cohort 留存更好 | 产品改进有效 | 继续迭代；找到具体改了什么然后加倍投入 |
| 新 cohort 留存更差 | 有什么在恶化（质量、引导、用户匹配度） | 检查获客渠道是否有变化；复查最近的产品改动 |
| 所有 cohort 都在 ~25% 处趋平 | 你有约 25% 的核心用户，其余全在流失 | 调查这 25% 有什么不同——那就是你的 ICP |
| 收入 cohort 扩展 >100% | 扩展收入健康 | 加大 upsell 投入和按用量计费 |
| 收入 cohort 缩减 <80% | 严重降级或流失 | 紧急调查流失原因；考虑基于价值的定价复盘 |

## 8. 流失分析

### 计算流失

流失有两种类型，告诉你不同的事情。

**Logo 流失**（客户流失）：有多少比例的客户离开了？

```
月度 Logo 流失率 = 本月流失客户数 / 月初客户数
```

**收入流失**（MRR 流失）：有多少比例的收入流失了？

```
月度收入流失率 = 本月因流失损失的 MRR / 月初 MRR
```

**毛流失 vs 净流失：**
- **毛收入流失：** 只算损失的收入（流失 + 缩减）。永远是负的。
- **净收入流失：** 把扩展收入也算进来。可以是负的（这是好事——说明扩展超过流失）。

```
毛收入流失 = (流失 MRR + 缩减 MRR) / 期初 MRR
净收入流失 = (流失 MRR + 缩减 MRR - 扩展 MRR) / 期初 MRR
```

**月度 vs 年度流失：** 不能简单乘 12。5% 月流失不是 60% 年流失——它是复利的。

```
年度流失 = 1 - (1 - 月度流失)^12
5% 月度 = 1 - (0.95)^12 = 46% 年度流失
2% 月度 = 1 - (0.98)^12 = 21.5% 年度流失
```

**基准值：**

| 客群 | 月度 Logo 流失 | 年度 Logo 流失 |
|:---|:---|:---|
| 企业 SaaS | 0.5-1% | 5-10% |
| 中型客户 SaaS | 1-2% | 10-22% |
| SMB SaaS | 3-5% | 31-46% |
| 消费者 SaaS | 5-10% | 46-72% |

### 流失预测指标

用这些信号建立流失预测模型。不需要 ML——加权评分系统对大多数创业公司就够了。

| 指标 | 信号强度 | 如何衡量 |
|:---|:---|:---|
| **登录频次下降** | 强 | 对比最近 2 周和之前 2 周的登录频次 |
| **核心功能使用量下降** | 很强 | 每次会话或每周的核心动作减少 |
| **特定类型的工单** | 中 | 计费问题、"如何取消"、功能投诉 |
| **合同即将到期** | 强 | 距续约 30-60 天内 |
| **关键对接人离职** | 很强 | 通过 LinkedIn 或邮件退信追踪 |
| **支付失败** | 强 | 非自愿流失占 SMB 总流失的 20-40% |
| **12 个月以上无扩展** | 中 | 用量持平说明他们触碰到了天花板 |
| **席位利用率低** | 强 | 买了 50 个席位，只有 10 个活跃 |
| **在评估竞品** | 很强 | G2 评测活动、下载竞品内容（如果你追踪这些的话） |
| **NPS 回复为贬损者** | 中 | NPS < 6 |

### 预警系统

对以下情况设置自动告警：

**红色告警（立即介入——CSM 联系客户）：**
- 核心功能使用量周环比下降 >50%
- 关键对接人邮箱退信（已离职）
- 客户提交了取消请求
- 客户降级了套餐

**黄色告警（一周内主动触达）：**
- 登录频次 2 周内下降 >30%
- 客服满意度评分为负面
- NPS 回复 0-6（贬损者）
- 席位利用率 <30%
- 14+ 天未登录（针对周频使用的产品）

**绿色监控（在月度流失分析中复盘）：**
- 用量持平（无增长）3 个月以上
- 12 个月以上无扩展
- 合同续约 60 天倒计时

### 分风险等级的干预方案

**高风险（红色告警）：**
1. CSM 在 24 小时内发送个性化邮件或电话联系
2. 提供免费策略会议或产品演示
3. 高价值账户（>$10K ARR）升级到管理层
4. 如果关键对接人离职，识别新的利益相关者并重新引导
5. 非自愿流失（支付失败）：自动催付序列 + 3 次重试失败后人工跟进

**中风险（黄色告警）：**
1. 自动邮件："我们注意到你最近没有使用 [功能]——这里有个小技巧"
2. App 内提示高亮低使用率功能
3. 邀请参加即将到来的 webinar 或 office hours
4. CSM 在下一个自然触达节点跟进（QBR、续约）

**低风险（绿色监控）：**
1. 纳入季度业务复盘
2. 分享相关案例和产品更新
3. 主动发起扩展对话（"你已经使用了套餐限额的 80%——我们聊聊下一步"）

### 减少非自愿流失

非自愿流失（支付失败）占 SMB SaaS 全部流失的 20-40%。也是最容易修复的。

1. **催付邮件：** 支付失败后 14 天内发 3-5 封邮件。包含直接更新支付方式的链接。态度友好，不要有威胁感。
2. **App 内支付更新提示：** 用户在支付失败状态下登录时展示横幅——不要弹窗阻断。不要立刻锁定他们的账户。
3. **智能重试逻辑：** 在不同日期和时间重试失败的扣款。周二和周三上午成功率更高。
4. **自动更卡服务：** Stripe 等支付处理商可以自动更新过期卡。启用它。
5. **宽限期：** 降级或暂停前给 7-14 天缓冲。大多数人只是忘了更新卡。
6. **推动年付：** 将客户转为年付。支付事件更少 = 失败节点更少。

## 9. 定价分析

### 衡量定价效果

**需要追踪的关键指标：**

| 指标 | 它告诉你什么 | 健康信号 |
|:---|:---|:---|
| **ARPU 趋势** | 你是否随时间捕获更多价值？ | 年增长 5-20% |
| **套餐分布** | 客户集中在哪个套餐？ | 不是 90% 在最便宜的 |
| **升级率** | 客户多久升一次级？ | 每年 5-15% 的客户 |
| **降级率** | 客户多久降一次级？ | 每年 <3% 的客户 |
| **价格敏感度** | 客户是否愿意付更多？ | Van Westendorp 或联合分析 |
| **分价格段的赢单率** | 定价是否在丢单？ | 因定价丢单 <20% |
| **折扣频率** | 你是否在训练客户要折扣？ | <30% 的交易有折扣 |

### ARPU 分析

按月追踪 ARPU，分以下维度：
- 客户 cohort（新客户是否付费更高？）
- 客户规模（SMB vs 中型 vs 企业）
- 套餐等级
- 行业垂类

**关注点：**
- ARPU 逐月下降 → 可能在下沉市场或定价没有随价值增长
- ARPU 逐月上升 → 如果由扩展驱动则好，如果由小客户流失驱动则坏（幸存者偏差）
- ARPU 持平 → 没有捕获更多价值；考虑按用量计费

### 升降级分析

构建月度流向分析：

```
套餐流向分析 — [月份]

期初各套餐 MRR：
  Free:       — ($0)
  Starter:    $50K
  Pro:        $120K
  Enterprise: $200K

流动：
  Free → Starter:      +$8K  (40 客户 × $200 均值)
  Starter → Pro:       +$15K (30 客户 × $500 均值)
  Pro → Enterprise:    +$10K (5 客户 × $2,000 均值)
  Enterprise → Pro:    -$4K  (2 客户)
  Pro → Starter:       -$3K  (6 客户)
  Starter → Free:      -$2K  (10 客户)
  流失：               -$12K (多客户)

MRR 净变动：            +$12K
```

每月追踪。如果降级在增加，调查原因。常见原因：
- 他们依赖的某个功能被移到了更高套餐
- 客户公司预算削减
- 他们没在用自己付费的功能（价值落差）

### 价格敏感度测试

**Van Westendorp 价格敏感度分析：**

用四个问题调查客户：
1. 什么价格你会觉得太便宜，从而怀疑产品质量？（太便宜）
2. 什么价格你会觉得非常划算？（便宜/好价）
3. 什么价格你开始觉得贵但仍然愿意考虑？（贵/偏高）
4. 什么价格你会觉得贵到不会考虑？（太贵）

绘制累积分布图。交叉点给你：
- **边际低价点：** 太便宜 ∩ 贵
- **边际高价点：** 太贵 ∩ 便宜
- **最优价格点：** 太便宜 ∩ 太贵
- **可接受价格区间：** 边际低价点到边际高价点之间

每个细分群体需要 100-200 个受访者。不要混合 SMB 和企业客户的回复——他们的支付意愿完全不同。

**联合分析：** 更严谨。向用户展示不同价格下的功能组合，让他们排序或选择。需要更多准备工作，但能给你功能级别的支付意愿。工具：Conjointly, SurveyMonkey。

**A/B 测试价格：** 有争议且有风险。有些公司对不同访客展示不同价格。如果客户发现了，信任就毁了。如果你要做：
- 只对新注册测试，不要对现有客户改价
- 价格差异保持在小范围内（±20%）
- 如果有人投诉，以较低价格为准
- 更好的做法：测试不同的套餐配置（功能、限额）而非纯价格

## 10. 董事会与投资人指标

### 各阶段 VC 关注什么

#### 种子轮（$0-$1M ARR）

种子轮 VC 押注的是团队、市场和早期信号。他们想看到：

| 指标 | 他们想看到什么 |
|:---|:---|
| **团队** | 相关领域经验、技术联创、过往创业经验 |
| **市场规模** | TAM > $1B，年增长 >10% |
| **早期牵引力** | 有付费客户，哪怕只有 5-10 个 |
| **用户粘性** | 高激活率，用户每周回来 |
| **留存曲线** | 在趋平，即使数据量小 |
| **烧钱速度** | <$50K/月，融资后跑道 18 个月以上 |

种子轮的收入指标不如粘性和留存信号重要。先证明有人需要这个产品。

#### A 轮（$1M-$5M ARR）

A 轮 VC 想看到可重复的增长和清晰的 PMF 信号。

| 指标 | 他们想看到什么 |
|:---|:---|
| **ARR** | $1-3M，年增长 2-3 倍 |
| **MRR 增长** | 月环比 10-20% |
| **NDR** | >100%，最好 >110% |
| **Logo 留存** | 年度 >85% |
| **LTV/CAC** | >3:1 |
| **CAC Payback** | <18 个月 |
| **毛利率** | >65% |
| **Burn Multiple** | <2× |
| **销售效率** | Magic Number >0.5 |

证明你能持续获客且客户留得住。

#### B 轮（$5M-$20M ARR）

B 轮 VC 想看到规模化的证明。

| 指标 | 他们想看到什么 |
|:---|:---|
| **ARR** | $5-15M，年增长 2 倍 |
| **NDR** | >110%，最好 >120% |
| **Logo 留存** | 年度 >90% |
| **毛利率** | >70% |
| **40 法则** | >40 |
| **Burn Multiple** | <1.5× |
| **Magic Number** | >0.75 |
| **CAC Payback** | <12 个月 |
| **配额完成率** | >60% 的销售代表达标 |
| **销售周期** | 稳定且在缩短 |

他们还关注 GTM 效率以及增加人头是否能带来成比例的收入增长。

#### C 轮及之后（$20M+ ARR）

到这个阶段，你被按照 Pre-IPO 公司的标准审视。

| 指标 | 他们想看到什么 |
|:---|:---|
| **ARR** | $20M+，年增长 50%+ |
| **NDR** | >120% |
| **毛利率** | >75% |
| **40 法则** | >60 |
| **FCF 利润率** | 可见的盈利路径 |
| **Burn Multiple** | <1× |
| **市场份额** | 在品类中有可防御的位置 |
| **国际收入** | 开始地域多元化 |

### 如何在董事会材料中呈现指标

**董事会材料结构（月度或季度）：**

```
1. 高管摘要 (1 页)
   - 3 件进展顺利的事
   - 3 件需要关注的事
   - 1 个需要董事会决策的关键问题

2. KPI 看板 (1-2 页)
   ┌─────────────────────┬──────────┬──────────┬──────────┐
   │ 指标                │ 上季度   │ 本季度   │ 目标     │
   ├─────────────────────┼──────────┼──────────┼──────────┤
   │ ARR                 │          │          │          │
   │ MRR 增长率           │          │          │          │
   │ 新增客户             │          │          │          │
   │ NDR                 │          │          │          │
   │ 毛利率              │          │          │          │
   │ 烧钱/跑道           │          │          │          │
   │ CAC Payback         │          │          │          │
   │ Magic Number        │          │          │          │
   └─────────────────────┴──────────┴──────────┴──────────┘
   红/黄/绿标注。每个红色都要解释。

3. MRR 瀑布图 (1 页)
   展示：期初 MRR → + 新增 → + 扩展 → - 缩减 → - 流失 → 期末 MRR
   图表和数字都要有。

4. COHORT 分析 (1 页)
   Logo 留存 cohort 表和/或收入 cohort 表。
   标注趋势（改善还是恶化）。

5. 销售/GTM (1-2 页)
   Pipeline、赢单率、销售周期、配额完成率、分渠道 CAC。

6. 产品 (1 页)
   重点发布、对激活/留存的影响、路线图优先级。

7. 团队 (1 页)
   人员编制、关键招聘、空缺岗位、离职率。

8. 财务 (1-2 页)
   损益表摘要、现金头寸、跑道月数、烧钱速度。

9. 请求/讨论 (1 页)
   给董事会的具体问题。不要留成开放式的。
```

**呈现原则：**
1. 趋势优先于快照。始终展示 4-6 个季度的数据，让董事会看到轨迹。
2. 带上基准值。"我们的 NDR 是 115%"还行。"我们的 NDR 是 115%，上四分位基准是 120%"才有用。
3. 对 miss 坦诚。董事会在你粉饰数据时会失去信任。如果没达标，直说并解释你在做什么。
4. 不要给他们惊喜。如果有什么不好的，会前单独告知每位董事。
5. 控制在 15-20 页。没人看 40 页的材料。

### 各阶段基准值汇总

| 指标 | 种子轮 | A 轮 | B 轮 | C 轮 |
|:---|:---|:---|:---|:---|
| ARR | $0-$1M | $1-5M | $5-20M | $20M+ |

## 11. 数据基础设施

### 事件追踪搭建

好的分析始于好的数据。大多数分析问题本质上是数据采集问题。

#### 事件命名体系

在整个产品中使用一致的命名规范。这是你关于数据最重要的一个决策。

**命名规范：** `object_action`，使用 snake_case。

```
✓ 好的：
  project_created
  document_shared
  subscription_upgraded
  invite_sent
  report_exported

✗ 坏的：
  createProject         （大小写不一致）
  user clicked button   （太模糊，有空格）
  ButtonClick           （哪个按钮？）
  page_view_homepage    （对象和页面混在一起）
```

**每个 SaaS 都应该追踪的标准事件：**

| 类别 | 事件 | 属性 |
|:---|:---|:---|
| **账户** | `account_created` | plan, source, referrer |
| **账户** | `account_upgraded` | from_plan, to_plan, mrr_change |
| **账户** | `account_downgraded` | from_plan, to_plan, mrr_change, reason |
| **账户** | `account_churned` | plan, mrr_lost, reason, tenure_months |
| **认证** | `user_signed_up` | method (email, google, sso), source, referrer |
| **认证** | `user_logged_in` | method |
| **引导** | `onboarding_step_completed` | step_name, step_number, total_steps |
| **引导** | `onboarding_completed` | time_to_complete_seconds |
| **引导** | `onboarding_skipped` | step_name |
| **核心动作** | `[your_core_action]_completed` | 取决于你的产品 |
| **功能** | `feature_used` | feature_name, context |
| **协作** | `invite_sent` | role, method |
| **协作** | `invite_accepted` | time_to_accept |
| **计费** | `payment_succeeded` | amount, plan |
| **计费** | `payment_failed` | reason, retry_count |
| **互动** | `page_viewed` | page_name, referrer |
| **互动** | `search_performed` | query, results_count |

#### 事件属性

每个事件都应该自动包含以下标准属性（在 SDK 初始化时设置，不是每次事件单独设）：

```
标准属性（设置一次，每个事件自动带上）：
  user_id          — 你的内部用户 ID
  account_id       — B2B 场景：公司/工作空间
  session_id       — 用于会话分析
  timestamp        — ISO 8601
  platform         — web, ios, android, api
  app_version      — 你的应用版本
  plan             — 当前套餐等级
  user_role        — admin, member, viewer
```

#### 实施建议

1. **尽量用服务端追踪。** 客户端追踪会被广告拦截器屏蔽（15-30% 的用户）。关键事件（注册、支付、核心动作）应该在服务端追踪。

2. **使用追踪计划。** 实施任何追踪之前，在表格里写清楚每个事件：事件名、触发时机、属性和哪个团队需要。与所有相关方评审。避免出现"我们有 500 个事件但没人知道其中一半是什么意思"的问题。

3. **不要追踪所有东西。** 追踪决策和结果，不是每一次鼠标移动。你需要 30-50 个定义清晰的事件，不是 500 个噪声事件。

4. **使用 identify 调用。** 用户登录时，发送一个 identify 调用附带用户和账户属性（套餐、公司规模、注册日期）。这会丰富后续每个事件的上下文。

5. **测试你的追踪。** 部署前验证每个事件是否正确触发、属性是否正确。用分析工具的 debug 模式。错误的追踪比没有追踪更糟——你会基于错误数据做决策。

### 数据仓库基础

当你的分析工具不够用时（迟早会的），你需要数据仓库。

**什么时候搭建数仓：**
- 需要跨工具关联数据（Stripe + 产品分析 + CRM）
- 分析工具的 UI 回答不了你的问题
- 需要超出分析工具保留期限的历史数据
- 想在 Metabase/Superset 中构建自定义看板
- 有数据分析师想写 SQL

**创业公司的选择：**

| 数仓 | 优点 | 缺点 | 费用 |
|:---|:---|:---|:---|
| **PostgreSQL（你的生产库）** | 已有，免费 | 不要在生产库上跑重查询 | $0（读副本：$10-50/月） |
| **BigQuery** | Serverless，每月 1TB 免费，无限扩展 | Google 生态锁定 | 对大多数创业公司免费或很便宜 |
| **ClickHouse** | 分析查询极快，开源 | 运维工作量比 BigQuery 大 | Cloud：$200/月起，自托管：免费 |
| **Snowflake** | 复杂工作负载最优，生态好 | 贵，定价复杂 | $500/月起 |
| **DuckDB** | 本地运行，适合原型验证 | 不是共享数仓 | 免费 |

**$5M ARR 以下的大多数创业公司：** 用 PostgreSQL 读副本或 BigQuery。别想太多。

### 创业公司的 ETL 管线

ETL = Extract（从数据源提取）、Transform（清洗和转换）、Load（加载到数仓）。

**现代做法：ELT**（Extract、Load、Transform）。先加载原始数据，在数仓中用 SQL 转换。这样更简单也更灵活。

**推荐栈：**

```
数据源：              ETL/EL：           数仓：              转换：             BI：
┌─────────────┐     ┌──────────┐     ┌────────────┐     ┌──────────┐     ┌──────────┐
│ Stripe      │────→│          │────→│            │────→│          │────→│          │
│ PostHog     │────→│ Airbyte  │────→│ PostgreSQL │────→│ dbt      │────→│ Metabase │
│ HubSpot     │────→│          │────→│ or BigQuery│────→│          │────→│          │
│ Google Ads  │────→│          │────→│            │────→│          │────→│          │
└─────────────┘     └──────────┘     └────────────┘     └──────────┘     └──────────┘
```

**Airbyte**（开源，自托管或云版）：从 300+ 数据源拉数据到你的数仓。配置连接器，设定同步计划，完事。

**dbt**（data build tool，开源）：用 SQL 模型将原始数据转换为干净的、有文档的表。可版本控制、可测试，是行业标准。

**示例 dbt 模型：**
```sql
-- models/metrics/monthly_mrr.sql
WITH subscription_events AS (
  SELECT
    account_id,
    DATE_TRUNC('month', event_date) AS month,
    SUM(mrr_change) AS net_mrr_change
  FROM {{ ref('stg_stripe_invoices') }}
  GROUP BY 1, 2
)
SELECT
  month,
  SUM(net_mrr_change) OVER (ORDER BY month) AS ending_mrr,
  SUM(CASE WHEN net_mrr_change > 0 THEN net_mrr_change ELSE 0 END) AS expansion_mrr,
  SUM(CASE WHEN net_mrr_change < 0 THEN ABS(net_mrr_change) ELSE 0 END) AS churned_mrr
FROM subscription_events
GROUP BY month
ORDER BY month
```

### 数据质量实践

坏数据导致坏决策。像保护可用性一样保护数据质量。

1. **Schema 校验。** 定义每个事件的预期结构（必填属性、类型、允许值）。不符合的拒绝或标记。PostHog 和 Segment 都支持这个。

2. **数据时效性监控。** 如果数据管线超过正常间隔 2 倍没有运行则告警。如果 Stripe 同步是每小时一次，2 小时无数据就告警。

3. **数据量异常检测。** 追踪每日事件量。如果今天的量比 7 日均值高或低 30% 以上则告警。突然下降通常意味着追踪坏了。突然上升通常意味着有 Bug 在循环触发事件。

4. **去重。** 为事件使用幂等键。网络重试、重复点击和客户端 Bug 都会产生重复事件。在入库时或转换层做去重。

5. **dbt 中的数据测试。** 为你的转换数据写测试：

```yaml
# schema.yml
models:
  - name: monthly_mrr
    columns:
      - name: month
        tests:
          - not_null
          - unique
      - name: ending_mrr
        tests:
          - not_null
          - dbt_utils.accepted_range:
              min_value: 0  # MRR 不应为负
```

6. **文档。** 维护数据字典。每个表和字段：是什么、从哪来、已知注意事项、谁负责。如果只有一个人理解你的数据模型，你的 bus factor 是 1。

7. **权限控制。** 不要给所有人数仓的写权限。分析师获得转换后的表的读权限。只有数据管线有写权限。

### 命名规范

一致性比选哪种规范更重要。选一种然后强制执行。

**事件：** `object_action`，snake_case
- `document_created`, `subscription_upgraded`, `invite_accepted`

**属性：** snake_case
- `user_id`, `plan_name`, `created_at`, `is_active`

**表：** `schema.object_plural`，snake_case
- `raw.stripe_invoices`, `staging.stg_users`, `analytics.monthly_mrr`

**dbt 模型：**
- Raw：`raw_[source]_[table]`（如 `raw_stripe_invoices`）
- Staging：`stg_[source]_[table]`（如 `stg_stripe_invoices`）
- Intermediate：`int_[description]`（如 `int_monthly_revenue_by_account`）
- Marts：`[metric_or_entity]`（如 `monthly_mrr`, `customer_health_score`）
