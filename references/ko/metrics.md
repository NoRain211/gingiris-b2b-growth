# 파트 5: 데이터 기반 성장 루프

## 1. 북극성 지표

**정의:** 제품이 고객에게 창출하는 핵심 가치를 가장 잘 측정하는 단일 지표

**요구사항:**
- 전사(제품, 엔지니어링, 마케팅, 세일즈)가 이 지표를 중심으로 결집한다
- 매출이나 허영 지표가 아닌, 고객 가치를 직접 반영한다
- 주간 또는 일간 단위로 측정 가능하다
- 합리적인 기간 내에 제품 변경에 반응한다
- 장기적 비즈니스 건전성(매출, 리텐션)을 예측한다

### 북극성 지표 선정 방법

**프로세스:**
1. 고객이 제품에서 실제 가치를 얻는 모든 순간을 나열한다.
2. 그 가치 교환을 가장 잘 대표하는 행동이나 결과를 찾는다.
3. 이 지표의 변화가 6~12개월 과거 데이터에서 매출 및 리텐션 변화와 상관관계가 있는지 검증한다.
4. 스트레스 테스트: "이 지표가 두 배가 되었는데 매출이 움직이지 않아도 괜찮은가?" 만약 그렇다면 다른 지표를 선택해야 한다.
5. 모든 팀 리드의 동의를 얻는다. 엔지니어링 팀이 납득하지 않으면 계측이 제대로 이루어지지 않는다.

**흔한 실수:**
- 매출을 북극성으로 선정하는 것. 매출은 결과다. 북극성은 매출을 이끄는 입력 지표여야 한다.
- 움직일 수 없는 것을 선정하는 것. "전체 가입 사용자 수"는 올라가기만 하므로 제품 건전성에 대해 아무것도 알려주지 않는다.
- 너무 추상적인 것을 선정하는 것. "고객 만족도"는 측정 불가능하다. "주간 완료 태스크 수"는 측정 가능하다.
- 매 분기 바꾸는 것. 하나를 정해서 최소 1년간 유지해야 한다. 계속 바꾸면 이를 개선하는 역량이 축적되지 않는다.
- CEO 혼자 결정하는 것. 제품과 데이터에 가장 가까운 사람들이 논의에 참여해야 한다.

### SaaS 카테고리별 북극성 지표 예시

| 카테고리 | 제품 예시 | 북극성 지표 | 왜 효과적인가 |
|:---|:---|:---|:---|
| AI 영상 도구 | Synthesia, Loom AI | 주간 내보내기 영상 수 | 내보내기 = 사용자가 결과물을 공유할 만큼 충분한 가치를 얻었다 |
| AI 회의 어시스턴트 | Otter, Fireflies | 주간 생성 회의 요약 수 | 요약 생성 = 핵심 가치 전달 |
| 협업 플랫폼 | Notion, Slack | 주간 활성 협업 팀 수 | 사용자가 아니라 팀 단위 — 협업 자체가 가치 |
| CRM | HubSpot, Salesforce | 주간 업데이트된 딜 수 | 딜 활동 = 세일즈 팀이 가치를 얻고 있다 |
| DevTools / API | Stripe, Twilio | 주간 프로덕션 API 호출 수 | 프로덕션 사용 = 실제 통합, 실제 가치 |
| 프로젝트 관리 | Linear, Asana | 주간 완료 태스크 수 | 생성이 아니라 완료 — 완료가 가치다 |
| 애널리틱스 / BI | Amplitude, Metabase | 비분석가의 주간 쿼리 실행 수 | 데이터 활용 민주화 = 플랫폼 고착성 |
| 보안 | Snyk, Datadog | 주간 해결된 취약점/이슈 수 | 탐지가 아니라 해결 — 가치는 수정에 있다 |
| 전자 서명 / 법률 | DocuSign, PandaDoc | 주간 서명 완료 문서 수 | 서명 = 거래 완료 |
| 고객 지원 | Intercom, Zendesk | 주간 해결된 대화 수 | 해결 = 고객 문제 해소 |
| 디자인 도구 | Figma, Canva | 주간 협업자와 공유된 디자인 수 | 공유 = 디자인이 타인에게 보여줄 수준이다 |
| 이메일 마케팅 | Mailchimp, Brevo | 오픈율 20% 이상인 주간 발송 캠페인 수 | 품질 기준 적용: 발송 자체가 가치가 아니다 |
| 데이터 통합 | Fivetran, Airbyte | 주간 성공적으로 완료된 동기화 수 | 성공적 동기화 = 데이터 파이프라인이 가치를 전달 |

### 선행 지표 vs 후행 지표

북극성 지표는 흔히 **후행** 지표다 — 이미 일어난 일을 알려준다. 이를 예측하고 영향을 미치려면 **선행** 지표가 필요하다.

| 유형 | 정의 | 예시 ("주간 내보내기 영상 수" 기준) |
|:---|:---|:---|
| **후행** | 이미 발생한 결과 | 이번 주 내보내기된 영상 수 |
| **선행** | 후행 지표를 예측하는 초기 신호 | 새 프로젝트 시작, 템플릿 탐색, 첫 렌더링 완료 |

모델을 구축하라: 선행 지표 → 북극성 → 비즈니스 성과(매출, 리텐션).

### 북극성 지표를 입력 지표로 분해하기

북극성 지표를 **지표 트리**로 분해한다. 모든 팀이 하나의 가지를 소유해야 한다.

예시: 북극성 = "주간 활성 협업 팀 수"

```
주간 활성 협업 팀 수
├── 이번 주 신규 활성화 팀 (그로스/마케팅)
│   ├── 가입 → 팀 생성률
│   ├── 신규 팀당 발송된 초대 수
│   └── 초대 수락률
├── 이번 주 유지된 기존 팀 (제품)
│   ├── D7 팀 리텐션율
│   ├── 팀당 주간 사용 기능 수
│   └── 팀 간 협업 이벤트
└── 이번 주 재활성화된 팀 (라이프사이클/CRM)
    ├── 윈백 이메일 오픈율
    ├── 캠페인 기반 재활성화율
    └── 재활성화된 팀의 마지막 활동 이후 경과 시간
```

각 팀이 자신의 가지를 소유한다. 매주 트리를 검토하여 성과가 부진한 가지를 식별한다.

## 2. AARRR 퍼널 지표

| 단계 | 핵심 질문 | 주요 지표 |
|:---|:---|:---|
| **획득(Acquisition)** | 사용자는 어디서 오는가? | 채널 UV, 가입 수, 채널별 CAC |
| **활성화(Activation)** | 핵심 가치를 경험했는가? | "아하 모먼트" 전환율, 온보딩 완료율, Time to Value |
| **리텐션(Retention)** | 돌아오는가? | D1/D7/D30 리텐션, 기능 리텐션, 주간 활성률 |
| **수익(Revenue)** | 비용을 지불하는가? | 무료→유료 전환율, ARPU, LTV, 확장 수익 |
| **추천(Referral)** | 추천하는가? | NPS, K-factor, 추천 전환율 |

### 단계별 B2B SaaS 벤치마크

B2B SaaS의 중앙값 및 상위 사분위 수치다. 세그먼트, 가격대, GTM 방식에 따라 달라진다.

#### 획득

| 지표 | 중앙값 | 상위 사분위 | 비고 |
|:---|:---|:---|:---|
| 웹사이트 → 가입 전환율 | 2-5% | 8-12% | PLG 모션 기준; 세일즈 주도는 더 낮음 |
| 통합 CAC (SMB) | $200-500 | <$150 | 완전 부하: 인건비 + 도구 + 광고 |
| 통합 CAC (미드마켓) | $2,000-5,000 | <$1,500 | |
| 통합 CAC (엔터프라이즈) | $10,000-30,000 | <$8,000 | 긴 세일즈 사이클이 이를 높인다 |
| 가입 중 오가닉 비율 | 30-40% | >60% | 높은 오가닉 = 복리 성장 |

#### 활성화

| 지표 | 중앙값 | 상위 사분위 | 비고 |
|:---|:---|:---|:---|
| 가입 → 활성화 | 20-30% | 40-60% | "활성화" = 아하 모먼트 완료 |
| Time to Value | 1-3일 | <1시간 | 최고의 PLG 제품은 몇 분 만에 가치를 전달 |
| 온보딩 완료율 | 40-60% | >80% | 단계를 추가할 때마다 20% 하락 |

#### 리텐션

| 지표 | 중앙값 | 상위 사분위 | 비고 |
|:---|:---|:---|:---|
| 1개월 리텐션 | 40-50% | >60% | 로고가 아닌 사용자 기준 — 계정 단위는 다르다 |
| 3개월 리텐션 | 25-35% | >45% | 대부분의 제품이 여기서 사용자를 잃는다 |
| 12개월 리텐션 | 15-25% | >35% | 15% 미만이면 성장 문제가 아니라 제품 문제다 |
| 로고 리텐션 (연간) | 85-90% | >95% | B2B 계약이 여기에 도움이 된다 |
| Net Dollar Retention | 100-110% | >130% | 최고 수준은 이탈보다 확장이 빠르다 |

#### 수익

| 지표 | 중앙값 | 상위 사분위 | 비고 |
|:---|:---|:---|:---|
| 무료 → 유료 전환 | 2-5% | 7-12% | 프리미엄 기준; 무료 체험 전환은 더 높음(15-25%) |
| 체험 → 유료 (카드 미등록) | 10-15% | 20-30% | 카드 필수 시 40-60%까지 올라가지만 가입은 감소 |
| 월간 → 연간 전환 | 15-25% | >40% | 연간 결제에 15-20% 할인으로 현금흐름 개선 |
| ARPU 연간 성장률 | 5-10% | >20% | 가격 최적화 및 확장 |

#### 추천

| 지표 | 중앙값 | 상위 사분위 | 비고 |
|:---|:---|:---|:---|
| NPS | 30-40 | >50 | B2B SaaS 평균은 B2C보다 낮다 |
| K-factor | 0.1-0.3 | >0.5 | K > 1이면 바이럴; B2B 제품은 거의 달성 불가 |
| 추천을 통한 신규 사용자 비율 | 10-20% | >35% | 낮은 CAC 채널; 쉽게 만들 수 있도록 투자할 것 |

### 병목 식별 및 해결

퍼널에서 가장 큰 병목은 가장 많은 절대 사용자 수를 잃는 지점이다(비율이 아니라). 10,000명에서 50% 이탈은 100명에서 70% 이탈보다 심각하다.

**획득 병목 증상:**
- LTV 대비 높은 CAC
- 가입의 60% 이상을 유료 채널에 의존
- 타깃 시장에서 낮은 브랜드 인지도

**해결책:** 콘텐츠/SEO에 투자(6~12개월 후 성과), 추천 루프 구축, 랜딩 페이지 최적화, 채널 다각화 시도. 같은 유료 채널에 더 많이 쓰지 마라.

**활성화 병목 증상:**
- 가입 → 아하 모먼트 전환율 25% 미만
- Time to Value 24시간 초과
- 가입 후 온보딩을 완료하지 않는 사용자

**해결책:** 온보딩 단축(단계 제거, 계정 설정 지연), 샘플 데이터 미리 채우기, 트리거 기반 온보딩 이메일, 리텐션된 사용자와 이탈한 사용자의 첫 세션 행동을 비교하여 실제 아하 모먼트 식별.

**리텐션 병목 증상:**
- 1개월과 3개월 사이 급격한 하락
- 코호트의 기능 사용이 주 단위로 감소
- 로그인은 하지만 핵심 행동을 하지 않음

**해결책:** 재참여를 위한 라이프사이클 이메일, 미사용 기능에 대한 인앱 안내, 습관 형성(알림, 다이제스트), 이탈 사용자와 직접 대화 — 실제로 전화하라.

**수익 병목 증상:**
- 무료 사용자가 영원히 무료로 남음
- 낮은 확장 수익
- 가격 인상에 대한 높은 민감도

**해결책:** 무료 티어 제한 강화(기능 제한보다 사용량 기반 제한이 효과적), 사용량 기반 가격 도입, 자연스러운 마찰 지점에서 업그레이드 안내, 연간 결제 할인 제공.

**추천 병목 증상:**
- 리텐션이 좋은데도 NPS가 낮음
- 사용자가 추천 프로그램의 존재를 모름
- 추천 과정에 단계가 너무 많음

**해결책:** 제품에 공유 기능 내장(공유 링크, 브랜딩 포함 내보내기), 추천인과 피추천인 모두에게 인센티브, 추천 행동을 맥락에 맞게 배치(설정 페이지가 아닌, 성공 순간 직후).

### 단계별 대시보드 예시

각 단계별로 3~5개 지표가 포함된 자체 대시보드가 있어야 한다. 다음은 출발점이다.

**획득 대시보드:**
- 일간/주간 가입 수 (전체 및 채널별)
- 채널별 CAC (월간 업데이트)
- 웹사이트 트래픽 → 가입 전환율
- 전환율 기준 상위 10개 랜딩 페이지
- 채널 믹스 추세 (오가닉 vs 유료 vs 추천 vs 직접)

**활성화 대시보드:**
- 가입 → 아하 모먼트 퍼널 (단계별 이탈율)
- Time to Value 분포 (히스토그램)
- 단계별 온보딩 완료율
- 획득 채널별 활성화율 (어떤 채널이 고의도 사용자를 유입시키는가?)
- 가입 코호트별 활성화율 (주간 추세)

**리텐션 대시보드:**
- 코호트 리텐션 테이블 (행 = 가입 주, 열 = 가입 후 경과 주)
- 주간 활성 사용자 추세
- 기능 리텐션 (주간 각 핵심 기능 사용 활성 사용자 비율)
- 이탈률 추세 (월간)
- 복귀율 (이전에 이탈한 사용자의 복귀)

**수익 대시보드:**
- MRR 워터폴 (신규 + 확장 - 축소 - 이탈)
- ARPU 추세 (월간)
- 무료→유료 전환율 추세
- 채널별 LTV/CAC 비율
- 요금제 티어별 매출

**추천 대시보드:**
- 주간 추천 초대 발송 수
- 추천 전환율 (초대됨 → 가입 → 활성화)
- K-factor 추세
- NPS 추세 (분기별)
- 추천을 통한 신규 가입 비율

## 3. 주요 지표 정의

### 획득 지표

| 지표 | 정의 | 공식 | 건전 범위 |
|:---|:---|:---|:---|
| **CAC** | 고객 획득 비용 | 총 세일즈 & 마케팅 비용 / 신규 획득 고객 수 | ACV에 따라 다름 |
| **LTV** | 고객 생애 가치 | ARPU × 매출총이익률 × (1 / 월간 이탈률) | CAC의 3배 이상 |
| **LTV/CAC** | 획득 효율성 | LTV / CAC | 3:1 ~ 5:1 |
| **CAC Payback** | 획득 비용 회수 기간(월) | CAC / (ARPU × 매출총이익률) | 18개월 미만 (SMB: 12개월 미만) |

**LTV 계산 관련 참고:** 단순 공식(ARPU × 수명)은 매출총이익을 무시하기 때문에 LTV를 과대평가한다. ARPU가 월 $100이지만 매출총이익률이 70%라면 월 기여도는 $100이 아니라 $70이다. 항상 매출총이익 조정 LTV를 사용해야 한다.

**LTV/CAC 관련 참고:** 비율이 5:1을 초과하면 대개 효율적이라는 뜻이 아니라 성장에 과소 투자하고 있다는 뜻이다. 기회를 놓치고 있는 것이다.

### 활성화 지표

| 지표 | 정의 | 확인 방법 |
|:---|:---|:---|
| **아하 모먼트 달성률** | 핵심 가치 경험을 완료한 사용자 비율 | 첫 세션에서 리텐션된 사용자와 이탈한 사용자의 행동 비교 |
| **Time to Value** | 가입부터 첫 가치 수령까지의 시간 | 계정 생성부터 아하 모먼트 이벤트까지의 중앙값 시간 |
| **온보딩 완료율** | 모든 온보딩 단계를 완료한 비율 | 온보딩 플로우에 대한 퍼널 분석 |

### 리텐션 지표

| 지표 | 공식 | 비고 |
|:---|:---|:---|
| **D1 리텐션** | 2일차 활성 사용자 / 1일차 가입 사용자 | 일간 사용 제품에 더 유의미 |
| **D7 리텐션** | 8일차 활성 사용자 / 1일차 가입 사용자 | 대부분의 SaaS에서 핵심 변곡점 |
| **D30 리텐션** | 31일차 활성 사용자 / 1일차 가입 사용자 | 장기 리텐션의 강력한 예측 지표 |
| **로고 리텐션** | (기초 고객 수 - 이탈 고객 수) / 기초 고객 수 | 월간 또는 연간 |
| **Net Dollar Retention** | (기초 MRR + 확장 - 축소 - 이탈) / 기초 MRR | PMF를 가장 잘 나타내는 단일 지표 |

### 수익 지표

| 지표 | 정의 | 공식 |
|:---|:---|:---|
| **MRR** | 월간 반복 수익 | 모든 반복 구독 수익의 합 (연간 계약은 월 단위로 정규화) |
| **ARR** | 연간 반복 수익 | MRR × 12 |
| **ARPU** | 사용자당 평균 수익 | MRR / 총 유료 고객 수 |
| **ACV** | 연간 계약 가치 | 신규 계약의 평균 연간 딜 가치 |
| **NRR / NDR** | 순 수익 / 달러 리텐션 | 아래 섹션 참조 |

### 심화 지표

#### Payback Period (투자 회수 기간)

**설명:** 고객 획득 비용을 회수하는 데 걸리는 개월 수.

**공식:** CAC / (ARPU × 매출총이익률)

**LTV/CAC보다 중요한 이유:**
- LTV는 예측값이다. 미래가 과거와 같을 것이라고 가정한다. Payback Period는 실제 현재 숫자를 사용한다.
- 현금이 중요하다. LTV/CAC 비율이 5:1이어도 회수 기간이 36개월이고 18개월 후에 현금이 바닥나면 의미가 없다.
- 실행 가능하다. ARPU 인상(가격 정책), CAC 절감(효율성), 매출총이익률 개선(인프라 비용)으로 회수 기간을 단축할 수 있다.

**벤치마크:**
| 세그먼트 | 양호 | 우수 |
|:---|:---|:---|
| SMB SaaS | 12개월 미만 | 6개월 미만 |
| 미드마켓 SaaS | 18개월 미만 | 12개월 미만 |
| 엔터프라이즈 SaaS | 24개월 미만 | 18개월 미만 |

회수 기간이 24개월을 넘으면 성장을 위해 현금을 태우고 있는 것이다. 벤처 캐피탈이 뒷받침하는 경우에만 가능하다.

#### Quick Ratio

**설명:** MRR 성장의 효율성을 측정한다. 잃는 1달러당 얼마나 많은 신규 수익을 창출하는가?

**공식:** (신규 MRR + 확장 MRR) / (이탈 MRR + 축소 MRR)

**벤치마크:**
| Quick Ratio | 해석 |
|:---|:---|
| <1 | 축소 중이다. 모든 것을 멈추고 이탈을 해결하라. |
| 1-2 | 느린 성장. 이탈이 대부분의 신규 매출을 잠식한다. |
| 2-4 | 건전한 성장. 손실을 보충하고 더 많이 추가하고 있다. |
| >4 | 우수. 효율적으로 성장하고 있다. |

Quick Ratio 4는 이탈로 $1을 잃을 때마다 신규 + 확장 수익으로 $4를 벌어들인다는 뜻이다. 최고 수준의 B2B SaaS 기업은 4 이상을 유지한다.

**주의:** 신규 MRR(많은 신규 고객)에 의존한 높은 Quick Ratio는 확장 MRR(기존 고객의 추가 지출)에 기반한 것보다 지속 가능성이 떨어진다. 확장이 더 저렴하고 끈끈하다.

#### 매출총이익률(Gross Margin)

**설명:** 매출에서 서비스 제공 비용을 뺀 금액의 매출 대비 비율.

**공식:** (매출 - 매출원가) / 매출

**SaaS에서 매출원가(COGS)에 해당하는 것:**
- 호스팅/인프라 비용 (AWS, GCP, Azure)
- 서드파티 API 비용 (OpenAI, Twilio 등)
- 고객 지원팀 급여 (직접 지원, 어카운트 관리 제외)
- DevOps / SRE 급여 (논쟁의 여지가 있음 — 포함하는 곳도, 하지 않는 곳도 있다)
- 결제 처리 수수료 (Stripe의 2.9% + $0.30)

**매출원가에 포함되지 않는 것:**
- 엔지니어링 급여 (R&D)
- 세일즈 및 마케팅
- 관리비 (임대료, 법률, 회계)

**벤치마크:**
| 유형 | 매출총이익률 |
|:---|:---|
| 전통적 SaaS | 75-85% |
| AI/ML 집약 SaaS | 50-70% (GPU 비용이 실제로 크다) |
| 사용량 기반 SaaS | 60-75% |
| 최고 수준 | >80% |

**왜 중요한가:** 매출총이익률은 매출 1달러당 성장, R&D, 이익에 사용할 수 있는 금액을 결정한다. 매출총이익률 80%인 회사는 12개월 회수 기간을 감당할 수 있다. 50%인 회사는 감당할 수 없다 — 6개월 이하로 회수해야 한다.

AI 집약 SaaS 기업: 면밀히 지켜봐야 한다. 모든 LLM API 호출이 마진을 잠식한다. AI 기능의 운영 비용이 추가 수익보다 크다면, 성장 스토리로 포장된 마진 문제를 안고 있는 것이다.

#### Rule of 40

**설명:** 성장과 수익성을 결합한 척도. 핵심 아이디어: SaaS 기업은 빠르게 성장하거나, 수익을 내거나, 이상적으로는 둘 다여야 한다.

**공식:** 매출 성장률 (YoY %) + 잉여 현금 흐름 마진 (%)

**예시:**
| 기업 | 성장률 | FCF 마진 | Rule of 40 점수 |
|:---|:---|:---|:---|
| 초기 스타트업 | 150% | -80% | 70 ✓ |
| 성장 단계 | 60% | -10% | 50 ✓ |
| 성숙 SaaS | 20% | 25% | 45 ✓ |
| 부진 기업 | 15% | 5% | 20 ✗ |

**벤치마크:**
- 20 미만: 저조한 성과. 더 빨리 성장하거나 비용을 절감해야 한다.
- 20-40: 수용 가능하지만 인상적이지 않다.
- 40 이상: 건전하다. 고품질 SaaS의 최소 기준이다.
- 60 이상: 엘리트. 이를 지속하는 기업은 극소수다.

**참고:** EBITDA 마진 대신 FCF 마진을 사용하는 사람도 있다. FCF가 자본적 지출과 주식 보상을 포함하므로 더 정직하다.

#### Net Dollar Retention (NDR)

**설명:** 신규 고객을 완전히 무시하고, 기존 고객 기반에서 유지하고 확장하는 수익의 비율.

**공식:** (기초 MRR + 확장 MRR - 축소 MRR - 이탈 MRR) / 기초 MRR

**예시:** 기존 고객 MRR이 $100K로 시작한다. 확장 $15K, 축소 $3K, 이탈 $5K. NDR = ($100K + $15K - $3K - $5K) / $100K = 107%.

**벤치마크:**
| NDR | 해석 |
|:---|:---|
| <90% | 심각한 문제. 제품이 끈끈하지 않거나 가격이 사용량에 비례하여 성장하지 않는다. |
| 90-100% | 이탈을 보충하고 있지만 확장은 없다. |
| 100-110% | 양호. 기존 고객이 소폭 성장하고 있다. |
| 110-130% | 우수. 강력한 확장 수익. |
| >130% | 최고 수준. Snowflake, Datadog, Twilio 수준. |

NDR > 100%는 신규 고객이 전혀 없어도 성장할 수 있다는 의미다. 이것이 SaaS에서 가장 강력한 성장 엔진이다.

**NDR을 개선하는 방법:**
- 사용량 기반 가격 (고객이 성장하면 자연스럽게 더 많이 지불)
- 시트 기반 가격으로 쉬운 시트 추가
- 고객 규모 확대에 따라 잠금 해제되는 티어드 기능
- 인접 제품 크로스셀
- 확장 대화를 위한 전담 CSM 팀

#### Magic Number

**설명:** 세일즈 효율성을 측정한다 — 세일즈 & 마케팅에 쓴 1달러당 얼마나 많은 신규 매출을 생성하는가.

**공식:** (당 분기 ARR - 전 분기 ARR) / 전 분기 세일즈 & 마케팅 비용

**벤치마크:**
| Magic Number | 해석 |
|:---|:---|
| <0.5 | 비효율적. 비용 확대 전에 세일즈 프로세스나 PMF를 해결하라. |
| 0.5-0.75 | 괜찮지만 개선 여지가 있다. |
| 0.75-1.0 | 건전하다. 비용 확대를 시작하라. |
| >1.0 | 매우 효율적. 여기에 연료를 부어라. |

Magic Number 0.75는 세일즈 & 마케팅에 쓴 $1이 해당 분기에 $0.75의 신규 ARR을 가져온다는 뜻이다. 해당 ARR은 반복되므로 그 지출의 LTV는 훨씬 높다.

#### Burn Multiple

**설명:** 신규 ARR 1달러를 생성하기 위해 얼마나 많은 현금을 태우는가.

**공식:** 순 소진 현금 / 순 신규 ARR

**벤치마크:**
| Burn Multiple | 해석 |
|:---|:---|
| <1x | 놀라운 수준. 태우는 것보다 적은 비용으로 ARR을 생성하고 있다. 드물다. |
| 1-1.5x | 우수. 효율적 성장. |
| 1.5-2x | 초기 단계에서는 허용 가능. |
| 2-3x | 비용이 커지고 있다. 투자자가 질문하기 시작할 것이다. |
| >3x | 지속 불가능. 성장을 돈으로 사고 있다. |

Burn Multiple은 2022년 이후 VC들이 점점 더 주목하는 지표다. 무조건적인 성장보다 효율적 성장이 중요하다.

#### 채널별 CAC Payback

통합 CAC만 계산하지 마라. 채널별로 분해해야 한다. 통합 수치는 엄청난 차이를 숨기기 때문이다.

**분해 예시:**

| 채널 | CAC | 회수 기간 | LTV/CAC | 판정 |
|:---|:---|:---|:---|:---|
| 오가닉 검색 | $80 | 2개월 | 15:1 | 공격적으로 확대 |
| 콘텐츠 마케팅 | $150 | 4개월 | 8:1 | 더 투자 |
| Google Ads (브랜드) | $200 | 5개월 | 6:1 | 유지 |
| Google Ads (비브랜드) | $600 | 15개월 | 2:1 | 최적화 또는 중단 |
| LinkedIn Ads | $900 | 22개월 | 1.5:1 | 엔터프라이즈 타깃이 아니라면 중단 |
| 아웃바운드 SDR | $1,200 | 18개월 | 2.5:1 | 미드마켓 이상 딜에만 적용 |
| 컨퍼런스 스폰서십 | $2,500 | 30개월 이상 | 1:1 | 보통 효과가 없다 |

요점: "통합 CAC $400"은 의미가 없다. $80인 채널도 있고 $2,500인 채널도 있다. 저렴한 채널에 집중 투자하고, 비싼 채널은 중단하거나 최적화하라.

## 5. 성장 회의 베스트 프랙티스

### 빈도
주간. 격주는 너무 느리다 — 추진력을 잃는다. 매주 논의할 것이 부족하다면 실험 속도가 너무 낮은 것이다.

### 참가자
- 프로덕트 매니저 (아젠다 주관)
- 그로스/마케팅 리드
- 엔지니어링 리드 (전체 팀이 아님 — 실현 가능성과 일정에 대해 말할 수 있는 한 명)
- 데이터 분석가 / 분석 담당자
- 선택: 디자인 리드, 세일즈 리드 (세일즈 병행 PLG 모션의 경우)

4~6명으로 유지한다. 그 이상이면 의사결정이 이루어지지 않는다.

### 미팅 아젠다 템플릿 (45분)

```
성장 미팅 — [날짜]

1. 스코어카드 (5분)
   북극성 지표 및 입력 지표 vs 목표 검토
   ┌──────────────────────┬─────────┬─────────┬────────┐
   │ 지표                  │ 목표    │ 실적    │ 추세   │
   ├──────────────────────┼─────────┼─────────┼────────┤
   │ 북극성 지표            │         │         │ ↑↓→    │
   │ 입력 지표 1            │         │         │ ↑↓→    │
   │ 입력 지표 2            │         │         │ ↑↓→    │
   │ 입력 지표 3            │         │         │ ↑↓→    │
   └──────────────────────┴─────────┴─────────┴────────┘
   각각 빨강/노랑/초록 표시. 초록은 논의하지 않는다.

2. 실험 결과 (15분)
   완료된 각 실험에 대해:
   - 가설
   - 결과 (신뢰 구간 포함)
   - 결정: 출시 / 중단 / 반복
   - 다음 조치와 담당자

3. 파이프라인 리뷰 (10분)
   현재 진행 중인 실험:
   - 상태 (필요 표본 크기의 %)
   - 예상 완료일
   - 이슈 (오염, 버그, 낮은 트래픽)

4. 신규 실험 (10분)
   백로그 검토, 이번 주 런칭할 상위 1-3개 선정.
   각각에 대해:
   - ICE 스코어
   - 담당자
   - 런칭일
   - 예상 기간

5. 차단 요소 & 의사결정 (5분)
   즉시 결정이 필요한 사항.
   2분 이상 논의가 필요하면 별도로 처리한다.
```

### 데이터 발표 방법

**원칙:**
1. 숫자가 아니라 추세를 보여줘라. "활성화율이 32%입니다"는 쓸모없다. "활성화율이 지난 4주간 38%에서 32%로 하락했습니다"는 실행 가능하다.
2. 실험 결과에는 항상 신뢰 구간을 표시하라. "변형 B가 전환율을 12% 개선했습니다"는 불완전하다. "변형 B가 전환율을 12% 개선했습니다 (95% CI: 3% ~ 21%, p=0.01)"는 출시 여부를 판단할 수 있게 한다.
3. 같은 기간의 전주/전월/전년과 비교하라. 계절성은 B2B에서도 실재한다.
4. 모든 것을 세그먼트하라. "전체 리텐션이 45%입니다"는 엔터프라이즈 리텐션 80%와 SMB 리텐션 20%라는 사실을 숨긴다.
5. 주석을 사용하라. 모든 차트에 제품 출시, 가격 변경, 장애, 마케팅 캠페인을 표시하라.

### 의사결정 프레임워크

| 신호 | 결정 |
|:---|:---|
| 실험이 통계적 유의성에 도달, 효과가 긍정적이고 유의미 | 출시 |
| 실험이 유의성에 도달, 효과가 긍정적이지만 미미함 (상대적 2% 미만 리프트) | 유지보수 비용의 가치가 없을 가능성이 높다. 유지보수 없이 작동하지 않는 한 중단. |
| 실험이 유의성에 도달, 효과가 부정적 | 중단. 교훈을 얻어라. |
| 실험이 전체 기간 실행되었으나 유의성 없음 | 결론 불확실. 효과가 있다면 감지할 수 있는 수준보다 작다. 중단하거나 더 많은 트래픽으로 재설계. |
| 이번 주 지표가 급락 | 즉시 조사하라. 데이터 문제인가, 버그인가, 실제 제품 변화인가? |

### 에스컬레이션 기준

다음 상황에서는 성장 회의 밖으로 에스컬레이션한다:
- 북극성 지표가 원인 불명으로 주 단위 10% 이상 하락
- 출시된 실험이 핵심 지표에 회귀를 야기
- CAC Payback이 2개월 연속 18개월(SMB) 또는 24개월(미드마켓) 초과
- Net Dollar Retention이 2개월 연속 95% 미만으로 하락
- 활성화율이 20% 미만으로 하락

## 6. 성장 실험 프레임워크

### ICE 스코어링

| 차원 | 설명 | 점수 (1-10) |
|:---|:---|:---|
| **Impact** | 북극성 지표에 대한 잠재적 영향 | 1 = 미미, 10 = 판도를 바꿈 |
| **Confidence** | 가설에 대한 확신도 (데이터, 사용자 리서치, 이전 실험 기반) | 1 = 직감, 10 = 강력한 데이터 |
| **Ease** | 구현 용이성 | 1 = 수개월 소요, 10 = 오늘 배포 가능 |

**우선순위 = Impact × Confidence × Ease**

독립적으로 점수를 매긴 뒤 의견 차이를 논의한다. 논의가 점수보다 더 가치 있다.

**대안: RICE 스코어링** (백로그 항목의 도달 범위가 크게 다를 때 사용)

| 차원 | 설명 |
|:---|:---|
| **Reach** | 분기당 몇 명의 사용자에게 영향을 미치는가? |
| **Impact** | 각 사용자에게 얼마나 영향을 미치는가? (0.25 = 최소, 0.5 = 낮음, 1 = 중간, 2 = 높음, 3 = 대규모) |
| **Confidence** | 얼마나 확신하는가? (100% = 높음, 80% = 중간, 50% = 낮음) |
| **Effort** | 구축에 필요한 인주(person-weeks) |

**RICE 점수 = (Reach × Impact × Confidence) / Effort**

### 실험 문서 템플릿

```
실험: [이름]

담당자: [이름]
시작일: [날짜]
종료일: [날짜 또는 "유의성 도달 시"]

가설
[구체적 변경]을 하면 [구체적 지표]가 [예상 크기]만큼 [증가/감소]할 것이다.

[ ] 출시

```

### 통계적 유의성

**기본 개념:**
- **유의 수준 (α):** 위양성 확률. 표준은 α = 0.05 (실제 차이가 없는데 승자를 선언할 확률 5%).
- **통계적 검정력 (1-β):** 실제 효과를 탐지할 확률. 표준은 80%. 고위험 실험은 90%.
- **최소 탐지 효과 (MDE):** 관심을 가질 만한 최소 개선 폭. 솔직해져야 한다 — 1% 상대적 리프트는 엔지니어링 노력의 가치가 없을 것이다.

**표본 크기 계산 방법:**

두 비율을 비교하는 양측 검정의 경우:

```
n = (Z_α/2 + Z_β)² × (p₁(1-p₁) + p₂(1-p₂)) / (p₂ - p₁)²

여기서:
  n = 변형당 표본 크기
  Z_α/2 = α = 0.05일 때 1.96
  Z_β = 검정력 80%일 때 0.84, 90%일 때 1.28
  p₁ = 기준 전환율
  p₂ = 트리트먼트의 예상 전환율
```

**빠른 참조표 (α = 0.05, 검정력 = 80%):**

| 기준 전환율 | MDE (상대적) | 변형당 표본 크기 |
|:---|:---|:---|
| 2% | 20% (2.0% → 2.4%) | 47,000 |
| 2% | 50% (2.0% → 3.0%) | 6,300 |
| 5% | 10% (5.0% → 5.5%) | 58,000 |
| 5% | 20% (5.0% → 6.0%) | 13,000 |
| 10% | 10% (10% → 11%) | 26,000 |
| 10% | 20% (10% → 12%) | 5,800 |
| 20% | 10% (20% → 22%) | 10,000 |
| 20% | 20% (20% → 24%) | 2,400 |
| 50% | 10% (50% → 55%) | 3,100 |

2~4주 내에 이 수치에 도달할 트래픽이 없다면 더 큰 MDE가 필요하거나(대담한 변경만 테스트) 다른 방법론이 필요하다.

통계학 학위가 없다면? 다음 계산기를 사용하라:
- Evan Miller의 표본 크기 계산기 (무료, 간단)
- PostHog 내장 실험 계산기
- GrowthBook의 베이지안 분석 (표본 크기를 다르게 처리하지만 통계적으로 유효)

### 일반적인 A/B 테스트 함정

1. **결과를 너무 일찍 확인하는 것.** 매일 확인하고 좋아 보일 때 중단하면 위양성률이 30%를 초과할 수 있다. 고정 표본 크기에 확약하거나 순차 검정 방법(항상 유효 p-value 또는 베이지안 방법)을 사용하라.

2. **너무 짧게 테스트하는 것.** 요일 효과를 감안하려면 최소 1주일(월~일). 이상적으로는 2주. 3일 만에 "승리"했다면 의심하라.

3. **너무 길게 테스트하는 것.** 4~6주 후에도 유의성에 도달하지 못했다면 효과가 유의미하지 않을 가능성이 높다. 결론 불확실로 처리하고 넘어가라. 장기간 테스트는 외부 이벤트로 인한 편향이 축적된다.

4. **다중 비교를 고려하지 않는 것.** 5개의 지표를 테스트하면 그중 하나는 순전히 우연으로 α = 0.05에서 유의할 것이다. 본페로니 보정(α / 지표 수)을 사용하거나 테스트 전에 하나의 주요 지표를 지정하라.

5. **표본 비율 불일치(SRM).** A/B 분할은 정확히 50/50(또는 설정한 비율)이어야 한다. 48/52라면 할당 로직에 문제가 있다. 결과를 분석하기 전에 표본 크기에 대해 카이제곱 검정을 실행하라.

6. **세그먼트하지 않는 것.** 전체적으로 보면 평탄한 실험이 신규 사용자에게는 +20%, 기존 사용자에게는 -15%일 수 있다. 항상 핵심 세그먼트를 확인하라.

7. **생존 편향.** 실험이 1일차 행동을 변경하지만 지표가 30일에 걸쳐 측정된다면, 초기에 이탈한 사용자는 분석에서 제외된다. 치료 의도 분석(intention-to-treat)을 사용하라.

8. **너무 많은 변형을 테스트하는 것.** 추가 변형마다 컨트롤과 동일한 표본 크기가 필요하다. A/B/C/D/E 테스트는 A/B 테스트의 5배 트래픽이 필요하다. 2~3개 변형으로 유지하라.

### 다변량 테스트

여러 변경을 동시에 테스트하고 변경 간 상호작용을 이해하고 싶을 때 다변량 테스트를 사용한다.

**예시:** 랜딩 페이지의 헤드라인과 CTA 버튼을 모두 테스트하고 싶다.
- 헤드라인: A1 (원본), A2 (신규)
- CTA: B1 (원본), B2 (신규)
- 조합: A1B1, A1B2, A2B1, A2B2 (4개 변형)

**다변량 테스트를 사용할 때:**
- 충분한 트래픽이 있을 때 (각 조합에 대해 전체 표본 크기 필요)
- 변경 사항이 상호작용할 수 있을 때 (헤드라인 + CTA가 함께 작동)
- 페이지/플로우를 전체적으로 최적화하고 싶을 때

**다변량 테스트를 사용하지 말아야 할 때:**
- 트래픽이 적을 때 — 유의성에 절대 도달하지 못한다
- 변경 사항이 독립적이고 상호작용하지 않을 때
- 빠른 결과가 필요할 때

### A/B 테스트를 하지 말아야 할 때

모든 것에 A/B 테스트가 필요하지는 않다. 다음 경우에는 테스트하지 마라:

1. **변경이 명백히 맞을 때.** 고장난 가입 양식을 수정하는 데 테스트가 필요 없다. 그냥 고쳐라.
2. **트래픽이 충분하지 않을 때.** 유의성에 도달하는 데 3개월이 걸린다면 정성적 방법(사용자 인터뷰, 세션 녹화)을 사용하라.
3. **변경이 비가역적일 때.** 가격 변경, 대규모 아키텍처 변경, 브랜드 변경은 쉽게 롤백할 수 없다. 다른 검증 방법(설문, 지불 의향 연구, 베타 프로그램)을 사용하라.
4. **중요도가 너무 낮을 때.** 푸터 링크 색상 변경은 테스트 오버헤드의 가치가 없다.
5. **Pre-PMF 단계일 때.** 아직 PMF를 찾고 있다면 사용자 피드백을 기반으로 과감한 변경을 빠르게 해야 한다. 버튼 색상에 대해 2주짜리 테스트를 하는 것이 아니다. 학습 속도 > 통계적 엄밀성.

## 7. 코호트 분석

코호트 분석은 사용자를 시작 시점(가입 주 또는 월)별로 그룹화하고 시간에 따른 행동을 추적한다. 리텐션, 매출, 제품 건전성을 이해하는 데 가장 중요한 분석 기법이다.

### 코호트 리텐션 테이블 구축 방법

행 = 코호트 (가입 주 또는 월). 열 = 가입 후 경과 시간. 값 = 코호트 내 여전히 활성인 비율.

**예시: 주간 사용자 리텐션 코호트**

| 코호트 | 0주차 | 1주차 | 2주차 | 3주차 | 4주차 | 8주차 | 12주차 |
|:---|:---|:---|:---|:---|:---|:---|:---|
| 1월 1일 (n=500) | 100% | 45% | 35% | 30% | 28% | 22% | 20% |
| 1월 8일 (n=520) | 100% | 48% | 38% | 33% | 31% | 25% | 22% |
| 1월 15일 (n=480) | 100% | 52% | 42% | 38% | 35% | 29% | — |
| 1월 22일 (n=550) | 100% | 55% | 44% | 40% | 37% | — | — |
| 1월 29일 (n=600) | 100% | 58% | 47% | 42% | — | — | — |

### 확인해야 할 사항

1. **평탄해지는 곡선.** 건전한 리텐션 곡선은 초기 하락 후 평탄해진다. 곡선이 같은 비율로 계속 하락한다면 리텐션이 아니라 느린 이탈 문제다.

2. **시간이 지남에 따라 개선되는 코호트.** 위 예시에서 각 코호트가 이전보다 더 잘 유지되고 있다 (1월 29일 1주차 = 58% vs 1월 1일 1주차 = 45%). 제품 개선이 효과를 보고 있다는 뜻이다.

3. **절벽.** 가장 큰 단일 기간 하락은 어디인가? 0주차와 1주차 사이에 55%를 잃는다면 활성화 문제다. 3개월과 6개월 사이에 30%를 잃는다면 "새로움이 사라지는" 문제일 수 있다.

4. **스마일 곡선.** 때때로 리텐션 곡선이 하락했다가 다시 올라간다. 사용자가 떠났다가 제품이 그립다는 것을 깨닫고 돌아온다는 뜻이다. 긍정적 신호이지만 애초에 왜 떠나는지 이해해야 한다.

### 매출 코호트 테이블

리텐션과 같은 구조이지만 활성 사용자 대신 매출을 사용한다. 더 시사적이다.

| 코호트 | 0개월 | 3개월 | 6개월 | 12개월 | 매출 리텐션 |
|:---|:---|:---|:---|:---|:---|
| 2024 Q1 (n=100) | $10K | $9K | $11K | $13K | 12개월차 130% |
| 2024 Q2 (n=120) | $14K | $13K | $15K | — | — |
| 2024 Q3 (n=150) | $20K | $19K | — | — | — |

12개월차 매출 리텐션이 100%를 초과한다는 것은 기존 고객이 시작했을 때보다 더 많이 지불하고 있다는 뜻이다 — 확장 수익이 이탈을 초과한다. 이것이 목표다.

### SQL로 코호트 테이블 구축하기

```sql
-- 사용자 리텐션 코호트 (PostgreSQL)
WITH cohort AS (
  SELECT
    user_id,
    DATE_TRUNC('week', created_at) AS cohort_week
  FROM users
),
activity AS (
  SELECT
    user_id,
    DATE_TRUNC('week', event_timestamp) AS activity_week
  FROM events
  WHERE event_name = 'core_action'  -- "활성"을 로그인이 아닌 핵심 행동으로 정의
)
SELECT
  c.cohort_week,
  COUNT(DISTINCT c.user_id) AS cohort_size,
  COUNT(DISTINCT CASE WHEN a.activity_week = c.cohort_week + INTERVAL '1 week' THEN a.user_id END)::float
    / COUNT(DISTINCT c.user_id) AS week_1_retention,
  COUNT(DISTINCT CASE WHEN a.activity_week = c.cohort_week + INTERVAL '2 weeks' THEN a.user_id END)::float
    / COUNT(DISTINCT c.user_id) AS week_2_retention,
  COUNT(DISTINCT CASE WHEN a.activity_week = c.cohort_week + INTERVAL '4 weeks' THEN a.user_id END)::float
    / COUNT(DISTINCT c.user_id) AS week_4_retention
FROM cohort c
LEFT JOIN activity a ON c.user_id = a.user_id
GROUP BY c.cohort_week
ORDER BY c.cohort_week;
```

### 코호트 분석 해석 가이드

| 패턴 | 의미 | 대응 방법 |
|:---|:---|:---|
| 최근 코호트가 더 잘 유지됨 | 제품 개선이 효과를 보고 있다 | 계속 반복; 무엇이 바뀌었는지 찾아서 집중 투자 |
| 최근 코호트가 더 나빠짐 | 무언가 악화되고 있다 (품질, 온보딩, 타깃 적합성) | 획득 채널이 변경되었는지 확인; 최근 제품 변경 검토 |
| 모든 코호트가 ~25%에서 평탄 | ~25%의 핵심 사용자가 있고 나머지는 모두 잃고 있다 | 25%를 차별화하는 요인 조사 — 그것이 ICP다 |
| 매출 코호트가 100% 초과로 확장 | 확장 수익이 건전하다 | 업셀 모션과 사용량 기반 가격에 투자 |
| 매출 코호트가 80% 미만으로 축소 | 심각한 다운그레이드 또는 이탈 | 이탈 원인 긴급 조사; 가치 기반 가격 재검토 고려 |

## 8. 이탈 분석

### 이탈률 계산

이탈에는 두 가지 유형이 있으며 각각 다른 것을 알려준다.

**로고 이탈** (고객 이탈): 전체 고객 중 몇 퍼센트가 떠나는가?

```
월간 로고 이탈률 = 이번 달 이탈 고객 수 / 월초 고객 수
```

**매출 이탈** (MRR 이탈): 전체 매출 중 몇 퍼센트가 떠나는가?

```
월간 매출 이탈률 = 이번 달 이탈 MRR / 월초 MRR
```

**총 vs 순 매출 이탈:**
- **총 매출 이탈:** 손실 매출만 계산(이탈 + 축소). 항상 음수다.
- **순 매출 이탈:** 확장 수익을 반영한다. 음수가 될 수 있다(좋은 것 — 확장이 이탈을 초과한다는 뜻).

```
총 매출 이탈 = (이탈 MRR + 축소 MRR) / 기초 MRR
순 매출 이탈 = (이탈 MRR + 축소 MRR - 확장 MRR) / 기초 MRR
```

**월간 vs 연간 이탈:** 단순 환산하지 마라. 월 5% 이탈은 연 60% 이탈이 아니다 — 복리로 계산된다.

```
연간 이탈 = 1 - (1 - 월간 이탈)^12
월 5% = 1 - (0.95)^12 = 연 46% 이탈
월 2% = 1 - (0.98)^12 = 연 21.5% 이탈
```

**벤치마크:**

| 세그먼트 | 월간 로고 이탈 | 연간 로고 이탈 |
|:---|:---|:---|
| 엔터프라이즈 SaaS | 0.5-1% | 5-10% |
| 미드마켓 SaaS | 1-2% | 10-22% |
| SMB SaaS | 3-5% | 31-46% |
| 컨슈머 SaaS | 5-10% | 46-72% |

### 이탈 예측 지표

이 신호를 활용하여 이탈 예측 모델을 구축하라. ML이 필요 없다 — 가중 점수 체계가 대부분의 스타트업에 충분하다.

| 지표 | 신호 강도 | 측정 방법 |
|:---|:---|:---|
| **로그인 빈도 감소** | 강함 | 최근 2주 로그인 빈도를 그 이전 2주와 비교 |
| **핵심 기능 사용 감소** | 매우 강함 | 세션당 또는 주당 핵심 행동 횟수 감소 |
| **지원 티켓 (특정 유형)** | 보통 | 결제 문의, "어떻게 해지하나요", 기능 불만 |
| **계약 만료 임박** | 강함 | 갱신 30-60일 이내 |
| **챔피언 퇴사** | 매우 강함 | LinkedIn 또는 이메일 바운스를 통한 직장 변경 추적 |
| **결제 실패** | 강함 | SMB에서 비자발적 이탈이 전체 이탈의 20-40% |
| **12개월 이상 확장 없음** | 보통 | 사용량 정체는 한계에 도달했음을 시사 |
| **낮은 시트 활용도** | 강함 | 50시트를 결제하지만 10개만 활성 |
| **경쟁사 평가** | 매우 강함 | G2 리뷰 활동, 경쟁사 콘텐츠 다운로드 (추적 시) |
| **NPS 응답이 비추천** | 보통 | NPS < 6 |

### 조기 경보 시스템

다음 조건에 대한 자동 알림을 설정하라:

**빨간색 경보 (즉시 개입 — CSM이 고객에게 전화):**
- 핵심 기능 사용이 주 단위로 50% 이상 감소
- 챔피언 연락처 바운스 (퇴사)
- 고객이 해지 요청을 접수
- 고객이 플랜 다운그레이드

**노란색 경보 (주 내 선제적 연락):**
- 2주간 로그인 빈도 30% 이상 감소
- 지원 만족도 점수가 부정적
- NPS 응답 0-6 (비추천)
- 시트 활용도 30% 미만
- 14일 이상 로그인 없음 (주간 사용 제품 기준)

**초록색 모니터링 (월간 이탈 분석에서 검토):**
- 3개월 이상 사용량 정체 (성장 없음)
- 12개월 이상 확장 없음
- 60일 이내 계약 갱신

### 리스크 수준별 개입 플레이북

**고위험 (빨간색 경보):**
1. CSM이 24시간 내에 개인화된 이메일 또는 전화로 연락
2. 무료 전략 세션 또는 제품 안내 제공
3. 고가치 계정 (ARR $10K 이상)이면 리더십에 에스컬레이션
4. 챔피언이 퇴사한 경우 새 이해관계자를 파악하고 재온보딩
5. 비자발적 이탈(결제 실패): 자동화된 독촉 시퀀스 + 3회 재시도 실패 후 개인 팔로업

**중위험 (노란색 경보):**
1. 자동 이메일: "[기능]을 최근 사용하지 않으신 것 같습니다 — 빠른 팁을 알려드립니다"
2. 미사용 기능을 강조하는 인앱 안내
3. 예정된 웨비나 또는 오피스 아워 초대
4. 다음 자연스러운 접점(QBR, 갱신)에서 CSM 체크인

**저위험 (초록색 모니터링):**
1. 분기별 비즈니스 리뷰에 포함
2. 관련 케이스 스터디 및 제품 업데이트 공유
3. 선제적 확장 대화 ("플랜 한도의 80%를 사용하고 계십니다 — 다음 단계에 대해 이야기해볼까요")

### 비자발적 이탈 줄이기

비자발적 이탈(결제 실패)은 SMB SaaS 전체 이탈의 20-40%를 차지한다. 해결하기 가장 쉬운 이탈이다.

1. **독촉 이메일:** 결제 실패 후 14일간 3~5통의 이메일을 발송한다. 결제 수단 업데이트 직접 링크를 포함한다. 위협적이지 않고 친근하게.
2. **인앱 결제 수단 업데이트 안내:** 결제 실패 상태로 로그인한 사용자에게 차단이 아닌 배너를 표시한다. 즉시 잠그지 마라.
3. **스마트 재시도 로직:** 실패한 결제를 다른 날짜와 시간에 재시도한다. 화요일과 수요일 오전 중반이 성공률이 높다.
4. **카드 업데이터 서비스:** Stripe 등 결제 프로세서가 만료된 카드를 자동으로 업데이트할 수 있다. 이를 활성화하라.
5. **유예 기간:** 다운그레이드 또는 정지 전 7~14일을 준다. 대부분 카드 업데이트를 잊었을 뿐이다.
6. **연간 결제 전환 유도:** 고객을 연간 결제로 전환시킨다. 결제 이벤트 감소 = 실패 지점 감소.

## 9. 가격 분석

### 가격 효과 측정

**추적해야 할 핵심 지표:**

| 지표 | 의미 | 건전한 신호 |
|:---|:---|:---|
| **ARPU 추세** | 시간이 지남에 따라 더 많은 가치를 포착하고 있는가? | YoY 5-20% 증가 |
| **플랜 분포** | 고객이 어디에 몰려 있는가? | 가장 저렴한 플랜에 90%가 아니어야 |
| **업그레이드율** | 고객이 얼마나 자주 상위로 이동하는가? | 연간 고객의 5-15% |
| **다운그레이드율** | 고객이 얼마나 자주 하위로 이동하는가? | 연간 고객의 3% 미만 |
| **가격 민감도** | 고객이 더 지불할 의향이 있는가? | Van Westendorp 또는 컨조인트 분석 |
| **가격별 승률** | 가격이 딜을 잃게 하는가? | 패배 사유 중 가격이 20% 미만 |
| **할인 빈도** | 고객에게 할인을 요청하는 습관을 들이고 있는가? | 딜의 30% 미만이 할인 적용 |

### ARPU 분석

ARPU를 다음 기준으로 세그먼트하여 월간 추적하라:
- 고객 코호트 (최근 고객이 더 많이 지불하는가?)
- 고객 규모 (SMB vs 미드마켓 vs 엔터프라이즈)
- 플랜 티어
- 산업 버티컬

**주의해야 할 사항:**
- ARPU가 시간이 지남에 따라 하락 → 다운마켓으로 이동 중이거나 가격이 가치에 비례하여 확장되지 않음
- ARPU가 시간이 지남에 따라 상승 → 확장에 의한 것이면 좋고, 소규모 고객 이탈에 의한 것이면 나쁨 (생존 편향)
- ARPU가 정체 → 증가하는 가치를 포착하지 못하고 있음; 사용량 기반 가격을 고려하라

### 업그레이드/다운그레이드 분석

월간 플로우 분석을 구축하라:

```
플랜 플로우 분석 — [월]

플랜별 시작 MRR:
  Free:       — ($0)
  Starter:    $50K
  Pro:        $120K
  Enterprise: $200K

이동:
  Free → Starter:      +$8K  (40 고객 × 평균 $200)
  Starter → Pro:       +$15K (30 고객 × 평균 $500)
  Pro → Enterprise:    +$10K (5 고객 × 평균 $2,000)
  Enterprise → Pro:    -$4K  (2 고객)
  Pro → Starter:       -$3K  (6 고객)
  Starter → Free:      -$2K  (10 고객)
  이탈:                -$12K (다양)

순 MRR 이동:           +$12K
```

이를 매월 추적하라. 다운그레이드가 증가한다면 원인을 조사하라. 흔한 원인:
- 의존하던 기능이 상위 티어로 이동
- 고객사의 예산 삭감
- 지불하는 기능을 사용하지 않음 (가치 격차)

### 가격 민감도 테스트

**Van Westendorp 가격 민감도 측정기:**

고객에게 네 가지 질문을 한다:
1. 어느 가격에서 너무 싸서 품질이 의심되는가? (너무 저렴)
2. 어느 가격에서 좋은 딜이라고 느끼는가? (저렴 / 좋은 가치)
3. 어느 가격에서 비싸다고 느끼기 시작하지만 여전히 고려하겠는가? (비싼 / 높은)
4. 어느 가격에서 너무 비싸서 고려하지 않겠는가? (너무 비싼)

누적 분포를 그린다. 교차점이 다음을 제공한다:
- **한계 저렴점:** 너무 저렴 ∩ 비싼
- **한계 고가점:** 너무 비싼 ∩ 저렴
- **최적 가격대:** 너무 저렴 ∩ 너무 비싼
- **수용 가능 가격 범위:** 한계 저렴점과 한계 고가점 사이

세그먼트당 100~200명의 응답자로 실시하라. SMB와 엔터프라이즈 응답을 섞지 마라 — 지불 의향이 완전히 다르다.

**컨조인트 분석:** 더 엄밀하다. 사용자에게 다양한 가격의 기능 번들을 제시하고 순위를 매기거나 선택하게 한다. 설정이 더 필요하지만 기능 수준의 지불 의향을 파악할 수 있다. 도구: Conjointly, SurveyMonkey.

**A/B 테스트로 가격 테스트:** 논쟁적이고 위험하다. 일부 기업이 방문자마다 다른 가격을 테스트한다. 고객이 알게 되면 신뢰가 무너진다. 이를 시행한다면:
- 신규 가입자에게만 테스트하고, 기존 고객의 가격은 절대 변경하지 않는다
- 가격 차이를 작게 유지한다 (±20%)
- 누군가 불만을 제기하면 낮은 가격을 적용한다
- 더 나은 방법: 순수 가격이 아닌 다른 플랜 구성(기능, 한도)을 테스트한다

## 10. 이사회 & 투자자 지표

### 각 단계에서 VC가 주목하는 것

#### Seed ($0-$1M ARR)

Seed 단계의 VC는 팀, 시장, 초기 신호에 베팅한다. 보고 싶어하는 것:

| 지표 | 기대 수준 |
|:---|:---|
| **팀** | 관련 도메인 전문성, 기술 공동 창업자, 이전 스타트업 경험 |
| **시장 규모** | TAM > $1B, YoY 10% 이상 성장 |
| **초기 견인력** | 유료 고객이 5-10개라도 있어야 |
| **사용자 참여도** | 높은 활성화율, 사용자가 주간 복귀 |
| **리텐션 곡선** | 작은 수치라도 평탄해지는 모양 |
| **번율** | 월 $50K 미만, 투자 유치 후 18개월 이상 런웨이 확보 |

Seed 단계에서는 매출 지표보다 참여도와 리텐션 신호가 더 중요하다. 사람들이 이것을 원한다는 것을 증명하라.

#### 시리즈 A ($1M-$5M ARR)

시리즈 A의 VC는 반복 가능한 성장과 명확한 PMF 신호를 원한다.

| 지표 | 기대 수준 |
|:---|:---|
| **ARR** | $1-3M, YoY 2-3배 성장 |
| **MRR 성장** | 월 10-20% |
| **NDR** | >100%, 이상적으로는 >110% |
| **로고 리텐션** | 연간 >85% |
| **LTV/CAC** | >3:1 |
| **CAC Payback** | 18개월 미만 |
| **매출총이익률** | >65% |
| **Burn Multiple** | <2× |
| **세일즈 효율성** | Magic Number >0.5 |

반복적으로 고객을 획득할 수 있고 그들이 머문다는 것을 보여줘라.

#### 시리즈 B ($5M-$20M ARR)

시리즈 B의 VC는 확장할 수 있다는 증거를 원한다.

| 지표 | 기대 수준 |
|:---|:---|
| **ARR** | $5-15M, YoY 2배 성장 |
| **NDR** | >110%, 이상적으로는 >120% |
| **로고 리텐션** | 연간 >90% |
| **매출총이익률** | >70% |
| **Rule of 40** | >40 |
| **Burn Multiple** | <1.5× |
| **Magic Number** | >0.75 |
| **CAC Payback** | 12개월 미만 |
| **쿼터 달성률** | 세일즈 인력의 60% 이상이 쿼터 달성 |
| **세일즈 사이클** | 일관되고 단축되는 추세 |

GTM 효율성과 인원 추가가 비례적 매출 성장을 이끄는지도 중요하게 본다.

#### 시리즈 C+ ($20M+ ARR)

이 단계에서는 Pre-IPO 기업처럼 평가받는다.

| 지표 | 기대 수준 |
|:---|:---|
| **ARR** | $20M+, YoY 50% 이상 성장 |
| **NDR** | >120% |
| **매출총이익률** | >75% |
| **Rule of 40** | >60 |
| **FCF 마진** | 수익성 달성 경로가 가시적 |
| **Burn Multiple** | <1× |
| **시장 점유율** | 카테고리 내 방어 가능한 포지션 |
| **해외 매출** | 지역 다각화 시작 |

### 이사회 자료에서 지표를 발표하는 방법

**이사회 자료 구조 (월간 또는 분기별):**

```
1. 경영 요약 (1 슬라이드)
   - 잘되고 있는 3가지
   - 주의가 필요한 3가지
   - 이사회의 결정이 필요한 1가지 핵심 사안

2. KPI 대시보드 (1-2 슬라이드)
   ┌─────────────────────┬──────────┬──────────┬──────────┐
   │ 지표                 │ 전 분기  │ 금 분기  │ 목표     │
   ├─────────────────────┼──────────┼──────────┼──────────┤
   │ ARR                 │          │          │          │
   │ MRR 성장률           │          │          │          │
   │ 신규 고객            │          │          │          │
   │ NDR                 │          │          │          │
   │ 매출총이익률          │          │          │          │
   │ 번 / 런웨이          │          │          │          │
   │ CAC Payback         │          │          │          │
   │ Magic Number        │          │          │          │
   └─────────────────────┴──────────┴──────────┴──────────┘
   각각 빨강/노랑/초록 표시. 모든 빨간색은 설명한다.

3. MRR 워터폴 (1 슬라이드)
   표시: 기초 MRR → + 신규 → + 확장 → - 축소 → - 이탈 → 기말 MRR
   차트와 숫자를 모두 포함한다.

4. 코호트 분석 (1 슬라이드)
   로고 리텐션 코호트 테이블 및/또는 매출 코호트 테이블.
   추세 강조 (개선 또는 악화).

5. 세일즈 / GTM (1-2 슬라이드)
   파이프라인, 승률, 세일즈 사이클, 쿼터 달성률, 채널별 CAC.

6. 제품 (1 슬라이드)
   주요 출시, 활성화/리텐션 영향, 로드맵 우선순위.

7. 팀 (1 슬라이드)
   인원, 주요 채용, 오픈 포지션, 이직률.

8. 재무 (1-2 슬라이드)
   손익 요약, 현금 포지션, 런웨이(월), 번율.

9. 요청 / 토론 (1 슬라이드)
   이사회에 대한 구체적 질문. 오픈엔드로 두지 마라.
```

**발표 원칙:**
1. 스냅샷이 아닌 추세를 보여줘라. 항상 4~6분기 데이터를 표시하여 이사회가 궤적을 볼 수 있게 하라.
2. 벤치마크를 포함하라. "NDR이 115%입니다"는 좋다. "NDR이 115%이고 상위 사분위 벤치마크는 120%입니다"는 유용하다.
3. 미달에 대해 솔직하라. 나쁜 숫자를 포장하면 이사회의 신뢰를 잃는다. 목표를 달성하지 못했다면 그렇게 말하고 어떤 조치를 취하고 있는지 설명하라.
4. 서프라이즈를 주지 마라. 무언가 잘 안되고 있다면 미팅 전에 이사 개별적으로 알려라.
5. 15-20 슬라이드로 제한하라. 40 슬라이드를 읽는 사람은 없다.

## 11. 데이터 인프라

### 이벤트 추적 설정

좋은 분석은 좋은 데이터에서 시작된다. 대부분의 분석 문제는 데이터 수집 문제다.

#### 이벤트 분류 체계

제품 전체에 걸쳐 일관된 네이밍 규칙을 사용하라. 이것이 데이터에 대해 내릴 가장 중요한 결정이다.

**네이밍 규칙:** `object_action`, snake_case.

```
✓ 좋은 예:
  project_created
  document_shared
  subscription_upgraded
  invite_sent
  report_exported

✗ 나쁜 예:
  createProject         (일관성 없는 케이싱)
  user clicked button   (너무 모호, 공백)
  ButtonClick           (어떤 버튼?)
  page_view_homepage    (객체와 페이지가 혼합)
```

**모든 SaaS가 추적해야 할 표준 이벤트:**

| 카테고리 | 이벤트 | 속성 |
|:---|:---|:---|
| **계정** | `account_created` | plan, source, referrer |
| **계정** | `account_upgraded` | from_plan, to_plan, mrr_change |
| **계정** | `account_downgraded` | from_plan, to_plan, mrr_change, reason |
| **계정** | `account_churned` | plan, mrr_lost, reason, tenure_months |
| **인증** | `user_signed_up` | method (email, google, sso), source, referrer |
| **인증** | `user_logged_in` | method |
| **온보딩** | `onboarding_step_completed` | step_name, step_number, total_steps |
| **온보딩** | `onboarding_completed` | time_to_complete_seconds |
| **온보딩** | `onboarding_skipped` | step_name |
| **핵심 행동** | `[your_core_action]_completed` | 제품에 따라 다름 |
| **기능** | `feature_used` | feature_name, context |
| **협업** | `invite_sent` | role, method |
| **협업** | `invite_accepted` | time_to_accept |
| **결제** | `payment_succeeded` | amount, plan |
| **결제** | `payment_failed` | reason, retry_count |
| **참여** | `page_viewed` | page_name, referrer |
| **참여** | `search_performed` | query, results_count |

#### 이벤트 속성

모든 이벤트에 다음 표준 속성을 자동으로 포함해야 한다 (추적 SDK 초기화에서 설정, 이벤트별이 아님):

```
표준 속성 (한 번 설정, 모든 이벤트에 전송):
  user_id          — 내부 사용자 ID
  account_id       — B2B의 경우: 회사/워크스페이스
  session_id       — 세션 분석용
  timestamp        — ISO 8601
  platform         — web, ios, android, api
  app_version      — 앱 버전
  plan             — 현재 플랜 티어
  user_role        — admin, member, viewer
```

#### 구현 팁

1. **가능하면 서버 사이드로 추적하라.** 클라이언트 사이드 추적은 광고 차단기에 의해 차단된다(사용자의 15-30%). 핵심 이벤트(가입, 결제, 핵심 행동)는 서버 사이드로 추적해야 한다.

2. **추적 계획서를 사용하라.** 추적을 구현하기 전에 모든 이벤트를 스프레드시트에 기록하라: 이벤트 이름, 발생 시점, 속성, 어떤 팀이 필요로 하는지. 모든 이해관계자와 검토하라. "500개 이벤트가 있는데 절반이 뭔지 아무도 모른다"는 문제를 방지한다.

3. **모든 것을 추적하지 마라.** 의사결정과 결과를 추적하라, 모든 마우스 이동을 추적하지 마라. 30~50개의 잘 정의된 이벤트가 필요하지, 500개의 노이즈 이벤트가 필요한 것이 아니다.

4. **identify 호출을 사용하라.** 사용자가 로그인할 때 사용자 및 계정 속성(플랜, 회사 규모, 가입일)과 함께 identify 호출을 보내라. 이것이 이후 모든 이벤트를 풍부하게 한다.

5. **추적을 테스트하라.** 배포 전에 모든 이벤트가 올바른 속성으로 정확하게 실행되는지 확인하라. 분석 도구의 디버그 모드를 사용하라. 깨진 추적은 추적 없는 것보다 나쁘다 — 잘못된 데이터로 의사결정을 내리게 된다.

### 데이터 웨어하우스 기본

분석 도구가 충분하지 않을 때(그렇게 될 것이다), 데이터 웨어하우스가 필요하다.

**웨어하우스를 설정해야 할 때:**
- 도구 간 데이터를 조인해야 할 때 (Stripe + 제품 분석 + CRM)
- 분석 도구의 UI로 질문에 답할 수 없을 때
- 분석 도구의 보존 기한을 초과하는 과거 데이터가 필요할 때
- Metabase/Superset에서 커스텀 대시보드를 구축하고 싶을 때
- SQL을 작성하고 싶은 데이터 분석가가 있을 때

**스타트업을 위한 선택지:**

| 웨어하우스 | 장점 | 단점 | 비용 |
|:---|:---|:---|:---|
| **PostgreSQL (프로덕션 DB)** | 이미 보유, 무료 | 프로덕션에서 무거운 쿼리 실행 금지 | $0 (읽기 전용 복제본: $10-50/월) |
| **BigQuery** | 서버리스, 월 1TB까지 무료, 무한 확장 | Google 생태계 종속 | 대부분의 스타트업에 무료~저렴 |
| **ClickHouse** | 분석 쿼리에 매우 빠름, 오픈소스 | BigQuery보다 운영 부담 큼 | 클라우드: $200/월+, 셀프 호스팅: 무료 |
| **Snowflake** | 복잡한 워크로드에 최적, 우수한 에코시스템 | 비쌈, 복잡한 가격 | $500/월+ |
| **DuckDB** | 로컬 실행, 프로토타이핑에 적합 | 공유 웨어하우스가 아님 | 무료 |

**ARR $5M 미만 대부분의 스타트업:** PostgreSQL 읽기 전용 복제본 또는 BigQuery를 사용하라. 너무 고민하지 마라.

### 스타트업을 위한 ETL 파이프라인

ETL = Extract(소스에서 데이터 추출), Transform(정제 및 가공), Load(웨어하우스에 적재).

**현대적 접근: ELT** (Extract, Load, Transform). 원시 데이터를 먼저 적재하고 웨어하우스에서 SQL로 변환한다. 더 간단하고 유연하다.

**권장 스택:**

```
소스:              ETL/EL:           웨어하우스:        변환:             BI:
┌─────────────┐     ┌──────────┐     ┌────────────┐     ┌──────────┐     ┌──────────┐
│ Stripe      │────→│          │────→│            │────→│          │────→│          │
│ PostHog     │────→│ Airbyte  │────→│ PostgreSQL │────→│ dbt      │────→│ Metabase │
│ HubSpot     │────→│          │────→│ or BigQuery│────→│          │────→│          │
│ Google Ads  │────→│          │────→│            │────→│          │────→│          │
└─────────────┘     └──────────┘     └────────────┘     └──────────┘     └──────────┘
```

**Airbyte** (오픈소스, 셀프 호스팅 또는 클라우드): 300개 이상의 소스에서 웨어하우스로 데이터를 가져온다. 커넥터를 설정하고 동기화를 예약하면 끝.

**dbt** (data build tool, 오픈소스): 원시 데이터를 정제되고 문서화된 테이블로 변환하는 SQL 모델을 작성한다. 버전 관리 가능, 테스트 가능, 업계 표준.

**dbt 모델 예시:**
```sql
-- models/metrics/monthly_mrr.sql
WITH subscription_events AS (
  SELECT
    account_id,
    DATE_TRUNC('month', event_date) AS month,
    SUM(mrr_change) AS net_mrr_change
  FROM {{ ref('stg_stripe_invoices') }}
  GROUP BY 1, 2
)
SELECT
  month,
  SUM(net_mrr_change) OVER (ORDER BY month) AS ending_mrr,
  SUM(CASE WHEN net_mrr_change > 0 THEN net_mrr_change ELSE 0 END) AS expansion_mrr,
  SUM(CASE WHEN net_mrr_change < 0 THEN ABS(net_mrr_change) ELSE 0 END) AS churned_mrr
FROM subscription_events
GROUP BY month
ORDER BY month
```

### 데이터 품질 관리

나쁜 데이터는 나쁜 의사결정으로 이어진다. 데이터 품질을 가용성만큼 보호하라.

1. **스키마 검증.** 모든 이벤트의 예상 형태(필수 속성, 타입, 허용값)를 정의한다. 일치하지 않는 이벤트를 거부하거나 플래그를 지정한다. PostHog과 Segment 모두 이를 지원한다.

2. **데이터 신선도 모니터링.** 데이터 파이프라인이 정상 주기의 2배 동안 실행되지 않으면 알림을 보낸다. Stripe 동기화가 매시간 실행된다면 2시간 동안 데이터가 없을 때 알림.

3. **볼륨 이상 탐지.** 일일 이벤트 볼륨을 추적한다. 오늘의 볼륨이 7일 평균보다 30% 이상 높거나 낮으면 알림. 갑작스러운 감소는 보통 추적이 깨진 것이다. 갑작스러운 급증은 보통 이벤트를 루프에서 발생시키는 버그다.

4. **중복 제거.** 이벤트에 멱등성 키를 사용한다. 네트워크 재시도, 더블 클릭, 클라이언트 버그 모두 중복 이벤트를 일으킨다. 수집 시 또는 변환 레이어에서 중복을 제거한다.

5. **dbt에서의 데이터 테스트.** 변환된 데이터에 대한 테스트를 작성한다:

```yaml
# schema.yml
models:
  - name: monthly_mrr
    columns:
      - name: month
        tests:
          - not_null
          - unique
      - name: ending_mrr
        tests:
          - not_null
          - dbt_utils.accepted_range:
              min_value: 0  # MRR은 절대 음수가 되면 안 됨
```

6. **문서화.** 데이터 사전을 유지한다. 모든 테이블과 컬럼에 대해: 무엇인지, 어디서 오는지, 알려진 주의사항, 누가 소유하는지. 데이터 모델을 이해하는 사람이 한 명뿐이면 버스 팩터가 1이다.

7. **접근 제어.** 모든 사람에게 웨어하우스 쓰기 권한을 주지 마라. 분석가는 변환된 테이블에 대한 읽기 권한을 갖는다. 데이터 파이프라인만 쓰기 권한을 갖는다.

### 네이밍 규칙

어떤 규칙을 선택하는가보다 일관성이 더 중요하다. 하나를 정하고 강제하라.

**이벤트:** `object_action`, snake_case
- `document_created`, `subscription_upgraded`, `invite_accepted`

**속성:** snake_case
- `user_id`, `plan_name`, `created_at`, `is_active`

**테이블:** `schema.object_plural`, snake_case
- `raw.stripe_invoices`, `staging.stg_users`, `analytics.monthly_mrr`

**dbt 모델:**
- Raw: `raw_[source]_[table]` (예: `raw_stripe_invoices`)
- Staging: `stg_[source]_[table]` (예: `stg_stripe_invoices`)
- Intermediate: `int_[description]` (예: `int_monthly_revenue_by_account`)
- Marts: `[metric_or_entity]` (예: `monthly_mrr`, `customer_health_score`)
