## 契約体系の概要

| 文書 | 説明 |
|:---|:---|
| **SOR**（要件定義書） | クライアントが作成し、ビジネス目標と要件を記述する |
| **SOW**（作業範囲記述書） | ベンダーが作成し、スコープ、成果物、スケジュールを定義する |
| **MSA**（基本サービス契約） | 法的なマスター契約であり、一般条件を定義する |
| **PCR**（プロジェクト変更申請） | プロジェクト中のスコープ変更を管理する |
| **見積書** | 料金体系、支払条件、価格オプション |
| **NDA**（秘密保持契約） | 当事者間で共有される機密情報を保護する |
| **SaaS契約** | ホスト型ソフトウェアサービスのサブスクリプション条件 |
| **パートナー／再販契約** | チャネルパートナーシップと再販関係を規定する |

### 1. プロジェクト概要

**1.1 背景：**
Acme Corpは現在、メールとチャットで月間12,000件のカスタマーサポートチケットを処理しています。平均初回応答時間は4.7分です。チケットの62%は定型的な問い合わせです（パスワードリセット、注文状況確認、返品ポリシー）。15名のサポートチームに年間120万ドルのコストがかかっています。Tier 1の問い合わせを処理するAIエージェントを導入し、人間のエージェントを複雑な案件に集中させる必要があります。

**1.2 目標：**

### 2. 要件詳細

**2.1 機能要件**

各要件をユーザーストーリーと受入基準の形式で記述します：

**FR-001：自然言語理解およびインテント認識**
- *私は*顧客として、*自分の問題を自然な言葉で説明したい*。*そうすることで*AIエージェントが正しく理解し、リクエストを適切にルーティングできるようにします。
- 受入基準：
  - 上位20のインテントカテゴリにおけるインテント分類精度 95%以上
  - エンティティ抽出精度 90%以上（注文番号、日付、商品名）
  - 複合インテントメッセージへの対応（例：「注文はどこですか？住所も変更できますか？」）
  - 信頼度スコアが0.7未満の場合、人間のエージェントへの適切なフォールバック

**FR-002：会話コンテキスト管理**
- *私は*顧客として、*メッセージをまたいで会話を続けたい*。*そうすることで*同じことを繰り返さずに済みます。
- 受入基準：
  - 少なくとも20ターンの会話コンテキストを保持
  - 30日以内の同一顧客の過去チケットを参照
  - 会話途中のトピック変更にもコンテキストを失わずに対応

**FR-003：ナレッジベース連携**
- *私は*サポートマネージャーとして、*AIエージェントがナレッジベースから回答を取得するようにしたい*。*そうすることで*回答が正確かつ一貫したものになります。
- 受入基準：
  - 既存のZendeskヘルプセンター記事を取り込み（約500記事）
  - 記事変更後1時間以内に更新
  - 監査証跡として内部ノートにソース記事を引用
  - 矛盾する情報がある場合、最新更新の記事を使用

### 効果的な要件の書き方 — ガイダンス

**すべての目標にSMART基準を適用する：**
- **具体的（Specific）：**「サポートを改善する」ではなく「Tier 1チケット件数を削減する」
- **測定可能（Measurable）：** 数値を含めること。測定できなければ検収もできません。
- **達成可能（Achievable）：** 目標設定前に候補ベンダーと相談する
- **関連性がある（Relevant）：** すべての要件がビジネス目標に紐づいていること
- **期限付き（Time-bound）：**「いつか」ではなく「稼働開始後90日以内」

**ユーザーストーリーの形式：**
```
[役割]として、[アクション]をしたい。そうすることで[ビジネス価値]が得られる。
```

**受入基準のパターン：**
- **Given/When/Then：**「顧客が注文状況について問い合わせたとき、注文番号を提供すると、システムが1秒以内に現在のステータスを返す」
- **ルールベース：**「条件Yが真のとき、システムはXを実行しなければならない（SHALL）」
- **境界条件：** 入力が欠落、不正、または上限を超える場合に何が起こるかを常に明記する

**SOR作成で避けるべきよくある間違い：**
- 問題ではなくソリューションを記述する（ソリューションの提案はベンダーに任せる）
- 非機能要件の省略（パフォーマンス、セキュリティ、可用性）
- 受入基準の未指定（UAT時の紛争につながる）
- 既存システムの記載漏れ（ベンダーが連携の複雑さを後から発見し、コストが増大する）

### スコープ管理 — ガイダンス

**スコープ境界の定義：**
- 包含事項よりも先に除外事項リストを作成すること。含まれていると想定されがちな事項について考えるきっかけになります。
- 要件が曖昧な場合、それはスコープ内として扱われます。表現を厳密にしてください。
- 「新聞テスト」を活用する：ベンダーが何を納品すると約束したかを記者に説明しなければならないとしたら、SOWで明確に伝わりますか？

**スコープクリープへの対処：**
スコープクリープはプロジェクトを崩壊させます。以下のような場合に発生します：
1. クライアントが口頭で「小さな」追加を依頼し、ベンダーが関係維持のためにそれを実施する
2. 要件が曖昧で、クライアントの解釈がベンダーの解釈よりも広い
3. スコープ策定に関与していなかったステークホルダーがプロジェクト途中で新しい要件を提起する

**防止策：**
- SOWに明記されていないすべてのリクエストはPCR（セクション4参照）を通す
- PMは「はい、対応可能です。コストおよびスケジュールへの影響を記載したPCRがこちらです」と言う権限を持つ
- ミーティング中にスコープ変更に同意しない。必ず書面でフォローアップすること。
- 次フェーズに延期するリクエストの「パーキングロット」リストを管理する

**変更管理プロセス：**
1. リクエスト → PCR提出（PCRテンプレート参照）
2. 影響評価 → ベンダーがコスト、スケジュール、リスク影響を見積もり（3営業日）
3. レビュー → 両PMがレビュー
4. 承認／却下 → 両PMが署名（またはスポンサーにエスカレーション）
5. 実行 → SOWを改訂し、スケジュールを更新し、請求を調整

**現実的なスケジュール見積もり：**
- すべてのエンジニアリング見積もりに20%のバッファを追加する。エンジニアは楽観的です。
- 自身の管理外の依存関係を考慮する（クライアントのレビュー、サードパーティAPI、調達）
- クライアントが「6週間で必要」と言っても作業に10週間かかるなら、10週間と伝える。スケジュールを圧縮しても作業は圧縮されません — 品質が圧縮されます。
- フェーズゲート（進行前の承認要求）は無駄な作業を防ぎますが、カレンダー上の時間が増えます。レビューサイクルごとに3〜5営業日を見込んでください。

## 3. MSAの主要条項

基本サービス契約（MSA）は法的な基盤です。個別のSOW、見積書、PCRはMSAのもとで運用されます。本セクションでは条項ごとのガイダンスと条文例を提供します。

### 3.1 サービス

**目的：** 業務の構造とSOWのMSAとの関係を定義する。

**条文例：**
> ベンダーは、両当事者が締結した一つ以上の作業範囲記述書（「SOW」）に記載されたサービスを提供する。各SOWは本契約に組み込まれ、本契約に準拠するものとする。SOWと本契約の間に矛盾がある場合、SOWが特定のMSA条項を明示的に優先すると記載しない限り、本契約の条件が優先する。

**要点：**
- MSAがデフォルトを設定し、SOWは明示的に記載した場合にのみ特定の条件を上書きできる
- SOWには番号を付け、IDで参照する
- 両当事者がSOWに署名するまで業務は認可されない旨の条項を含める

### 3.2 支払条件

**目的：** ベンダーへの支払いの時期と方法、および支払いが行われなかった場合の対応を定義する。

**条文例：**
> クライアントは、未争議の請求書について、受領後30暦日以内に支払うものとする（「Net 30」）。請求書は、該当するSOWに定義されたマイルストーンの完了時に提出する。支払い遅延に対しては、月1.5%または法律で許容される最大利率のいずれか低い方の利息が発生する。支払いが60日を超えて遅延した場合、ベンダーは15日前の書面による通知をもってサービスを中断することができる。

**一般的な支払い構造：**

| 構造 | 適用場面 |
|:---|:---|
| タイムアンドマテリアル（T&M） | スコープが不明確、ディスカバリー／R&D業務 |
| 固定価格 | スコープが明確に定義され、双方がコストの確実性を求める場合 |
| マイルストーンベース | 定義されたチェックポイントで分割払いする固定価格 |
| リテイナー | 継続的なアドバイザリー、保守、フラクショナルロール |
| ハイブリッド | コアスコープは固定価格、変更リクエストはT&M |

**経費ポリシー：**
> 事前承認された合理的な自己負担費用（出張、ホスティング、サードパーティツール）は、領収書に基づき実費で精算される。1件あたり500ドルを超える費用には事前の書面承認が必要。出張はエコノミークラス、宿泊はGSA日当額以内とする。

### 3.3 知的財産

この条項は最も紛争が発生しやすい条項です。正しく定めてください。

**IP譲渡（クライアントがすべてを所有）：**
> 本契約に基づきベンダーがクライアントのために特別に作成したすべての成果物は、法律で認められる範囲において「職務著作」とみなされる。成果物が職務著作に該当しない範囲において、ベンダーは当該成果物に対するすべての権利、権原および利益（知的財産権を含む）をクライアントに取消不能の形で譲渡する。ベンダーは、クライアントの所有権を完全なものとするために合理的に必要な書類に署名する。

**IPライセンス（ベンダーが所有権を保持し、クライアントにライセンスを付与）：**
> ベンダーは成果物に対するすべての権利、権原および利益を保持する。ベンダーは、クライアントの社内業務目的に限り、成果物を使用、改変およびサブライセンスする非独占的、永続的、取消不能、ロイヤリティフリー、全世界を対象とするライセンスをクライアントに付与する。

**それぞれの使い分け：**

| モデル | 適用場面 | クライアントのリスク | ベンダーのリスク |
|:---|:---|:---|:---|
| **譲渡** | カスタムのオーダーメイド開発にクライアントが対価を支払う場合。競争優位性が重要 | 低 — クライアントが所有 | 他のクライアントに再利用できない |
| **ライセンス** | ベンダーが既存プラットフォーム／ツールを適用する場合。クライアントは継続的改善の恩恵を受ける | ベンダーが競合他社にもライセンス供与する可能性、ベンダーへの依存 | 低 — ベンダーがIPを保持 |
| **ハイブリッド** | 最も一般的。カスタム作業 → 譲渡、ベンダーの既存ツール／フレームワーク → ライセンス | 中程度 — カスタムと既存の境界を明確にする必要がある | 中程度 — 既存IPを事前に明確に定義する必要がある |

**既存IP除外条項（譲渡取引では不可欠）：**
> 上記にかかわらず、ベンダーは別紙Bに記載されたベンダーの既存知的財産（「ベンダーIP」）に対するすべての権利を保持する。ベンダーIPが成果物に組み込まれる範囲において、ベンダーは成果物に組み込まれた形でのみベンダーIPを使用する非独占的、永続的、取消不能、ロイヤリティフリーのライセンスをクライアントに付与する。ベンダーは納品前に成果物に組み込まれたベンダーIPを特定するものとする。

**オープンソースの開示：**
> ベンダーは、納品前に成果物に組み込まれたすべてのオープンソースコンポーネント（ライセンス種別を含む）を開示するものとする。ベンダーは、クライアントの事前書面承認なしに、コピーレフトライセンス（例：GPL、AGPL）のオープンソースコンポーネントを組み込んではならない。

### 3.4 秘密保持

**条文例：**
> 「機密情報」とは、本契約に関連して一方の当事者（「開示者」）から他方の当事者（「受領者」）に対して、口頭、書面、電子的手段、または検査により開示されたすべての非公開情報を意味し、技術データ、営業秘密、事業計画、顧客リスト、価格情報、財務情報、および本契約の条件を含むがこれらに限定されない。受領者は、(a) 本契約の目的にのみ機密情報を使用し、(b) 事前の書面承認なく第三者に開示せず（ただし、知る必要があり、少なくとも本条項と同等の秘密保持義務に拘束される従業員、請負業者、アドバイザーを除く）、(c) 自社の機密情報に対して使用するのと少なくとも同程度の注意をもって保護するものとする（ただし、合理的な注意を下回らないものとする）。これらの義務は、本契約終了後3年間存続する。

**標準的な除外事項：**
> 機密情報には以下は含まれない：(a) 受領者の過失によらず公知となった情報、(b) 開示前に受領者が既に知っていた情報、(c) 機密情報を参照せずに受領者が独自に開発した情報、(d) 制限なく第三者から正当に受領した情報。

### 3.5 データ保護およびセキュリティ

**データ所有権：**
> クライアントデータ（クライアントが提供したデータ、またはサービス提供の過程でクライアントのエンドユーザーから収集されたデータと定義）は、クライアントの財産であり、引き続きクライアントの財産とする。ベンダーは、サービス提供以外の目的でクライアントデータを使用してはならない。契約終了時、ベンダーは30日以内にすべてのクライアントデータを返却または破棄し、書面で破棄を証明するものとする。

**データ処理補遺（DPA）— GDPR/CCPAの要点：**

ベンダーがクライアントに代わって個人データを処理する場合、DPAが必要です。主な規定は以下のとおりです：

| DPA規定 | 対象事項 |
|:---|:---|
| **役割** | クライアントはデータ管理者、ベンダーはデータ処理者 |
| **処理指示** | ベンダーはクライアントからの文書化された指示に基づいてのみデータを処理する |
| **再委託先** | ベンダーは再委託先を一覧化する。クライアントは30日前の通知をもって新しい再委託先に異議を申し立てられる |
| **セキュリティ対策** | ベンダーはTOM（技術的および組織的対策）を実施する — 具体的に記載すること |
| **侵害通知** | ベンダーはデータ侵害を発見後72時間以内にクライアントに通知する（GDPR要件） |
| **データ主体の権利** | ベンダーは、データ主体のアクセス／削除リクエストへの対応においてクライアントを支援する |
| **データ移転** | データが国境を越える場合、法的根拠を明記する（SCC、十分性認定等） |
| **監査権** | クライアントまたは指定の監査人がベンダーのデータ処理活動を監査できる（セクション3.10参照） |
| **削除／返却** | 契約終了時、ベンダーは30日以内にすべての個人データを削除する |

**CCPA固有の追加条項：**
> ベンダーは、サービス提供以外の目的でクライアントの個人情報を販売、共有、または使用してはならない。ベンダーは、CCPAで許可される場合を除き、クライアントの個人情報を他のソースから受領した個人情報と結合してはならない。ベンダーはこれらの制限を理解し遵守することを証明する。

**セキュリティ要件：**
> ベンダーは、SOC 2 Type II基準に準拠した情報セキュリティ管理体制を維持するものとし、アクセス制御（ロールベース、MFA）、転送時の暗号化（TLS 1.3）および保存時の暗号化（AES-256）、ログ記録および監視、脆弱性管理（重大な脆弱性は14日以内にパッチ適用）、年次ペネトレーションテストを含むがこれらに限定されない。ベンダーは、リクエストから10営業日以内に最新のSOC 2 Type IIレポートをクライアントに提供する。

### 3.6 サービスレベル契約（SLA）

**稼働率のコミットメント：**

| ティア | 月間稼働率 | 月間最大ダウンタイム | クレジット |
|:---|:---|:---|:---|
| スタンダード | 99.5% | 約3.6時間 | 月額料金の10% |
| プロフェッショナル | 99.9% | 約43分 | 月額料金の10% |
| エンタープライズ | 99.95% | 約22分 | 月額料金の25% |
| ミッションクリティカル | 99.99% | 約4.3分 | 月額料金の30% |

**応答時間のコミットメント：**

| 重要度 | 定義 | 応答時間 | 解決目標 |
|:---|:---|:---|:---|
| P1 — Critical | サービスが停止、回避策なし | 15分 | 4時間 |
| P2 — High | 主要機能が劣化、回避策あり | 1時間 | 8営業時間 |
| P3 — Medium | 軽微な機能の問題、影響限定的 | 4営業時間 | 2営業日 |
| P4 — Low | 外観上の問題または改善要望 | 1営業日 | 次回リリース |

**SLAクレジットの条文例：**
> ベンダーが特定の暦月にSLAの稼働率を達成できなかった場合、クライアントは以下の計算式に基づくサービスクレジットを受ける権利を有する：（月間稼働率目標 − 実際の稼働率）× 10 × 月額料金。ただし、当該月の月額料金の30%を上限とする。サービスクレジットは、ダウンタイムに対するクライアントの唯一かつ排他的な救済手段とする。クレジットは次回の請求書に適用され、12か月を超えて繰り越すことはできない。

**SLAの除外事項（一般的）：**
- 計画メンテナンス（48時間前の事前通知あり）
- クライアントに起因する問題（例：連携の設定ミス）
- 不可抗力事象
- ベータ版またはプレビュー機能
- サードパーティサービスの障害（例：クラウドプロバイダーのダウンタイム）

### 3.7 保証

**ベンダーの保証：**
> ベンダーは以下を保証する：(a) 本契約を締結する権限を有すること、(b) サービスが一般的に受け入れられている業界標準に合致した専門的かつ適切な方法で提供されること、(c) 成果物が検収後90日間（「保証期間」）、該当するSOWの仕様に実質的に適合すること、(d) ベンダーの知る限り、成果物が第三者の知的財産権を侵害しないこと、(e) ベンダーがサービス提供において適用されるすべての法律を遵守すること。

**クライアントの保証：**
> クライアントは以下を保証する：(a) 本契約を締結する権限を有すること、(b) 本契約に記載された目的のためにベンダーにクライアントデータを提供する権利を有すること、(c) クライアントの成果物の使用が適用される法律に準拠すること。

**保証の免責：**
> 本契約に明示的に定める場合を除き、ベンダーは、商品性、特定目的への適合性、および非侵害の保証を含む、明示または黙示のすべての保証を否認する。

**この免責が重要な理由：** この条項がなければ、UCC（米国の場合）または同等の法律に基づく黙示の保証により、ベンダーが交渉された範囲を超えるクレームにさらされる可能性があります。上記の明示的な保証こそが実際に約束している内容です。

### 3.8 補償

補償とは、特定のシナリオにおいて一方の当事者が他方の損失を填補することに同意することを意味します。最も交渉が集中する条項の一つです。

**ベンダーによる補償（ベンダーがクライアントを保護）：**
> ベンダーは、以下に起因する第三者の請求、損害、損失、責任、費用（合理的な弁護士費用を含む）から、クライアントおよびその役員、取締役、従業員、代理人（「クライアント被補償者」）を防御し、補償し、免責する：(a) ベンダーの本契約違反、(b) ベンダーの過失または故意の不正行為、(c) 成果物が第三者の知的財産権を侵害するとの請求、(d) ベンダーの適用法違反。成果物が侵害していると認定された場合、またはその合理的な可能性がある場合、ベンダーは自己の費用で、(i) クライアントが引き続き使用する権利を取得する、(ii) 非侵害となるよう修正する、(iii) 機能的に同等のもので置き換えるものとする。上記のいずれも商業的に合理的でない場合、ベンダーは該当するSOWを終了し、侵害した成果物に対して支払われた料金を返金することができる。

**クライアントによる補償（クライアントがベンダーを保護）：**
> クライアントは、以下に起因する第三者の請求から、ベンダーおよびその役員、取締役、従業員、代理人を防御し、補償し、免責する：(a) クライアントデータ（クライアントデータが第三者の権利を侵害するとの請求を含む）、(b) 本契約で認められていない方法でのクライアントの成果物の使用、(c) クライアントの過失または故意の不正行為。

**補償手続き：**
> 被補償者は、(a) 書面にて速やかに補償者に通知し（通知の不履行は、実際の不利益が生じた範囲を除き補償者を免責しない）、(b) 防御および和解の単独の管理を補償者に委ね、(c) 補償者の費用で合理的に協力するものとする。補償者は、事前の書面承認なく、被補償者に義務を課す形での和解を行ってはならない。

### 3.9 責任制限

**目的：** 両当事者の最大の金銭的エクスポージャーに上限を設定する。この条項がなければ、10万ドルのプロジェクトが理論上数百万ドルの損害賠償につながる可能性があります。

> (A) 第[X]条に基づく補償義務、(B) 秘密保持義務の違反、(C) 知的財産権の侵害、および(D) 重過失または故意の不正行為を除き：

>
> (ii) 本契約に基づく各当事者の累計総責任額は、請求に先立つ12か月間にクライアントが本契約に基づいて支払ったまたは支払うべき総料金（「責任上限」）を超えないものとする。

**一般的な責任上限の構造：**

| 構造 | 例 | 適用場面 |

- 前金は最低20% — クライアントのコミットメントを確認し、初期費用をカバーする
- 支払いを成果物に紐づける。カレンダーの日付ではなく — 支払日に合わせて急ぐプレッシャーを避ける
- 最終支払いは20%以下にする — 稼働後にクライアントが連絡を絶った場合、大きな残高を追いかけたくない
- プロジェクトが6か月超の場合、マイルストーンゲート付きの月次請求を検討する — 双方のキャッシュフローが安定する
- マイルストーン支払いには必ず「みなし検収」条項を含める — クライアントがX日以内に回答しない場合、マイルストーンは検収され、支払いが発生する

**値引きについて：**
- 料金単価を値引きしない — スコープを値引きする。「Salesforce連携を外せば8万ドル以内に収まります」とすれば料金表の整合性が保たれる。
- ボリュームディスカウント／複数年ディスカウントは問題ない：「2年間のサポート契約を締結いただければ導入費用を10%割引します」
- 初回取引で15%を超える値引きは絶対にしない。後から逃れられないアンカーを設定してしまいます。

## 6. SaaSサブスクリプション契約

本セクションでは、ホスト型SaaS製品の主要条件を取り上げます。ベンダーの観点から構成していますが、エンタープライズ購入者が修正要求する可能性のある点についても注記しています。

### 6.1 サブスクリプション条件

**条文例：**
> クライアントは、該当するオーダーフォームに記載されたティアおよび価格で[製品名]サービス（「サービス」）を購読する。サブスクリプション期間はオーダーフォームに記載された発効日に開始し、指定された期間（「サブスクリプション期間」）存続する。いずれかの当事者がサブスクリプション期間の終了60日前までに書面で更新しない旨の通知を行わない限り、サブスクリプションは当時の定価で初期サブスクリプション期間と同等の連続した期間で自動更新される。ただし、更新価格は前期間の価格から7%を超えて増額しないものとする。

**購入者の注意点：** 値上げ付き自動更新は標準的ですが、強引になる場合があります。エンタープライズ取引では更新価格の上限（5〜7%）と長めの通知期間（90日）を交渉してください。

### 6.2 利用規約（Acceptable Use Policy）

一般的な制限事項：

> クライアントは以下を行ってはならない：(a) サービスのサブライセンス、再販、またはタイムシェア、(b) 第三者の権利を侵害する資料の保存または送信にサービスを使用すること、(c) サービスのリバースエンジニアリング、逆コンパイル、または逆アセンブルの試行、(d) マルウェアの送信またはサービス拒否攻撃の実行にサービスを使用すること、(e) 文書化されたレート制限の超過、または他のユーザーのパフォーマンスを低下させる方法での自動化されたアクセス、(f) 適用法で禁止された目的でのサービスの使用、(g) 競合製品の構築にサービスを使用すること。

### 6.3 データ所有権およびポータビリティ

**条文例：**
> クライアントはクライアントデータに対するすべての権利、権原および利益を保持する。ベンダーは、サービス提供のために処理する限定的な権利を除き、クライアントデータに対する権利を取得しない。終了または更新しない場合、ベンダーは30日間、標準的な機械可読形式（CSV、JSON、またはSQLダンプ）でクライアントデータをエクスポート可能にする。この期間の経過後、ベンダーはすべてのクライアントデータを削除し、書面で削除を証明するものとする。

**利用データ／テレメトリ：**
> ベンダーは、サービスの改善、ベンチマーク、およびマーケティングの目的で、匿名化された集計利用データ（「サービスデータ」）を収集することができる。サービスデータはクライアントまたは個々のユーザーを特定するものであってはならない。クライアントは書面によるリクエストでサービスデータの収集をオプトアウトすることができる。

### 6.4 SLAおよび稼働率保証

**稼働率の計算：**

**クレジットスケジュール：**

|:---|:---|
| 99.9% – 99.5% | 10% |
| 99.5% – 99.0% | 20% |
| 99.0%未満 | 30% |

**クレジット手続き：**
> クライアントはダウンタイム発生から30日以内にクレジットを申請しなければならない。クレジットは次回の請求書に適用され、ダウンタイムに対するクライアントの唯一の救済手段とする。クレジットは月額料金の30%を超えて発生せず、現金や返金に交換することはできない。

## 7. パートナー／再販契約

### 7.1 任命および地域

**条文例：**
> ベンダーは、パートナーを別紙Aに定義された地域における非独占的[正規再販業者／紹介パートナー／付加価値再販業者]として任命する。パートナーは地域外の顧客を積極的に勧誘してはならない。この任命はベンダーが地域内で直接販売したり、他のパートナーを任命したりすることを妨げるものではない。

**地域モデル：**

| モデル | 説明 | 適用場面 |
|:---|:---|:---|
| 非独占、地域制限なし | パートナーはどこでも販売可能。ベンダーも同様 | 初期段階のパートナープログラムのデフォルト |
| 非独占、指定地域 | パートナーは特定地域に注力するが、ベンダーもその地域で販売可能 | 直販体制のある確立された市場 |
| 独占地域 | 当該地域ではこのパートナーのみが販売可能 | コミットメントのある現地パートナーが必要な新興市場 |

### 7.2 コミッション構造

**一般的なモデル：**

| モデル | 仕組み | 例 |
|:---|:---|:---|
| **紹介料** | パートナーがリードを紹介し、ベンダーがクローズ | 初年度ACVの10〜20% |
| **再販マージン** | パートナーが割引価格で購入し、定価で販売 | 定価から20〜30%の割引 |
| **レベニューシェア** | 顧客の支払いに対する継続的な割合 | 継続収益の15〜25%（継続的） |
| **段階的コミッション** | 販売量に応じてレートが上昇 | 最初の10万ドルは15%、10万〜50万ドルは20%、50万ドル超は25% |

**条文例（レベニューシェア）：**
> ベンダーは、パートナーが獲得した顧客の各サブスクリプション開始から24か月間（「コミッション期間」）、純収益の20%に相当するコミッションをパートナーに支払うものとする。「純収益」とは、返金、クレジット、および税金を差し引いたサブスクリプション料金の総額を意味する。コミッションは月次で計算され、月末から30日以内に支払われる。コミッション期間の経過後、パートナーが更新を積極的にサポートしない限り、更新に対する追加コミッションは支払われない。

**主な検討事項：**
- 「パートナーが獲得した」の定義を正確にする — ファーストタッチ？ラストタッチ？ディール登録？
- ディール登録はチャネルコンフリクトを防止する — パートナーがベンダーに案件を登録し、ベンダーが既存関係がないことを確認する
- 更新に対するコミッションは大きな交渉ポイント。パートナーは永続的なコミッションを望みますが、ベンダーは1〜2年に限定したいと考えます。

### 7.3 義務

**パートナーの義務：**
- 認定セールス担当者および認定テクニカルリソースを各1名以上維持する
- 90日以内にベンダーのパートナー認定プログラムを修了する
- 四半期の最低予約目標$[X]を達成する（未達の場合は独占権／ティアステータスを喪失）
- 四半期ごとのビジネスレビューにベンダーと参加する
- ベンダー承認のマーケティング資料のみ使用する
- パイプラインと顧客フィードバックを月次で報告する

**ベンダーの義務：**
- セールストレーニングおよび認定プログラムを提供する
- デモ環境およびサンドボックスアカウントを提供する
- 共同マーケティング基金（MDF）：ベンダーは承認されたアクティビティに対し、四半期あたり$[X]を上限にパートナーのマーケティング支出と同額をマッチングする
- 単一窓口としてパートナーアカウントマネージャーを割り当てる
- 登録されたディールを90日間保護する
- 登録されたディールでパートナーの価格を下回らない

### 7.4 競業禁止

**条文例：**
> 本契約の期間中およびその後12か月間、パートナーはベンダーのサービスと同じ市場カテゴリにおいて直接競合する製品を開発、販売、または再販してはならない。この制限は、本契約の対象となる特定の製品カテゴリにのみ適用され、パートナーの一般的な事業活動を制限するものではない。

**実務上の注意点：** パートナー契約における競業禁止条項は議論を呼びます。CRMツールを販売するパートナーは、すべてのCRMツールの販売停止には同意しません。競業禁止の範囲を特定の名前の競合製品に絞るか、完全に削除して経済的インセンティブに頼る方が現実的です。

### 7.5 認定および品質

> パートナーは、(a) 年次再認定トレーニングの修了、(b) ベンダーの契約後アンケートで測定される顧客満足度評価4.0/5.0以上の維持、(c) ベンダーのパートナー基準に定められた期間内での顧客エスカレーションの解決により、認定ステータスを維持するものとする。パートナーが認定を維持できない場合、またはパートナーの顧客満足度評価が2四半期連続で3.5/5.0を下回った場合、ベンダーは本契約を停止または終了することができる。

## 8. NDAテンプレート

### 8.1 相互NDA

両当事者が機密情報を共有する場合（B2B営業やパートナーシップではほぼ常にそうです）に相互NDAを使用します。

**条文例：**

> **相互秘密保持契約**
>
> 本相互秘密保持契約（「本契約」）は、[日付]付で[当事者A]と[当事者B]（それぞれ「当事者」、総称して「両当事者」）との間で締結される。
>
> **1. 目的。** 両当事者は潜在的なビジネス関係（「目的」）を検討しており、機密情報の共有が必要となる場合がある。
>
> **2. 機密情報。**「機密情報」とは、いずれかの当事者が他方に対して、口頭、書面、電子的手段、または検査により開示したすべての非公開情報であって、機密として指定されたもの、または情報の性質および開示の状況から合理的な人が機密であると理解するものを意味する。機密情報には、ソースコード、アルゴリズム、製品ロードマップ、顧客リスト、価格情報、財務データ、事業戦略、および本契約の条件を含むがこれらに限定されない。
>
> **3. 除外事項。** 機密情報には以下は含まれない：(a) 受領者の過失によらず公知となった、または公知となる情報、(b) 書面による記録で証明される開示前に受領者が既に知っていた情報、(c) 開示者の機密情報を参照せずに受領者が独自に開発した情報、(d) 開示の制限なく第三者から正当に受領した情報。
>
> **4. 義務。** 各受領者は、(a) 目的のためにのみ機密情報を使用し、(b) 知る必要があり、かつ本条項と少なくとも同等の秘密保持義務に拘束される従業員、請負業者、およびアドバイザーへの開示に限定し、(c) 自己の機密情報に対して用いるのと少なくとも同程度の注意をもって（ただし合理的な注意を下回らない）機密情報を保護し、(d) 不正使用または開示を発見した場合は速やかに開示者に通知するものとする。
>
> **5. 法令に基づく開示。** 受領者が法律、規制、または法的手続きにより機密情報の開示を強制される場合、受領者は、(a) （法的に許容される範囲で）開示者に速やかに事前の書面通知を行い、(b) 保護命令の取得に向けた開示者の取り組みに協力し、(c) 法的に必要な最小限の情報のみを開示するものとする。
>
> **6. 期間。** 本契約は上記の日付から2年間有効とする。秘密保持義務は本契約の満了または終了後3年間存続する。
>
> **7. 資料の返還。** 要求時または終了時に、受領者はすべての機密情報を速やかに返却または破棄し、書面で破棄を証明するものとする。
>
> **8. ライセンスの不付与。** 本契約のいかなる規定も、特許、著作権、営業秘密、またはその他の知的財産権に基づくライセンスを受領者に付与するものではない。
>
> **9. 義務の不発生。** 本契約はいずれの当事者に対しても、さらなる契約または取引を締結する義務を課すものではない。
>
> **10. 救済手段。** 各当事者は、本契約の違反が金銭的損害賠償では不十分な回復不能の損害を引き起こす可能性があることを認める。開示者は、他の利用可能な救済手段に加え、差止命令を含む衡平法上の救済を求める権利を有する。
>
> **11. 準拠法。** 本契約は、抵触法の原則にかかわらず、[州／国]の法律に準拠し、同法に基づいて解釈されるものとする。

### 8.2 片務NDA

一方の当事者のみが機密情報を開示する場合に使用します。上記と同じ構造ですが、一方の当事者のみが義務を負います。以下のような場面で一般的です：
- ベンダーのデモで見込み客がデータを共有するがベンダーは共有しない場合
- デューデリジェンス（買収者が対象企業の帳簿を確認する場合）
- 請負業者のオンボーディング

### 8.3 NDAガイダンス

**合理的な期間：**
- NDAの期間（情報を共有する期間）：1〜3年
- 秘密保持義務の存続期間（情報を秘密に保つべき期間）：NDA満了後2〜5年
- 営業秘密：無期限に保護すべき、または「営業秘密としての性質を保つ限り」

**保護すべき対象：**
- ソースコード、アルゴリズム、アーキテクチャ
- 顧客リスト、価格情報、財務データ
- 製品ロードマップおよび戦略
- 未公開の特許出願
- トレーニングデータおよびモデル
- 取引条件

**合意すべきでないもの：**
- 無期限の秘密保持義務（営業秘密を除く）
- 競合他社との取引を妨げるNDA条項
- NDAに埋め込まれた競業禁止条項（これらは別途、慎重に交渉すべき）
- すべての「メンタルインプレッション」の追跡と返還を要求するNDA — これは執行不能であり、レッドフラグ
- NDAにおける違約金条項 — 標準的なNDAは衡平法上の救済に依拠する

**署名者：**
- 両当事者が署名すべき。一方の当事者のみが署名したNDAは片務的な約束に過ぎない。
- 署名者に権限があることを確認する。営業担当者には会社を拘束する権限がない場合がある。
- 日付を入れる。日付のないNDAは執行が困難になる。

## 9. 交渉プレイブック

### 9.1 交渉の対象を理解する

交渉に入る前に、契約の類型を分類します：

| 契約タイプ | レバレッジを持つ側 | 一般的な所要期間 |
|:---|:---|:---|
| エンタープライズSaaS（自社がベンダー） | 購入者 — 選択肢がある | 2〜6か月 |
| カスタム開発SOW | バランス型 — ベンダーは専門知識を持ち、買い手は予算を持つ | 2〜4週間 |
| パートナーシップ／再販 | バランス型 — 双方が相手を必要とする | 4〜8週間 |
| NDA | どちらでもない — 迅速に完了すべき | 1〜3日 |
| 調達主導（RFP対応） | 購入者 — 受け入れるか辞退するか | 2〜3か月 |

### 9.2 企業規模別の一般的な修正要求

**スタートアップ（従業員1〜50名）：**
- 保険要件を満たせないことが多い — 代替案を交渉する（例：SOC 2 Type II取得中）
- 無制限の責任または上限なしの補償には同意しない
- 法務チームがないことが多い — 創業者が署名することを想定する
- 通常、IPおよび条件は柔軟で、ボイラープレートについて争わない
- 修正要求ポイント：契約金額未満の責任上限、過度な保険要件、費用保護のない監査権

**中堅企業（従業員50〜500名）：**
- 法務顧問はいるが外部事務所を利用することがある（レビューが遅くなる）
- 標準的な修正要求：IP譲渡（ライセンスを希望）、責任上限、データ処理条件
- 通常、サイバー保険に加入しているが、エンタープライズの要件を満たさない場合がある
- 修正要求ポイント：既存ツールのIP譲渡、無制限の補償、米国以外の企業に対する米国準拠法の限定

**エンタープライズ（従業員500名以上）：**
- 標準契約テンプレートを持つ調達部門 — 自社の書式を使うよう求める
- 法務レビューに最低4〜8週間
- 標準的な修正要求：すべて。修正要求が仕事です。
- 主な攻防点：どちらの書式か（自社のMSA vs 先方のMSA）、責任上限（IP／データについて上限なしを希望）、監査権（広範な権利を希望）、準拠法（自社の本拠地法を希望）、データ処理（厳格なDPA要件がある）
- 取引に価値があるなら相手方の書式を受け入れるが、商業条件は厳しく交渉する

### 9.3 譲歩すべき点と譲らない点

**気軽に譲歩：**
- 準拠法（同一国内であれば — 米国の州間の法的差異はサービス契約では重大ではないことが多い）
- コミュニケーションおよび報告要件（遵守が容易）
- 成果物およびドキュメントのフォーマット
- 保険証書への追加被保険者記名（日常的な作業）
- 勧誘禁止（相互的かつ合理的な期間）
- 保証に関する文言の外観上の変更

**交渉するが柔軟に：**
- 責任上限の金額（契約金額の1倍を主張し、求められれば2倍を受け入れる）
- 支払条件（Net 30が標準。エンタープライズならNet 45を受け入れる。Net 60以上は抵抗する）
- 補償の範囲（相互を目指し、正当な理由があれば非対称を受け入れる）
- 監査権（年次監査を受け入れ、無制限／無通告には抵抗する）
- 任意解約の通知期間（60〜90日が合理的）
- 自動更新条件（価格上限があれば受容可能）

**譲らない — 絶対に譲歩しない：**
- 無制限の責任（破産の原因になり得る）
- 既存のツールおよびフレームワークのIP譲渡
- 広範な競業禁止条項（他のクライアントへのサービス提供を続ける必要がある）
- 「不満」を理由とした無期限の支払い留保権（すべての紛争には解決メカニズムが必要）
- 上限のないペナルティ条項
- 一方的な契約修正権（「ベンダーはいつでも条件を変更できる」）
- 企業規模に対して商業的に利用可能または合理的な範囲を超える保険の維持要件

### 9.4 法務レビュープロセス

**法務に送る前の社内チェックリスト：**
1. まず契約書全体を自分で読む。40ページのMSAを、懸念点のサマリーなしに法務に送ってはいけない。
2. 交渉したい条項をハイライトする（知的財産、責任制限、補償、データ処理）
3. 通常ではない、または非標準的な条項を記録する
4. コンテキストを提供する：取引金額、戦略的重要性、相手方との関係
5. 期限を設定する：「[日付]までに署名が必要です。[日付 − 1週間]までにレビューをお願いします。」

**タイムラインの目安：**

| 契約タイプ | 自社の法務レビュー | 相手方の法務レビュー | 合計 |
|:---|:---|:---|:---|
| NDA | 1〜2日 | 1〜2日 | 3〜5日 |
| SOW（既存MSA下） | 2〜3日 | 3〜5日 | 1〜2週間 |
| MSA（自社書式） | 完了済み（自社テンプレート） | 2〜4週間 | 2〜4週間 |
| MSA（相手方書式） | 1〜2週間 | 1〜2週間（修正返送） | 4〜8週間 |
| エンタープライズ調達 | 2〜3週間 | 4〜8週間 | 6〜12週間 |

### 9.5 交渉テクニック

**ベンダー向け：**
- 自社の書式を先に送る。テンプレートを使用する側が構造的な優位性を持つ — 相手方はあなたのデフォルトに対して変更を加える立場になる。
- すべての譲歩にはリクエストを添える。「責任上限はお受けしますので、支払条件は当社の条件で合意いただけますか。」
- 相手方が修正を加えてきたら、一つずつ対応しない。回答をまとめて一つの包括的な修正版を送る。
- 法務が遅延している場合、ビジネススポンサーにエスカレーションする。「開始の準備はできています。法務に契約書を送って6週間です。前に進めていただけますか？」
- ハッタリを使わない。「これは破談の条件です」と言うなら、本当にそうでなければならない。

**購入者向け：**
- まずベンダーの標準契約を入手し、修正を加える。それにより、ベンダーがどれだけ合理的かがわかる。
- 競合入札をレバレッジとして使う — ただし、実際にある場合のみ
- ベンダーがスタートアップの場合、財務の安定性を確認する。ベンダーが倒産すれば、素晴らしい契約も無意味です。
- データポータビリティと退出権を最初から交渉する。署名前が最もレバレッジが大きい。
- ベンダーがDPAに抵抗する場合、それはレッドフラグ。基本的なデータ保護要件を満たせないベンダーとは契約しない。

### 10.2 インシデント対応手順

**重要度の定義：**

| 重要度 | 定義 | 例 |
|:---|:---|:---|
| **P1 — Critical** | 本番システムが停止、またはコア機能が完全に利用不能。回避策なし。 | AIエージェントが応答しない。全会話が失敗。データ侵害が検出された |
| **P2 — High** | コア機能が著しく劣化。回避策はあるが持続不可能。 | 応答レイテンシが10秒超。精度が70%未満に低下。Zendeskとの連携障害 |
| **P3 — Medium** | 非クリティカルな機能が損なわれている。ユーザーへの影響は限定的。 | ダッシュボードメトリクスが更新されない。1つのインテントカテゴリの誤分類。KB索引作成の遅延 |
| **P4 — Low** | 軽微な問題または改善要望。即時のビジネス影響なし。 | UIの外観上の問題。ドキュメントの誤り。機能リクエスト |

**インシデントのライフサイクル：**

```
検知 → トリアージ → 確認 → 調査 → 解決 → ポストモーテム
```

1. **検知：** クライアントがサポートチャネル経由で報告、またはベンダーの監視が自動検知
2. **トリアージ：** 重要度（P1〜P4）を割り当て、担当者を指定し、エスカレーションマトリクスに従いステークホルダーに通知
3. **確認：** SLA枠内で応答。問題を確認し、チケット番号を提供
4. **調査：** 根本原因分析、ステータス更新の頻度（P1は30分ごと、P2は2時間ごと、P3/P4は日次）
5. **解決：** 修正をデプロイし、クライアントに確認し、ステータスページを更新
6. **ポストモーテム：** すべてのP1およびP2インシデントについて — 5営業日以内に予防措置を含む根本原因分析を書面で作成。クライアントと共有。

### 10.3 エスカレーションパス

|:---|:---|:---|:---|

### 10.4 メンテナンスウィンドウ

**計画メンテナンス：**
> ベンダーは、指定されたメンテナンスウィンドウ（毎週日曜日 午前2時〜午前6時 ET）中に計画メンテナンスを実施するものとする。ベンダーは、標準的なメンテナンスについては48時間前、30分を超えるダウンタイムが見込まれるメンテナンスについては7日前に通知するものとする。メンテナンスウィンドウ中の計画メンテナンスは稼働率SLAの計算に含まれない。

**緊急メンテナンス：**
> ベンダーは、重大なセキュリティ脆弱性またはシステム安定性の問題に対処するため、予定されたウィンドウ外で緊急メンテナンスを実施することができる。ベンダーは可能な限りの事前通知（最低2時間）を行い、メンテナンス時間を必要最小限に抑えるものとする。

**変更凍結期間：**
> 以下の期間中は、緊急修正を除き、デプロイまたはシステム変更を行わない：(a) クライアントの四半期末（最終3営業日）、(b) 主要なショッピングホリデー（ブラックフライデーからサイバーマンデー、12月20日〜1月2日）、(c) クライアントが30日以上前の通知により指定した期間。

### 10.5 サポート終了（EOL）ポリシー

**条文例：**
> ベンダーは、サービスの主要バージョンのサポート終了（「サポート終了通知」）の12か月前までに書面で通知するものとする。通知期間中、ベンダーは (a) セキュリティパッチおよびクリティカルバグ修正の提供を継続し、(b) 後継バージョンへの移行支援を無償で提供し（20時間まで）、(c) データエクスポート機能を維持する。EOL日以降、廃止されたバージョンにはアップデート、パッチ、サポートは提供されない。

**バージョンサポートのライフサイクル：**

| フェーズ | 期間 | 内容 |
|:---|:---|:---|
| **アクティブ** | 現行バージョン | フル機能開発、すべてのバグ修正、フルサポート |
| **メンテナンス** | 現行 − 1バージョン（次のメジャーリリース後12か月） | セキュリティパッチ、クリティカルバグ修正、サポート |

| **EOL** | 延長期間終了後 | パッチなし、サポートなし、SLAなし |

## 付録：文書間の関係

```
MSA（すべてを統括）
 ├── NDA（独立文書の場合が多いが、MSAの別紙にもなり得る）
 ├── DPA（MSAの別紙）
 ├── SOW #1（SORを参照）
 │    ├── 見積書 #1
 │    ├── PCR #1a
 │    └── PCR #1b
 ├── SOW #2
 │    └── 見積書 #2
 ├── サポート契約（稼働後）
 ├── SaaSサブスクリプション契約（該当する場合）
 └── パートナー／再販契約（該当する場合）
```

**締結順序：**
1. NDAを最初に（機密情報を共有する前に）
2. MSA＋DPA（法的関係を確立）
3. SOW＋見積書（具体的な作業を定義し価格設定）
4. PCRは必要に応じて（変更を管理）
5. サポート契約（稼働開始前に）

**文書の優先順位（矛盾がある場合）：**
1. PCR（最新の改訂が優先）
2. SOW（個別条件が一般条件に優先）
3. MSA（一般条件）
4. DPA（データに関する条件。データに関してはMSAに優先する場合がある）
5. 別紙および添付文書
