# パート5：データドリブン成長ループ

## 1. ノーススターメトリック

**定義：** プロダクトが顧客に提供するコアバリューを最もよく測定する、唯一の指標。

**要件：**
- 全社（プロダクト、エンジニアリング、マーケティング、営業）がこの指標を中心に結集する
- 顧客価値を直接反映する――売上やバニティメトリクスではない
- 週次または日次で測定可能である
- プロダクトの変更に対して合理的な期間内に反応する
- 長期的なビジネスの健全性（売上、リテンション）を予測できる

### ノーススターの選び方

**プロセス：**
1. 顧客がプロダクトから本当の価値を得る瞬間をすべてリストアップする。
2. その価値交換を最もよく表すアクションまたは成果を見つける。
3. この指標の変動が、6〜12ヶ月の過去データにおける売上とリテンションの変動と相関していることを検証する。
4. ストレステストを行う：「この指標が2倍になっても売上が動かなかったとしたら、それでも問題ないか？」答えがYesなら、別の指標を選ぶ。
5. 全チームリーダーの合意を得る。エンジニアリングが信じなければ、適切に計装されない。

**よくある間違い：**
- 売上をノーススターに選ぶこと。売上はアウトプットだ。ノーススターは売上を生み出すインプットであるべきだ。
- 動かせないものを選ぶこと。「総登録ユーザー数」は増え続けるだけで、プロダクトの健全性については何も教えてくれない。
- 抽象的すぎるものを選ぶこと。「顧客の幸福度」は測定できない。「週間タスク完了数」なら測定できる。
- 四半期ごとに変えること。一つに決めたら、最低1年はコミットする。変え続けていると、改善する力が身につかない。
- CEOだけで決めること。プロダクトとデータに最も近い人たちが議論に参加すべきだ。

### SaaSカテゴリ別ノーススターの例

| カテゴリ | プロダクト例 | ノーススターメトリック | この指標が有効な理由 |
|:---|:---|:---|:---|
| AI動画ツール | Synthesia, Loom AI | 週間エクスポート動画数 | エクスポート＝アウトプットを共有するほどの価値を得た |
| AI会議アシスタント | Otter, Fireflies | 週間生成議事録サマリー数 | サマリー生成＝コアバリューの提供 |
| コラボレーションプラットフォーム | Notion, Slack | 週間アクティブコラボチーム数 | ユーザーではなくチーム――コラボレーションこそが価値 |
| CRM | HubSpot, Salesforce | 週間更新案件数 | 案件の更新＝営業チームが価値を得ている |
| DevTools / API | Stripe, Twilio | 週間プロダクションAPIコール数 | 本番利用＝本物のインテグレーション、本物の価値 |
| プロジェクト管理 | Linear, Asana | 週間タスク完了数 | 作成ではなく完了――完了こそが価値 |
| アナリティクス / BI | Amplitude, Metabase | 非アナリストによる週間クエリ実行数 | データ利用の民主化＝プラットフォームの粘着性 |
| セキュリティ | Snyk, Datadog | 週間脆弱性/課題解決数 | 検出ではなく解決――修正にこそ価値がある |
| 電子署名 / リーガル | DocuSign, PandaDoc | 週間署名済み文書数 | 署名＝取引の完了 |
| カスタマーサポート | Intercom, Zendesk | 週間解決済み会話数 | 解決＝顧客の問題が解決された |
| デザインツール | Figma, Canva | コラボレーターと共有された週間デザイン数 | 共有＝人に見せるに値するデザイン |
| メールマーケティング | Mailchimp, Brevo | 開封率20%以上の週間キャンペーン送信数 | 品質でゲート：送信だけでは価値ではない |
| データインテグレーション | Fivetran, Airbyte | 週間同期成功数 | 同期成功＝データパイプラインが価値を提供している |

### 先行指標 vs 遅行指標

ノーススターは多くの場合**遅行**指標だ――すでに起きたことを示している。それを予測し、影響を与えるために**先行**指標が必要になる。

| タイプ | 定義 | 例（「週間エクスポート動画数」の場合） |
|:---|:---|:---|
| **遅行** | すでに発生した結果 | 今週エクスポートされた動画数 |
| **先行** | 遅行指標を予測する早期シグナル | 新規プロジェクト作成数、テンプレート閲覧数、初回レンダリング完了数 |

モデルを構築する：先行指標 → ノーススター → ビジネス成果（売上、リテンション）。

### ノーススターのインプットメトリクスへの分解

ノーススターを**メトリクスツリー**に分解する。各チームがツリーの枝を担当すべきだ。

例：ノーススター＝「週間アクティブコラボチーム数」

```
週間アクティブコラボチーム数
├── 今週新規アクティベートされたチーム（グロース/マーケティング）
│   ├── サインアップ → チーム作成率
│   ├── 新規チームあたりの招待送信数
│   └── 招待承諾率
├── 今週リテインされた既存チーム（プロダクト）
│   ├── D7チームリテンション率
│   ├── チームあたり週間利用機能数
│   └── チーム間コラボレーションイベント数
└── 今週リアクティベートされたチーム（ライフサイクル/CRM）
    ├── ウィンバックメール開封率
    ├── キャンペーンからのリアクティベーション率
    └── リアクティベートされたチームの最終アクティビティからの経過時間
```

各チームが自分の枝を担当する。週次でツリーをレビューし、どの枝がパフォーマンス不足かを特定する。

## 2. AARRRファネル指標

| ステージ | コアクエスチョン | 主要指標 |
|:---|:---|:---|
| **獲得（Acquisition）** | ユーザーはどこから来るのか？ | チャネルUV、サインアップ数、チャネル別CAC |
| **アクティベーション（Activation）** | コアバリューを体験したか？ | 「Ahaモーメント」コンバージョン、オンボーディング完了率、Time to Value |
| **リテンション（Retention）** | 再び戻ってくるか？ | D1/D7/D30リテンション、機能リテンション、週間アクティブ率 |
| **収益（Revenue）** | 支払ってくれるか？ | 無料→有料コンバージョン、ARPU、LTV、拡張収益 |
| **紹介（Referral）** | 推薦してくれるか？ | NPS、K-factor、リファラルコンバージョン率 |

### B2B SaaS ステージ別ベンチマーク

以下はB2B SaaSの中央値と上位四分位のデータだ。具体的な数値はセグメント、価格帯、GTM戦略によって異なる。

#### 獲得

| 指標 | 中央値 | 上位四分位 | 備考 |
|:---|:---|:---|:---|
| Webサイト → サインアップ率 | 2-5% | 8-12% | PLGモーション向け。セールス主導はさらに低い |
| ブレンデッドCAC（SMB） | $200-500 | <$150 | フルロード：人件費 + ツール + 広告費 |
| ブレンデッドCAC（ミッドマーケット） | $2,000-5,000 | <$1,500 | |
| ブレンデッドCAC（エンタープライズ） | $10,000-30,000 | <$8,000 | 長い営業サイクルが数値を押し上げる |
| サインアップに占めるオーガニック比率 | 30-40% | >60% | オーガニック比率が高い＝複利的成長 |

#### アクティベーション

| 指標 | 中央値 | 上位四分位 | 備考 |
|:---|:---|:---|:---|
| サインアップ → アクティベート | 20-30% | 40-60% | 「アクティベート」＝Ahaモーメントを完了 |
| Time to Value | 1-3日 | <1時間 | 最良のPLGプロダクトは数分で価値を提供する |
| オンボーディング完了率 | 40-60% | >80% | ステップを1つ追加するたびに20%離脱する |

#### リテンション

| 指標 | 中央値 | 上位四分位 | 備考 |
|:---|:---|:---|:---|
| Month 1リテンション | 40-50% | >60% | ロゴではなくユーザー数ベース――アカウント単位は別指標 |
| Month 3リテンション | 25-35% | >45% | ほとんどのプロダクトがここでユーザーを失う |
| Month 12リテンション | 15-25% | >35% | 15%未満ならグロースの問題ではなくプロダクトの問題 |
| ロゴリテンション（年間） | 85-90% | >95% | B2Bの契約がここで有利に働く |
| Net Dollar Retention | 100-110% | >130% | トップクラスはチャーン以上に拡張する |

#### 収益

| 指標 | 中央値 | 上位四分位 | 備考 |
|:---|:---|:---|:---|
| 無料 → 有料コンバージョン | 2-5% | 7-12% | フリーミアム。無料トライアルのコンバージョンはもっと高い（15-25%） |
| トライアル → 有料（カード不要） | 10-15% | 20-30% | カード必須にすると40-60%に上がるがサインアップは減る |
| 月額 → 年額コンバージョン | 15-25% | >40% | 年額に15-20%割引を適用してキャッシュフローを改善 |
| ARPU成長率（YoY） | 5-10% | >20% | プライシング最適化と拡張 |

#### 紹介

| 指標 | 中央値 | 上位四分位 | 備考 |
|:---|:---|:---|:---|
| NPS | 30-40 | >50 | B2B SaaSの平均はB2Cより低い |
| K-factor | 0.1-0.3 | >0.5 | K > 1はバイラル。ほぼすべてのB2Bプロダクトでこれは達成できない |
| リファラルからの新規ユーザー比率 | 10-20% | >35% | 低CACチャネル。簡単にできるよう投資すべき |

### ボトルネックの特定と改善

ファネルで最大のボトルネックは、最も多くの絶対数のユーザーを失っている箇所だ（割合ではない）。10,000ユーザーからの50%離脱は、100ユーザーからの70%離脱よりも深刻だ。

**獲得のボトルネック症状：**
- LTVに対してCACが高い
- サインアップの60%以上を有料チャネルに依存
- ターゲット市場でのブランド認知度が低い

**改善策：** コンテンツ/SEOに投資する（回収まで6〜12ヶ月）、リファラルループを構築する、ランディングページを最適化する、チャネルの多角化を試みる。同じ有料チャネルにさらに投下するだけでは解決しない。

**アクティベーションのボトルネック症状：**
- サインアップ → Ahaモーメントのコンバージョンが25%未満
- Time to Valueが24時間超
- サインアップするがオンボーディングを完了しないユーザーがいる

**改善策：** オンボーディングを短縮する（ステップの削減、アカウント設定の後回し）、サンプルデータで事前設定する、トリガーベースのオンボーディングメールを送る、リテインされたユーザーとチャーンしたユーザーの初回セッション行動を比較して本当のAhaモーメントを特定する。

**リテンションのボトルネック症状：**
- Month 1からMonth 3の間で急激な離脱
- コホートの週次機能利用率が低下
- ログインはするがコアアクションを実行しないユーザーがいる

**改善策：** 再エンゲージメントのためのライフサイクルメール、未活用機能へのアプリ内プロンプト、習慣化（通知、ダイジェスト）、チャーンしたユーザーに直接話を聞く――実際に電話する。

**収益のボトルネック症状：**
- 無料ユーザーがいつまでも無料のまま
- 拡張収益が低い
- 値上げに対する感度が高い

**改善策：** 無料プランの制限を引き締める（利用量ベースの制限は機能ゲートよりも効果的）、従量課金を導入する、自然なフリクションポイントでアップグレードプロンプトを表示する、年額割引を提供する。

**紹介のボトルネック症状：**
- リテンションは良いのにNPSが低い
- リファラルプログラムの存在が知られていない
- リファラルフローのステップが多すぎる

**改善策：** プロダクトに共有機能を組み込む（共有リンク、ブランド入りエクスポート）、紹介者と被紹介者の両方にインセンティブを提供する、リファラルアクションを文脈に即したものにする（設定ページではなく、成功体験の直後に促す）。

### ステージ別ダッシュボードの例

各ステージに3〜5の指標を備えた専用ダッシュボードを用意する。以下が出発点だ。

**獲得ダッシュボード：**
- 日次/週次サインアップ数（全体およびチャネル別）
- チャネル別CAC（月次更新）
- Webサイトトラフィック → サインアップコンバージョン率
- コンバージョン率上位10ランディングページ
- チャネルミックスの推移（オーガニック vs 有料 vs リファラル vs ダイレクト）

**アクティベーションダッシュボード：**
- サインアップ → Ahaモーメントファネル（ステップごとの離脱）
- Time to Valueの分布（ヒストグラム）
- ステップ別オンボーディング完了率
- 獲得チャネル別アクティベーション率（どのチャネルが高インテントユーザーを連れてくるか？）
- サインアップコホート別アクティベーション率（週次推移）

**リテンションダッシュボード：**
- コホートリテンションテーブル（行＝サインアップ週、列＝サインアップからの経過週数）
- 週間アクティブユーザー推移
- 機能リテンション（週ごとの各主要機能の利用率）
- チャーン率の推移（月次）
- 復帰率（以前チャーンしたユーザーの再訪）

**収益ダッシュボード：**
- MRRウォーターフォール（新規 + 拡張 - 縮小 - チャーン）
- ARPU推移（月次）
- 無料→有料コンバージョン率の推移
- チャネル別LTV/CAC比率
- プランティア別収益

**紹介ダッシュボード：**
- 週間リファラル招待送信数
- リファラルコンバージョン率（招待 → サインアップ → アクティベート）
- K-factorの推移
- NPS推移（四半期）
- リファラル経由の新規サインアップ比率

## 3. 主要指標の定義

### 獲得指標

| 指標 | 定義 | 計算式 | 健全な範囲 |
|:---|:---|:---|:---|
| **CAC** | 顧客獲得コスト | 総セールス＆マーケティング費用 / 新規獲得顧客数 | ACVに依存 |
| **LTV** | 顧客生涯価値 | ARPU × 粗利率 × (1 / 月次チャーン率) | CACの3倍以上 |
| **LTV/CAC** | 獲得効率 | LTV / CAC | 3:1〜5:1 |
| **CAC回収期間** | 獲得コストの回収にかかる月数 | CAC / (ARPU × 粗利率) | 18ヶ月未満（SMB：12ヶ月未満） |

**LTV計算に関する注意：** シンプルな計算式（ARPU × 顧客寿命）は粗利を無視するためLTVを過大評価する。ARPUが月額$100でも粗利率が70%であれば、月あたりの貢献は$100ではなく$70だ。必ず粗利調整後のLTVを使用すること。

**LTV/CACに関する注意：** 5:1を超える比率は通常、効率的なのではなく成長への投資が不足していることを意味する。機会を逃している。

### アクティベーション指標

| 指標 | 定義 | 発見方法 |
|:---|:---|:---|
| **Ahaモーメント率** | コアバリュー体験を完了したユーザーの割合 | 初回セッションにおけるリテインユーザーとチャーンユーザーのアクションを比較 |
| **Time to Value** | サインアップから最初の価値体験までの時間 | アカウント作成からAhaモーメントイベントまでの中央値 |
| **セットアップ完了率** | 全オンボーディングステップを完了した割合 | オンボーディングフローのファネル分析 |

### リテンション指標

| 指標 | 計算式 | 備考 |
|:---|:---|:---|
| **D1リテンション** | 2日目にアクティブなユーザー / 1日目にサインアップしたユーザー | デイリーユースのプロダクトにより関連性が高い |
| **D7リテンション** | 8日目にアクティブなユーザー / 1日目にサインアップしたユーザー | ほとんどのSaaSにおける重要な変曲点 |
| **D30リテンション** | 31日目にアクティブなユーザー / 1日目にサインアップしたユーザー | 長期リテンションの強力な予測因子 |
| **ロゴリテンション** | (期首顧客数 - チャーン顧客数) / 期首顧客数 | 月次または年次 |
| **Net Dollar Retention** | (期首MRR + 拡張 - 縮小 - チャーン) / 期首MRR | PMFを測る最も優れた単一指標 |

### 収益指標

| 指標 | 定義 | 計算式 |
|:---|:---|:---|
| **MRR** | 月間経常収益 | 全サブスクリプション収益の合計（年間契約は月額に正規化） |
| **ARR** | 年間経常収益 | MRR × 12 |
| **ARPU** | ユーザーあたり平均収益 | MRR / 有料顧客総数 |
| **ACV** | 年間契約価値 | 新規契約の平均年間取引額 |
| **NRR / NDR** | 純収益リテンション / Net Dollar Retention | 下記セクションを参照 |

### 高度な指標

#### 回収期間

**概要：** 顧客獲得コストを回収するまでの月数。

**計算式：** CAC / (ARPU × 粗利率)

**LTV/CACよりも重要な理由：**
- LTVは将来の予測だ。過去と同じ未来を仮定している。回収期間は実際の現在の数値を使う。
- キャッシュが重要だ。LTV/CACが5:1でも回収期間が36ヶ月で、18ヶ月後にキャッシュが尽きるなら意味がない。
- アクション可能だ。ARPUの向上（プライシング）、CACの削減（効率化）、粗利率の改善（インフラコスト）で回収期間を短縮できる。

**ベンチマーク：**
| セグメント | Good | Great |
|:---|:---|:---|
| SMB SaaS | 12ヶ月未満 | 6ヶ月未満 |
| ミッドマーケットSaaS | 18ヶ月未満 | 12ヶ月未満 |
| エンタープライズSaaS | 24ヶ月未満 | 18ヶ月未満 |

回収期間が24ヶ月を超えている場合、成長のためにキャッシュを燃やしていることになる。それはベンチャーキャピタルの支援がなければ成り立たない。

#### Quick Ratio

**概要：** MRR成長の効率を測定する。失う1ドルに対してどれだけの新規収益を生み出しているか。

**計算式：** (新規MRR + 拡張MRR) / (チャーンMRR + 縮小MRR)

**ベンチマーク：**
| Quick Ratio | 解釈 |
|:---|:---|
| <1 | 縮小している。すべてを止めてチャーンを修正せよ。 |
| 1-2 | 成長が遅い。チャーンが新規収益の大部分を食い潰している。 |
| 2-4 | 健全な成長。損失を補填しつつ、さらに上積みしている。 |
| >4 | 優秀。効率的に成長している。 |

Quick Ratioが4ということは、チャーンで失う$1ごとに、新規＋拡張で$4を獲得していることを意味する。トップクラスのB2B SaaS企業は4以上で運営している。

**注意：** 新規MRR（多くの新規顧客）で構築された高いQuick Ratioは、拡張MRR（既存顧客がより多く支払う）で構築されたものほど持続可能ではない。拡張のほうがコストが低く、粘着性が高い。

#### 粗利率

**概要：** 収益からサービス提供コストを引いた額を、収益に対する割合で表したもの。

**計算式：** (収益 - COGS) / 収益

**SaaSにおけるCOGSに含まれるもの：**
- ホスティング / インフラコスト（AWS, GCP, Azure）
- サードパーティAPIコスト（OpenAI, Twilio等）
- カスタマーサポートチームの人件費（直接サポート。アカウント管理は含まない）
- DevOps / SREの人件費（議論の余地あり――含める場合も含めない場合もある）
- 決済処理手数料（Stripeの2.9% + $0.30）

**COGSに含まれないもの：**
- エンジニアリングの人件費（R&D）
- セールス＆マーケティング
- G&A（家賃、法務、会計）

**ベンチマーク：**
| タイプ | 粗利率 |
|:---|:---|
| 従来型SaaS | 75-85% |
| AI/MLヘビーなSaaS | 50-70%（GPUコストは現実的） |
| 従量課金型SaaS | 60-75% |
| トップクラス | >80% |

**重要な理由：** 粗利率は、収益の1ドルのうちどれだけが成長、R&D、利益に充てられるかを決定する。粗利率80%の企業は12ヶ月の回収期間を許容できる。粗利率50%の企業はそうはいかない――6ヶ月以下で回収する必要がある。

AIヘビーなSaaS企業はこの指標を注視すべきだ。LLMのAPIコール1回ごとにマージンが削られる。AI機能の運用コストがその機能による増分収益を上回っているなら、成長のストーリーに見せかけたマージン問題を抱えている。

#### Rule of 40

**概要：** 成長と収益性を組み合わせた指標。SaaS企業は急成長しているか、利益を出しているか、理想的にはその両方であるべきという考え方。

**計算式：** 売上成長率（YoY %） + フリーキャッシュフローマージン（%）

**例：**
| 企業 | 成長率 | FCFマージン | Rule of 40スコア |
|:---|:---|:---|:---|
| アーリーステージスタートアップ | 150% | -80% | 70 ✓ |
| グロースステージ | 60% | -10% | 50 ✓ |
| 成熟SaaS | 20% | 25% | 45 ✓ |
| 苦戦中の企業 | 15% | 5% | 20 ✗ |

**ベンチマーク：**
- 20未満：低パフォーマンス。より速く成長するかコストを削減する必要がある。
- 20-40：許容範囲だがインパクトは弱い。
- 40以上：健全。高品質SaaSの最低ラインがここ。
- 60以上：エリート。これを維持できる企業はごく少数。

**注記：** EBITDAマージンをFCFマージンの代わりに使う人もいる。FCFのほうが正直だ。設備投資と株式報酬を含むからだ。

#### Net Dollar Retention (NDR)

**概要：** 新規顧客を完全に除外した上で、既存顧客ベースからどれだけの収益をリテインし拡張しているか。

**計算式：** (期首MRR + 拡張MRR - 縮小MRR - チャーンMRR) / 期首MRR

**例：** 既存顧客のMRRが$100Kで月をスタートした。$15K拡張し、$3K縮小し、$5Kがチャーンした。NDR = ($100K + $15K - $3K - $5K) / $100K = 107%。

**ベンチマーク：**
| NDR | 解釈 |
|:---|:---|
| <90% | 深刻な問題。プロダクトに粘着性がないか、プライシングが利用に応じてスケールしていない。 |
| 90-100% | チャーンを補填しているが拡張していない。 |
| 100-110% | 良好。既存顧客がわずかに成長している。 |
| 110-130% | 優秀。強力な拡張収益。 |
| >130% | トップクラス。Snowflake、Datadog、Twilioの領域。 |

NDRが100%を超えるということは、新規顧客がゼロでも成長できることを意味する。これはSaaSにおける最も強力な成長エンジンだ。

**NDRを改善する方法：**
- 従量課金（顧客の成長に伴い自然に支払いが増える）
- シートベースプライシングと簡単なシート追加
- 顧客のスケールに応じてアンロックされるティア別機能
- 隣接プロダクトのクロスセル
- 拡張に特化したCSMチームの配置

#### Magic Number

**概要：** セールス効率を測定する――セールス＆マーケティングに投じた1ドルあたりどれだけの新規収益を生み出しているか。

**計算式：** (当四半期ARR - 前四半期ARR) / 前四半期セールス＆マーケティング費用

**ベンチマーク：**
| Magic Number | 解釈 |
|:---|:---|
| <0.5 | 非効率。支出をスケールする前に営業プロセスかPMFを修正せよ。 |
| 0.5-0.75 | まずまずだが改善の余地あり。 |
| 0.75-1.0 | 健全。支出のスケールを開始してよい。 |
| >1.0 | 非常に効率的。この勢いに全力投資せよ。 |

Magic Numberが0.75ということは、セールス＆マーケティングに投じた$1が、その四半期に$0.75の新規ARRを生み出していることを意味する。そのARRは経常的に発生するため、投資のLTVはもっと高くなる。

#### Burn Multiple

**概要：** 新規ARR 1ドルを生み出すためにどれだけのキャッシュを燃やしているか。

**計算式：** 純キャッシュバーン / 純新規ARR

**ベンチマーク：**
| Burn Multiple | 解釈 |
|:---|:---|
| <1x | 驚異的。バーン以下のコストでARRを生み出している。稀。 |
| 1-1.5x | 優秀。効率的な成長。 |
| 1.5-2x | アーリーステージとしては許容範囲。 |
| 2-3x | コスト高。投資家から質問が出始める。 |
| >3x | 持続不可能。成長を買っている状態。 |

Burn Multipleは2022年以降、VCがますます重視するようになった指標だ。あらゆるコストを度外視した成長よりも、効率的な成長が重要視されている。

#### チャネル別CAC回収期間

ブレンデッドCACだけを計算してはいけない。チャネルごとに分解すべきだ。ブレンデッドの数値は大きな差異を隠してしまう。

**分解の例：**

| チャネル | CAC | 回収期間 | LTV/CAC | 判定 |
|:---|:---|:---|:---|:---|
| オーガニック検索 | $80 | 2ヶ月 | 15:1 | 積極的にスケールすべき |
| コンテンツマーケティング | $150 | 4ヶ月 | 8:1 | さらに投資すべき |
| Google Ads（ブランド） | $200 | 5ヶ月 | 6:1 | 維持 |
| Google Ads（ノンブランド） | $600 | 15ヶ月 | 2:1 | 最適化するか削減 |
| LinkedIn Ads | $900 | 22ヶ月 | 1.5:1 | エンタープライズ向けでなければ削減 |
| アウトバウンドSDR | $1,200 | 18ヶ月 | 2.5:1 | ミッドマーケット以上の案件のみ |
| カンファレンススポンサー | $2,500 | 30ヶ月以上 | 1:1 | 通常は価値がない |

要点：「ブレンデッドCACが$400」という数字に意味はない。$80のチャネルもあれば$2,500のチャネルもある。安いチャネルに倍賭けし、高いチャネルは削減または最適化する。

## 5. 成長会議のベストプラクティス

### 頻度
週次。隔週では遅すぎる――勢いが失われる。週次で議論することが足りないなら、実験のベロシティが低すぎる。

### 参加者
- プロダクトマネージャー（アジェンダのオーナー）
- グロース/マーケティングリード
- エンジニアリングリード（チーム全体ではなく、実現可能性とタイムラインについて話せる1名）
- データアナリスト / アナリティクスオーナー
- 任意：デザインリード、セールスリード（セールスアシストPLGモーションの場合）

4〜6名に抑える。それ以上だと意思決定が進まない。

### 会議アジェンダテンプレート（45分）

```
成長会議 — [日付]

1. スコアカード（5分）
   ノーススターメトリックとインプットメトリクスの目標対実績レビュー
   ┌──────────────────────┬─────────┬─────────┬────────┐
   │ 指標                 │ 目標    │ 実績    │ 傾向   │
   ├──────────────────────┼─────────┼─────────┼────────┤
   │ ノーススター         │         │         │ ↑↓→    │
   │ インプット指標1      │         │         │ ↑↓→    │
   │ インプット指標2      │         │         │ ↑↓→    │
   │ インプット指標3      │         │         │ ↑↓→    │
   └──────────────────────┴─────────┴─────────┴────────┘
   各指標を赤/黄/緑で色分け。緑については議論しない。

2. 実験結果（15分）
   完了した各実験について：
   - 仮説
   - 結果（信頼区間付き）
   - 判定：Ship / Kill / Iterate
   - 次のアクションと担当者

3. パイプラインレビュー（10分）
   現在実施中の実験：
   - ステータス（必要サンプルサイズの何%か）
   - 完了予定日
   - 課題（汚染、バグ、トラフィック不足）

4. 新規実験（10分）
   バックログをレビューし、今週ローンチする上位1〜3件を選定。
   各実験について：
   - ICEスコア
   - 担当者
   - ローンチ日
   - 予想期間

5. ブロッカー＆意思決定（5分）
   今すぐ決定が必要な事項。
   2分以上の議論が必要なら、別途対応する。
```

### データの提示方法

**ルール：**
1. 数値だけでなくトレンドを示す。「アクティベーション率は32%」では使えない。「アクティベーション率は過去4週間で38%から32%に低下した」なら行動に移せる。
2. 実験結果には必ず信頼区間を示す。「バリアントBはコンバージョンを12%改善した」では不十分。「バリアントBはコンバージョンを12%改善した（95% CI：3%〜21%、p=0.01）」なら出荷すべきかどうか判断できる。
3. 同時期の前週/前月/前年と比較する。季節性はB2Bにも存在する。
4. あらゆるものをセグメント化する。「全体のリテンションは45%」だと、エンタープライズのリテンションが80%でSMBが20%という事実が隠れてしまう。
5. アノテーションを使用する。すべてのチャートにプロダクトのローンチ、価格変更、障害、マーケティングキャンペーンをマークする。

### 意思決定フレームワーク

| シグナル | 判定 |
|:---|:---|
| 実験が統計的有意に達し、効果がポジティブかつ有意義 | Ship |
| 実験が有意に達したが、効果がポジティブでも微小（相対リフト2%未満） | メンテナンスコストに見合わない可能性が高い。メンテナンス不要でない限りKill。 |
| 実験が有意に達し、効果がネガティブ | Kill。そこから学ぶ。 |
| 実験を全期間実施したが有意差なし | 判定不能。効果があったとしても検出できないほど小さい。Killするか、より多くのトラフィックで再設計する。 |
| 今週、指標が急落した | 直ちに調査する。データの問題か、バグか、実際のプロダクト変更か？ |

### エスカレーション基準

以下の場合は成長会議の外にエスカレーションする：
- ノーススターメトリックが原因不明で週次10%以上低下
- 出荷した実験がコア指標のリグレッションを引き起こした
- CAC回収期間が2ヶ月連続で18ヶ月（SMB）または24ヶ月（ミッドマーケット）を超過
- Net Dollar Retentionが2ヶ月連続で95%を下回った
- アクティベーション率が20%を下回った

## 6. 成長実験フレームワーク

### ICEスコアリング

| 次元 | 説明 | スコア（1-10） |
|:---|:---|:---|
| **Impact** | ノーススターメトリックへの潜在的影響 | 1＝微小、10＝ゲームチェンジャー |
| **Confidence** | 仮説への確信度（データ、ユーザーリサーチ、過去の実験に基づく） | 1＝直感、10＝強力なデータ |
| **Ease** | 実装の容易さ | 1＝数ヶ月の工数、10＝今日デプロイ可能 |

個別にスコアリングし、その後、食い違いについて議論する。議論こそがスコアよりも価値がある。

**代替手段：RICEスコアリング**（バックログにリーチが大きく異なるアイテムが含まれる場合）

|:---|:---|
| **Reach** | 四半期で何人のユーザーに影響するか？ |
| **Impact** | 各ユーザーにどの程度の影響を与えるか？（0.25＝最小、0.5＝低、1＝中、2＝高、3＝甚大） |
| **Confidence** | どの程度確信しているか？（100%＝高、80%＝中、50%＝低） |
| **Effort** | 構築に必要な人週 |

### 統計的有意性

**基礎知識：**
- **有意水準（α）：** 偽陽性の確率。標準はα = 0.05（真の差がないのに勝者を宣言してしまう確率が5%）。
- **統計的検出力（1-β）：** 真の効果を検出する確率。標準は80%。重要な実験では90%。
- **最小検出可能効果量（MDE）：** 意味があると判断する最小の改善幅。正直になること――1%の相対リフトはおそらくエンジニアリングの工数に見合わない。

**サンプルサイズの計算方法：**

2つの比率を比較する両側検定の場合：

```
n = (Z_α/2 + Z_β)² × (p₁(1-p₁) + p₂(1-p₂)) / (p₂ - p₁)²

各記号の意味:
  n = バリアントあたりのサンプルサイズ
  Z_α/2 = α = 0.05のとき1.96
  Z_β = 検出力80%のとき0.84、検出力90%のとき1.28
  p₁ = ベースラインコンバージョン率
  p₂ = トリートメント適用時の期待コンバージョン率
```

**クイックリファレンステーブル（α = 0.05、検出力 = 80%）：**

| ベースライン率 | MDE（相対） | バリアントあたりのサンプルサイズ |
|:---|:---|:---|
| 2% | 20% (2.0% → 2.4%) | 47,000 |
| 2% | 50% (2.0% → 3.0%) | 6,300 |
| 5% | 10% (5.0% → 5.5%) | 58,000 |
| 5% | 20% (5.0% → 6.0%) | 13,000 |
| 10% | 10% (10% → 11%) | 26,000 |
| 10% | 20% (10% → 12%) | 5,800 |
| 20% | 10% (20% → 22%) | 10,000 |
| 20% | 20% (20% → 24%) | 2,400 |
| 50% | 10% (50% → 55%) | 3,100 |

2〜4週間でこれらの数値に到達するトラフィックがない場合、より大きなMDE（大胆な変更のみテストする）を設定するか、別の手法を使う必要がある。

統計学の学位がなくても大丈夫。以下のツールを使用すればよい：
- Evan Millerのサンプルサイズ計算機（無料、シンプル）
- PostHog内蔵の実験計算機
- GrowthBookのベイズ分析（サンプルサイズの扱いは異なるが統計的に妥当）

### よくあるA/Bテストの落とし穴

1. **結果を途中で覗き見する。** 毎日チェックして良さそうに見えたら止めると、偽陽性率は30%を超える場合がある。固定のサンプルサイズにコミットするか、逐次検定法（常に有効なp値やベイズ法など）を使用する。

2. **テスト期間が短すぎる。** 曜日効果を考慮するため、最低1営業週間（月〜日）が必要。理想的には2週間。テストが3日で「勝った」場合は疑いを持つべきだ。

3. **テスト期間が長すぎる。** 4〜6週間経っても有意差に達しないなら、効果はおそらく問題にならないほど小さい。判定不能として先に進む。長期間のテストは外的イベントによるバイアスが蓄積する。

4. **多重比較を考慮しない。** 5つの指標をテストすると、α = 0.05の下で純粋な偶然により1つが有意になる。ボンフェローニ補正（α / 指標数）を使用するか、テスト開始前に主要指標を1つ指定する。

5. **サンプル比率の不一致（SRM）。** A/Bの分割は正確に50/50（または設定した比率）であるべきだ。48/52になっていたら、割り当てロジックに問題がある。結果を分析する前にサンプルサイズのカイ二乗検定を行う。

6. **セグメント分析をしない。** 全体ではフラットな実験でも、新規ユーザーで+20%、既存ユーザーで-15%ということがありえる。主要セグメントは必ず確認する。

7. **生存者バイアス。** 実験が初日の行動を変えるが指標が30日間で測定される場合、早期にチャーンしたユーザーが分析から除外される。Intention-to-treat分析を使用する。

8. **バリアントが多すぎる。** 追加のバリアントそれぞれにコントロールと同じサンプルサイズが必要。A/B/C/D/Eテストには、A/Bテストの5倍のトラフィックが必要。2〜3バリアントに抑える。

### 多変量テスト

複数の変更を同時にテストし、それらの間の相互作用を理解したい場合に多変量テストを使用する。

**例：** ランディングページの見出しとCTAボタンの両方をテストしたい場合。
- 見出し：A1（オリジナル）、A2（新）
- CTA：B1（オリジナル）、B2（新）
- 組み合わせ：A1B1、A1B2、A2B1、A2B2（4バリアント）

**多変量テストを使うべき場合：**
- 十分なトラフィックがある（各組み合わせに対してフルサンプルサイズが必要）
- 変更同士が相互作用する可能性がある（見出しとCTAが連動する）
- ページやフローを総合的に最適化したい

**多変量テストを使うべきでない場合：**
- トラフィックが少ない――有意差に達しない
- 変更が独立しており相互作用しない
- 迅速な結果が欲しい

### A/Bテストをすべきでない場合

すべてにA/Bテストが必要なわけではない。以下の場合はテストしない：

1. **変更が明らかに正しい場合。** 壊れたサインアップフォームの修正にテストは不要。直すだけだ。
2. **十分なトラフィックがない場合。** 有意差に達するまで3ヶ月かかるなら、定性的な手法（ユーザーインタビュー、セッション録画）を使う。
3. **変更が不可逆な場合。** 価格変更、大規模なアーキテクチャ変更、ブランド変更は簡単にロールバックできない。他の検証手法（アンケート、支払い意欲調査、ベータプログラム）を使う。
4. **影響が小さすぎる場合。** フッターリンクの色を変えることにテストのオーバーヘッドは不要だ。
5. **PMF前の段階にある場合。** PMFを模索中は、ユーザーフィードバックに基づいて大胆かつ迅速に変更すべきであり、ボタンの色の2週間テストを実施すべきではない。学習速度 > 統計的厳密さ。

## 7. コホート分析

コホート分析は、ユーザーを開始時期（サインアップ週または月）でグループ化し、その行動を時間経過とともに追跡する。リテンション、収益、プロダクトの健全性を理解するための最も重要な分析手法だ。

### コホートリテンションテーブルの構築方法

行＝コホート（サインアップ週または月）。列＝サインアップからの経過期間。値＝コホートのうちまだアクティブな割合。

**例：週次ユーザーリテンションコホート**

| コホート | Week 0 | Week 1 | Week 2 | Week 3 | Week 4 | Week 8 | Week 12 |
|:---|:---|:---|:---|:---|:---|:---|:---|
| 1月1日 (n=500) | 100% | 45% | 35% | 30% | 28% | 22% | 20% |
| 1月8日 (n=520) | 100% | 48% | 38% | 33% | 31% | 25% | 22% |
| 1月15日 (n=480) | 100% | 52% | 42% | 38% | 35% | 29% | — |
| 1月22日 (n=550) | 100% | 55% | 44% | 40% | 37% | — | — |
| 1月29日 (n=600) | 100% | 58% | 47% | 42% | — | — | — |

### 着目すべきポイント

1. **カーブの平坦化。** 健全なリテンションカーブは初期の急落後に平坦化する。同じ率で下がり続けるなら、リテンションがあるのではなく、緩やかなチャーン問題を抱えている。

2. **時間の経過に伴うコホートの改善。** 上記の例では、各コホートが前のコホートよりもリテンションが良い（1月29日のWeek 1＝58% vs 1月1日のWeek 1＝45%）。これはプロダクトの改善が効果を上げていることを意味する。

3. **「崖」の位置。** 最大の単一期間の落ち込みはどこか？Week 0からWeek 1で55%のユーザーを失っているなら、アクティベーションの問題だ。Month 3からMonth 6で30%失っているなら、「目新しさの消失」問題かもしれない。

4. **スマイルカーブ。** リテンションカーブが一旦下がってから再び上がることがある。ユーザーが離れた後、プロダクトが恋しくなって戻ってくるという意味だ。ポジティブなシグナルだが、そもそもなぜ離れるのかを理解すべきだ。

### 収益コホートテーブル

リテンションと同じ構造だが、アクティブユーザーの代わりに収益を使用する。さらに多くのことが見えてくる。

| コホート | Month 0 | Month 3 | Month 6 | Month 12 | 収益リテンション |
|:---|:---|:---|:---|:---|:---|
| 2024 Q1 (n=100) | $10K | $9K | $11K | $13K | Month 12で130% |
| 2024 Q2 (n=120) | $14K | $13K | $15K | — | — |
| 2024 Q3 (n=150) | $20K | $19K | — | — | — |

Month 12での収益リテンションが100%を超えているということは、既存顧客が開始時よりも多く支払っていることを意味する――拡張収益がチャーンを上回っている。これが目標だ。

### SQLでコホートテーブルを構築する方法

```sql
-- ユーザーリテンションコホート（PostgreSQL）
WITH cohort AS (
  SELECT
    user_id,
    DATE_TRUNC('week', created_at) AS cohort_week
  FROM users
),
activity AS (
  SELECT
    user_id,
    DATE_TRUNC('week', event_timestamp) AS activity_week
  FROM events
  WHERE event_name = 'core_action'  -- 「アクティブ」はログインではなくコアアクションで定義
)
SELECT
  c.cohort_week,
  COUNT(DISTINCT c.user_id) AS cohort_size,
  COUNT(DISTINCT CASE WHEN a.activity_week = c.cohort_week + INTERVAL '1 week' THEN a.user_id END)::float
    / COUNT(DISTINCT c.user_id) AS week_1_retention,
  COUNT(DISTINCT CASE WHEN a.activity_week = c.cohort_week + INTERVAL '2 weeks' THEN a.user_id END)::float
    / COUNT(DISTINCT c.user_id) AS week_2_retention,
  COUNT(DISTINCT CASE WHEN a.activity_week = c.cohort_week + INTERVAL '4 weeks' THEN a.user_id END)::float
    / COUNT(DISTINCT c.user_id) AS week_4_retention
FROM cohort c
LEFT JOIN activity a ON c.user_id = a.user_id
GROUP BY c.cohort_week
ORDER BY c.cohort_week;
```

### コホート分析の解釈ガイド

| パターン | 意味すること | 対応策 |
|:---|:---|:---|
| 新しいコホートほどリテンションが良い | プロダクトの改善が効果を上げている | イテレーションを続ける。何が変わったかを見つけて強化する |
| 新しいコホートほどリテンションが悪い | 何かが劣化している（品質、オンボーディング、オーディエンスの適合性） | 獲得チャネルが変わっていないか確認。最近のプロダクト変更をレビュー |
| 全コホートが約25%で平坦化 | 約25%のコアオーディエンスがいて、残りの全員を失っている | その25%の特徴を調査する――それがICP（理想顧客像）だ |
| 収益コホートが100%超で拡張 | 拡張収益が健全 | アップセルモーションと従量課金に投資する |
| 収益コホートが80%未満に縮小 | 大幅なダウングレードまたはチャーン | チャーン理由を緊急に調査する。バリューベースプライシングの見直しを検討 |

## 8. チャーン分析

### チャーンの計算

チャーンには2つの種類があり、それぞれ異なる情報を示す。

**ロゴチャーン**（顧客チャーン）：顧客の何%が離脱したか？

```
月次ロゴチャーン率 = 今月の離脱顧客数 / 月初の顧客数
```

**収益チャーン**（MRRチャーン）：収益の何%が離脱したか？

```
月次収益チャーン率 = 今月のチャーンによるMRR損失 / 月初のMRR
```

**グロス vs ネット収益チャーン：**
- **グロス収益チャーン：** 失われた収益のみをカウント（チャーン + 縮小）。常にマイナス。
- **ネット収益チャーン：** 拡張収益を考慮する。マイナスになることがある（これは良いこと――拡張がチャーンを上回っている）。

```
グロス収益チャーン = (チャーンMRR + 縮小MRR) / 期首MRR
ネット収益チャーン = (チャーンMRR + 縮小MRR - 拡張MRR) / 期首MRR
```

**月次 vs 年次チャーン：** 単純に変換してはいけない。月次5%のチャーンは年次60%のチャーンではない――複利で効く。

```
年次チャーン = 1 - (1 - 月次チャーン)^12
月次5% = 1 - (0.95)^12 = 年次46%のチャーン
月次2% = 1 - (0.98)^12 = 年次21.5%のチャーン
```

**ベンチマーク：**

| セグメント | 月次ロゴチャーン | 年次ロゴチャーン |
|:---|:---|:---|
| エンタープライズSaaS | 0.5-1% | 5-10% |
| ミッドマーケットSaaS | 1-2% | 10-22% |
| SMB SaaS | 3-5% | 31-46% |
| コンシューマーSaaS | 5-10% | 46-72% |

### チャーン予測指標

以下のシグナルを使ってチャーン予測モデルを構築する。MLは不要だ――ほとんどのスタートアップでは重み付きスコアリングシステムで十分。

| 指標 | シグナル強度 | 測定方法 |
|:---|:---|:---|
| **ログイン頻度の低下** | 強 | 直近2週間のログイン頻度をその前の2週間と比較 |
| **コア機能利用の減少** | 非常に強 | セッションあたりまたは週あたりのコアアクション数の減少 |
| **サポートチケット（特定の種類）** | 中 | 請求に関する質問、「解約方法」、機能への不満 |
| **契約満了が近い** | 強 | 更新まで30〜60日以内 |
| **チャンピオンの退社** | 非常に強 | LinkedInまたはメールバウンスで職種変更を追跡 |
| **決済失敗** | 強 | SMBにおける非自発的チャーンは全チャーンの20-40% |
| **12ヶ月以上拡張なし** | 中 | 利用が横ばいなら上限に達した可能性 |
| **低いシート利用率** | 強 | 50シート分を支払っているが、アクティブは10のみ |
| **競合の検討** | 非常に強 | G2レビュー活動、競合コンテンツのダウンロード（追跡している場合） |
| **NPSがデトラクター** | 中 | NPS < 6 |

### 早期警戒システム

以下の条件でアラートを自動化する：

**レッドアラート（即座の介入――CSMが顧客に連絡）：**
- コア機能の利用が週次で50%以上低下
- チャンピオンの連絡先がバウンス（退社）
- 顧客が解約リクエストを提出
- 顧客がプランをダウングレード

**イエローアラート（1週間以内にプロアクティブなアウトリーチ）：**
- ログイン頻度が2週間で30%以上低下
- サポート満足度スコアがネガティブ
- NPSの回答が0-6（デトラクター）
- シート利用率が30%未満
- 14日以上ログインなし（週次利用プロダクトの場合）

**グリーンモニタリング（月次チャーン分析でレビュー）：**
- 利用が3ヶ月以上横ばい（成長なし）
- 12ヶ月以上拡張なし
- 契約更新まで60日

### リスクティア別介入プレイブック

**ハイリスク（レッドアラート）：**
1. CSMが24時間以内にパーソナライズされたメールまたは電話で連絡
2. 無料の戦略セッションまたはプロダクトウォークスルーを提案
3. 高額アカウント（ARR $10K超）の場合はマネジメントにエスカレーション
4. チャンピオンが退社した場合、新しいステークホルダーを特定して再オンボーディング
5. 非自発的チャーン（決済失敗）の場合：自動ダニングシーケンス + リトライ3回失敗後に個別フォローアップ

**ミドルリスク（イエローアラート）：**
1. 自動メール：「[機能]をお使いいただいていないようです。簡単なヒントをご紹介します」
2. 未活用機能をハイライトするアプリ内プロンプト
3. 今後のウェビナーやオフィスアワーへの招待
4. 次の自然なタッチポイント（QBR、更新）でCSMがチェックイン

**ローリスク（グリーンモニタリング）：**
1. 四半期ビジネスレビューに含める
2. 関連するケーススタディやプロダクトアップデートを共有
3. プロアクティブな拡張提案（「プランの制限の80%をご利用いただいています。次のステップについてご相談しませんか」）

### 非自発的チャーンの削減

非自発的チャーン（決済失敗）はSMB SaaSの全チャーンの20-40%を占める。修正が最も容易なチャーンだ。

1. **ダニングメール：** 決済失敗後14日間で3〜5通のメールを送信する。支払い方法の更新への直接リンクを含める。脅迫的ではなくフレンドリーに。
2. **アプリ内の支払い更新プロンプト：** 決済が失敗しているユーザーがログインしたら、バナーを表示する――ブロッカーではなく。即座にロックアウトしない。
3. **スマートリトライロジック：** 失敗したチャージを異なる日時にリトライする。火曜日と水曜日の午前中が成功率が高い。
4. **カードアップデーターサービス：** Stripeなどのプロセッサーは期限切れカードを自動更新できる。これを有効にする。
5. **猶予期間：** ダウングレードまたは停止まで7〜14日の猶予を設ける。ほとんどの場合、カードの更新を忘れているだけだ。
6. **年額請求への移行推進：** 顧客を年額請求に移行させる。決済イベントが少ない＝失敗ポイントも少ない。

### アップグレード/ダウングレード分析

月次のフロー分析を構築する：

```

プラン別期首MRR：

```

- 依存していた機能が上位ティアに移動した
- 顧客企業での予算削減
- 支払っている機能を使っていない（バリューギャップ）

### 価格感度テスト

**Van Westendorp価格感度メーター：**

4つの質問で顧客にアンケートする：
1. いくらなら安すぎて品質を疑いますか？（安すぎる）
2. いくらならお買い得だと感じますか？（安い / 良い価値）
3. いくらなら高いと感じ始めるが、それでも検討しますか？（高い）
4. いくらなら高すぎて検討の対象外ですか？（高すぎる）

累積分布をプロットする。交点から以下が得られる：
- **限界的安さの点：** 安すぎる ∩ 高い
- **限界的高さの点：** 高すぎる ∩ 安い
- **最適価格点：** 安すぎる ∩ 高すぎる
- **許容価格帯：** 限界的安さと限界的高さの間

セグメントあたり100〜200人の回答者で実施する。SMBとエンタープライズの回答を混ぜてはいけない――支払い意欲がまったく異なる。

**Conjoint分析：** より厳密な手法。ユーザーに異なる価格の機能バンドルを提示し、ランク付けまたは選択させる。セットアップは多いが、機能レベルの支払い意欲がわかる。ツール：Conjointly, SurveyMonkey。

**A/Bテストによる価格テスト：** 議論の余地があり、リスクもある。訪問者ごとに異なる価格をテストする企業もある。顧客に知られると信頼が壊れる。実施する場合：
- 新規サインアップのみでテストし、既存顧客の価格は決して変更しない
- 価格差を小さく保つ（±20%）
- クレームがあれば低い方の価格を適用する
- より良い方法：純粋な価格ではなくプラン構成（機能、制限）をテストする

## 11. データインフラストラクチャ

### イベントトラッキングの設定

良いアナリティクスは良いデータから始まる。アナリティクスの問題の大半はデータ収集の問題だ。

#### イベントタクソノミー

プロダクト全体で一貫した命名規則を使用する。これはデータに関して行う最も重要な意思決定だ。

**命名規則：** `object_action` のsnake_case。

```
✓ Good:
  project_created
  document_shared
  subscription_upgraded
  invite_sent
  report_exported

✗ Bad:
  createProject         （ケーシングが不統一）
  user clicked button   （曖昧すぎる、スペース）
  ButtonClick           （どのボタン？）
  page_view_homepage    （オブジェクトとページが混在）
```

**すべてのSaaSが追跡すべき標準イベント：**

| カテゴリ | イベント | プロパティ |
|:---|:---|:---|
| **Account** | `account_created` | plan, source, referrer |
| **Account** | `account_upgraded` | from_plan, to_plan, mrr_change |
| **Account** | `account_downgraded` | from_plan, to_plan, mrr_change, reason |
| **Account** | `account_churned` | plan, mrr_lost, reason, tenure_months |
| **Auth** | `user_signed_up` | method (email, google, sso), source, referrer |
| **Auth** | `user_logged_in` | method |
| **Onboarding** | `onboarding_step_completed` | step_name, step_number, total_steps |
| **Onboarding** | `onboarding_completed` | time_to_complete_seconds |
| **Onboarding** | `onboarding_skipped` | step_name |
| **Core action** | `[your_core_action]_completed` | プロダクト固有 |
| **Feature** | `feature_used` | feature_name, context |
| **Collaboration** | `invite_sent` | role, method |
| **Collaboration** | `invite_accepted` | time_to_accept |
| **Billing** | `payment_succeeded` | amount, plan |
| **Billing** | `payment_failed` | reason, retry_count |
| **Engagement** | `page_viewed` | page_name, referrer |
| **Engagement** | `search_performed` | query, results_count |

#### イベントプロパティ

すべてのイベントに以下の標準プロパティを自動的に含めるべきだ（イベントごとではなく、トラッキングSDKの初期化時に設定する）：

```
標準プロパティ（一度設定し、すべてのイベントに送信）：
  user_id          — 内部ユーザーID
  account_id       — B2Bの場合：企業/ワークスペース
  session_id       — セッション分析用
  timestamp        — ISO 8601
  platform         — web, ios, android, api
  app_version      — アプリのバージョン
  plan             — 現在のプランティア
  user_role        — admin, member, viewer
```

#### 実装のヒント

1. **可能な限りサーバーサイドでトラッキングする。** クライアントサイドのトラッキングは広告ブロッカーによりブロックされる（ユーザーの15-30%）。重要なイベント（サインアップ、決済、コアアクション）はサーバーサイドで追跡すべきだ。

2. **トラッキングプランを使用する。** トラッキングを実装する前に、すべてのイベントをスプレッドシートに記載する：イベント名、発火タイミング、プロパティ、どのチームが必要としているか。全ステークホルダーとレビューする。これにより「500イベントあるが半分は何を意味するか誰もわからない」問題を防ぐ。

3. **すべてを追跡しない。** 意思決定と成果を追跡する。マウスの動きをすべて追跡するのではない。30〜50の明確に定義されたイベントが必要であり、500のノイズまみれのイベントではない。

4. **identifyコールを使用する。** ユーザーがログインした際に、ユーザーおよびアカウントのプロパティ（プラン、企業規模、サインアップ日）を含むidentifyコールを送信する。これにより後続のすべてのイベントが充実する。

5. **トラッキングをテストする。** デプロイ前に、すべてのイベントが正しいプロパティで正しく発火することを確認する。アナリティクスツールのデバッグモードを使用する。壊れたトラッキングはトラッキングなしよりも悪い――誤ったデータに基づいて意思決定することになる。

### データウェアハウスの基礎

アナリティクスツールだけでは不十分になったとき（必ずそうなる）、データウェアハウスが必要になる。

**ウェアハウスを構築すべきタイミング：**
- ツール間のデータを結合する必要がある（Stripe + プロダクトアナリティクス + CRM）
- アナリティクスツールのUIでは質問に答えられない
- アナリティクスツールの保持期間を超えた過去データが必要
- Metabase/Supersetでカスタムダッシュボードを構築したい
- SQLを書きたいデータアナリストがいる

**スタートアップ向けの選択肢：**

| ウェアハウス | 長所 | 短所 | コスト |
|:---|:---|:---|:---|
| **PostgreSQL（本番DB）** | すでに持っている、無料 | 本番環境で重いクエリを実行しない | $0（リードレプリカ：$10-50/月） |
| **BigQuery** | サーバーレス、月1TBまで無料、無限にスケール | Googleエコシステムへのロックイン | ほとんどのスタートアップは無料〜安価 |
| **ClickHouse** | 分析クエリが超高速、オープンソース | BigQueryより運用負荷が高い | Cloud：$200/月〜、セルフホスト：無料 |
| **Snowflake** | 複雑なワークロードに最適、エコシステムが充実 | 高価、複雑な料金体系 | $500/月〜 |
| **DuckDB** | ローカルで稼働、プロトタイピングに最適 | 共有ウェアハウスではない | 無料 |

**ARR $5M未満のほとんどのスタートアップ向け：** PostgreSQLのリードレプリカまたはBigQueryを使用する。考えすぎないこと。

### スタートアップ向けETLパイプライン

ETL = Extract（ソースからデータを取得）、Transform（クリーニングと整形）、Load（ウェアハウスに投入）。

**現代的なアプローチ：ELT**（Extract, Load, Transform）。まず生データをロードし、ウェアハウス内でSQLを使って変換する。こちらのほうがシンプルで柔軟だ。

**推奨スタック：**

```
ソース:              ETL/EL:           ウェアハウス:       変換:             BI:
┌─────────────┐     ┌──────────┐     ┌────────────┐     ┌──────────┐     ┌──────────┐
│ Stripe      │────→│          │────→│            │────→│          │────→│          │
│ PostHog     │────→│ Airbyte  │────→│ PostgreSQL │────→│ dbt      │────→│ Metabase │
│ HubSpot     │────→│          │────→│ or BigQuery│────→│          │────→│          │
│ Google Ads  │────→│          │────→│            │────→│          │────→│          │
└─────────────┘     └──────────┘     └────────────┘     └──────────┘     └──────────┘
```

**Airbyte**（オープンソース、セルフホストまたはクラウド）：300以上のソースからウェアハウスにデータを取り込む。コネクタを設定し、同期をスケジュールすれば完了。

**dbt**（data build tool、オープンソース）：生データをクリーンで文書化されたテーブルに変換するSQLモデルを記述する。バージョン管理可能、テスト可能、そして業界標準。

**dbtモデルの例：**
```sql
-- models/metrics/monthly_mrr.sql
WITH subscription_events AS (
  SELECT
    account_id,
    DATE_TRUNC('month', event_date) AS month,
    SUM(mrr_change) AS net_mrr_change
  FROM {{ ref('stg_stripe_invoices') }}
  GROUP BY 1, 2
)
SELECT
  month,
  SUM(net_mrr_change) OVER (ORDER BY month) AS ending_mrr,
  SUM(CASE WHEN net_mrr_change > 0 THEN net_mrr_change ELSE 0 END) AS expansion_mrr,
  SUM(CASE WHEN net_mrr_change < 0 THEN ABS(net_mrr_change) ELSE 0 END) AS churned_mrr
FROM subscription_events
GROUP BY month
ORDER BY month
```

### データ品質のプラクティス

質の低いデータは質の低い意思決定につながる。データ品質をアップタイムと同じように守るべきだ。

1. **スキーマバリデーション。** すべてのイベントの期待される形状を定義する（必須プロパティ、型、許容値）。一致しないイベントは拒否またはフラグを立てる。PostHogとSegmentはいずれもこれをサポートしている。

2. **データ鮮度モニタリング。** データパイプラインが通常のインターバルの2倍の間、実行されていない場合にアラートを出す。Stripeの同期が1時間ごとに実行されるなら、2時間データがない時点でアラート。

3. **ボリューム異常検知。** 日次のイベントボリュームを追跡する。今日のボリュームが7日間平均より30%以上上下した場合にアラートを出す。急激な減少は通常、トラッキングの故障を意味する。急激な増加は通常、ループ内でイベントを発火するバグを意味する。

4. **重複排除。** イベントに冪等性キーを使用する。ネットワークリトライ、ダブルクリック、クライアントのバグはすべて重複イベントを引き起こす。取り込み時または変換レイヤーで重複排除する。

5. **dbtでのデータテスト。** 変換済みデータに対してテストを記述する：

```yaml
# schema.yml
models:
  - name: monthly_mrr
    columns:
      - name: month
        tests:
          - not_null
          - unique
      - name: ending_mrr
        tests:
          - not_null
          - dbt_utils.accepted_range:
              min_value: 0  # MRRは決してマイナスにならないはず
```

6. **ドキュメンテーション。** データディクショナリを維持する。すべてのテーブルとカラムについて：何であるか、どこから来るか、既知の注意事項、誰が所有者か。データモデルを理解しているのが1人だけなら、バスファクターが1だ。

7. **アクセス制御。** 全員にウェアハウスの書き込み権限を与えない。アナリストには変換済みテーブルへの読み取り権限。書き込み権限を持つのはデータパイプラインのみ。

### 命名規則

どの規則を選ぶかよりも一貫性が重要だ。一つ選んで、徹底する。

**イベント：** `object_action` のsnake_case
- `document_created`, `subscription_upgraded`, `invite_accepted`

**プロパティ：** snake_case
- `user_id`, `plan_name`, `created_at`, `is_active`

**テーブル：** `schema.object_plural` のsnake_case
- `raw.stripe_invoices`, `staging.stg_users`, `analytics.monthly_mrr`

**dbtモデル：**
- Raw: `raw_[source]_[table]`（例：`raw_stripe_invoices`）
- Staging: `stg_[source]_[table]`（例：`stg_stripe_invoices`）
- Intermediate: `int_[description]`（例：`int_monthly_revenue_by_account`）
- Marts: `[metric_or_entity]`（例：`monthly_mrr`, `customer_health_score`）
