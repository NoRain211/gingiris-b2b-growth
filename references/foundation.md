# 第一部分：奠定基石

## 1. 用户调研与 PMF 验证

### 为什么这件事至关重要

大多数 B2B 创业公司死于一个原因：做了没人需要的东西。用户调研不是走流程——它决定了你是在一个没前途的产品上耗 18 个月，还是找到切入市场的楔子。目标不是"验证你的想法"，而是搞清楚用户到底需要什么，以及你的产品是不是那个答案。

### HeyGen 案例：1800 次访谈找到 PMF

**第一步：Wizard of Oz MVP**
团队在 Fiverr 上提供了一项服务："为你制作一个会说话的数字人视频"。以极低成本验证了两个核心假设：
1. 市场对"让静态图片开口说话"存在真实需求
2. 用户愿意为此付费

这是成本最低的验证方式。在写一行代码之前，他们先手动交付了产品的核心结果。总成本：一条 Fiverr 商品和自己的时间。关键洞察在于——他们测试的是付费意愿，而不仅仅是兴趣。很多人会说"好酷"，但真正掏钱的人远比你想象得少。

**第二步：大规模用户访谈**
超过 1800 次访谈帮助团队从"有趣的工具"转变为"商业解决方案"。数量级本身就有意义：做 10 次访谈你听到的是个例，50 次你开始看到规律，200 次以上你就能按用户画像分群，识别出哪些场景有最强的拉动力。

### 衡量 PMF 的方法论

**Sean Ellis PMF 调研（40% 测试）**

问现有用户："如果明天就不能用 [产品] 了，你会有什么感受？"
- 非常失望
- 有点失望
- 不会失望
- 不适用——我已经不用了

如果 40% 以上选"非常失望"，说明你已经有了 PMF。低于 25%，说明还没有。25%-40% 之间，说明快了，但需要更明确你的价值主张。

调研至少需要 40 个有效回复，且受访者应在过去两周内至少使用过产品两次。别去调研那些注册后根本没激活的人——他们会稀释你的信号。

一些公司找到 PMF 时的实际数据：
- Slack：约 51% 选择"非常失望"
- Notion：约 45% 选择"非常失望"
- Superhuman：从 22% 起步，迭代到 58%

**Superhuman 的 PMF 引擎**

Rahul Vohra 团队打造了目前公开可见的最系统化的 PMF 提升方法论，完整流程如下：

1. **对用户分群。** 按用户画像拆分 Sean Ellis 调研结果。你可能整体只有 22%，但某个特定人群达到了 60%。那个人群就是你的桥头堡。

2. **找出"非常失望"用户最看重什么。** 问他们："你从 [产品] 获得的最大价值是什么？"归类他们的回答。这就是你的核心价值主张——你需要全力加码的方向。

3. **找出"有点失望"用户缺什么。** 问他们："我们怎么做才能让 [产品] 更好？"这些人差一点就爱上你了。他们的反馈是你最高杠杆率的改进方向。

4. **忽略"不会失望"的用户。** 他们的反馈会把你拉向平庸。别为不在乎你产品的人做需求。

5. **根据第 2-3 步制定产品路线图。** 一半用来强化忠实用户喜爱的点，另一半解决观望用户的痛点。每季度重新调研，追踪 40% 这个数字的变化。

**The Mom Test（Rob Fitzpatrick）**

用户访谈最重要的一条原则：永远别问别人你的想法好不好。他们会出于礼貌说好。正确的做法是：

- 错误："你会用一个做 X 的工具吗？"
- 正确："跟我说说你上次遇到 [这个问题] 是什么情况，具体发生了什么？"

- 错误："你愿意为这个每月付 50 美元吗？"
- 正确："你现在花多少钱解决这个问题？用的是什么付费工具？"

- 错误："你觉得这个功能有用吗？"
- 正确："你现在怎么处理这件事？最烦人的部分是什么？"

原则就是：聊他们的生活，而不是你的想法。问过去发生的事，而不是假设性问题。追求具体细节，而不是笼统感受。

### 如何分析访谈数据

**第一步：全程录音并转录。** 使用 Otter.ai、Grain 或 Fireflies 等工具。别只靠笔记——你会不自觉地筛选出符合自己预期的内容。

**第二步：给每句话打标签。** 用电子表格或 Dovetail 之类的工具。标签应包括：
- 提到的痛点
- 当前的解决方案/临时方案
- 功能需求
- 付费意愿信号
- 情绪强度（沮丧、无奈、中性、兴奋）
- 用户画像/角色

**第三步：统计出现频率。** 当 30 个企业市场经理中有 8 个人不约而同地用自己的话描述了同一个痛点，那就是信号。当 30 个人中只有 2 个提到某件事，那就是噪音。

**第四步：构建痛点-频率矩阵。**

| 痛点 | 画像 A (n=15) | 画像 B (n=20) | 画像 C (n=10) |
|:---|:---|:---|:---|
| 手动报表耗时数小时 | 12/15 (80%) | 4/20 (20%) | 1/10 (10%) |
| 无法与团队共享结果 | 3/15 (20%) | 16/20 (80%) | 7/10 (70%) |
| 现有工具太贵 | 9/15 (60%) | 11/20 (55%) | 2/10 (20%) |

这个矩阵告诉你哪个画像的痛感最强，以及具体痛在哪里。优先为百分比最高的那个格子做产品。

**第五步：识别用户当前行为中的规律。** 预测未来行为最可靠的依据是当前行为。如果用户已经在用电子表格、Zapier 自动化和各种手工流程来解决这个问题——这是很强的购买信号。如果他们只是说"嗯，是挺烦的"但什么都没做，说明这个痛点还不足以驱动购买。

### 何时转型 vs 坚持

转型信号：
- 迭代 3 次以上后 Sean Ellis 分数仍低于 20%
- 用户有参与感但第二周之后就不再留存
- 找不到任何一个 PMF 强劲的用户分群
- 你最好的用户所在的市场太小，撑不起一门生意
- CAC 是 LTV 的 3 倍以上，且没有改善趋势

坚持信号：
- 某个分群显示出强 PMF，即使整体数据偏弱
- 正确激活的用户留存率很高
- 已经有小规模的口碑传播在自然发生
- 你在解决的问题正在扩大（法规变化、市场变迁、新平台兴起）

最常见的错误，一是因为调研了错误的人群而过早转型，二是因为对第一个想法有执念而转型太晚。设定一个有时间限制的验证期（通常 2-3 个月），在开始之前定义好什么算成功，然后严格执行。

## 2. 用户访谈大纲模板

### 访谈前准备

在打电话之前，你需要知道：
- 对方的角色和所在公司
- 对方是现有用户、已流失用户还是潜在用户
- 对方的使用行为是什么样的（如果是现有用户）
- 这次访谈你具体想了解什么

别即兴发挥。事先写好最重要的 3 个问题，其他可以灵活调整。

### 1. 开场（3-5 分钟）

**目标：** 建立信任，设定预期，获得录音许可。

示例话术：
> "感谢你抽出时间。我是 [姓名]，来自 [公司]。我们正在改进 [产品/方向]，你的视角对我们非常有价值。整个过程大约 25-30 分钟。没有标准答案——我真的想听听什么好用、什么不好用。我可以录音吗？这样我就能专注在对话上，不用一直记笔记。录音仅供内部使用。"

快速了解背景：
- "你在 [公司] 负责什么？你典型的一周是什么样的？"
- "你的团队有多大？日常合作最紧密的是谁？"
- "你最早是怎么知道 [产品] 的？"（如果是现有用户）

### 2. 当前工作流与竞品分析（8-10 分钟）

**目标：** 了解他们今天怎么解决这个问题，用什么工具，摩擦在哪里。

核心问题：
- "带我走一遍你现在处理 [核心任务] 的流程。从什么触发开始？"
- "你目前用哪些工具来做这件事？当时是怎么选的？"
- "这个流程中最耗时的部分是什么？"
- "跟我说说上次这个流程出问题的情况。发生了什么？"
- "如果可以挥挥魔法棒解决当前流程中的一个问题，你会选哪个？"
- "在用我们之前你用什么？"（如果是现有用户）
- "你试过 [竞品] 吗？体验怎么样？"
- "你目前在这方面的工具上花多少钱？大概知道吗？"

注意倾听：
- 临时方案（电子表格、手动操作、"我们就硬扛过去"）——这些是未被满足的需求
- 频率和情绪权重（"快把我逼疯了" vs "有点烦"）
- 还有谁受这个问题影响（同一客户内的拓展机会）

### 3. 产品深度体验与功能反馈（8-10 分钟）

**目标：** 了解他们具体怎么用你的产品，什么好用，什么不好用。

针对现有用户：
- "给我演示一下你一般怎么用 [产品]。打开后第一件事做什么？"
- "你用得最多的功能是什么？为什么？"
- "有没有某个功能你试过一次就再没用过？什么原因？"
- "产品现在让你觉得困惑或者不爽的地方是什么？"
- "你有没有试过在 [产品] 里做某件事但没搞定？"
- "如果我们明天消失了，你会用什么替代？怎么处理 [任务]？"

针对潜在用户或已流失用户：
- "你注册了但没有继续使用——发生了什么？不用客气，我就是想了解实际情况。"
- "需要哪些不一样的东西才能让你继续用下去？"
- "有没有一个具体的时刻让你觉得这个产品不适合你？"

UI/UX 相关（仅在相关时问）：
- "你第一次用 [功能] 的时候，知道该怎么操作吗？"
- "有什么东西你希望更容易找到？"

### 4. 未来需求与付费意愿（5-7 分钟）

**目标：** 了解未被满足的需求和价格敏感度，但不要诱导。

- "接下来在这个领域你需要解决的最大问题是什么？"
- "如果你有预算去解决工作流中的一个问题，你会花在哪里？"
- "我们正在考虑做 [某个能力]。这对你的工作有用吗？"（慎用——容易产生假阳性）
- "你一般怎么向领导或财务部门申请工具采购？那个流程是什么样的？"
- "什么条件下你的团队会毫不犹豫地购买？"

探测价格信号：
- "你现在为 [最接近的同类工具] 付多少钱？"
- "如果我们能帮你每周省 [X 小时]，这对你的团队值多少钱？"
- 不要问："你会付 X 美元吗？"——要问："做这件事的工具你觉得合理的价位是多少？"

### 5. 结束（2-3 分钟）

- "有什么我该问但没问到的吗？"
- "你的团队或人脉中有没有也遇到这个问题、可能愿意聊几句的人？"
- "过几周如果我们在你提到的方面有了进展，我可以再联系你吗？"

感谢对方。24 小时内发送跟进邮件。如果你承诺了任何事（功能更新、内测邀请、引荐），务必兑现。

**报酬：** 对于 B2B 访谈，25-50 美元的 Amazon 礼品卡或产品额度是标准做法。对于企业决策者，别给礼品卡——以他们的名义向慈善机构捐 50 美元，或者就尊重他们的时间：控制通话时长，结束后发一份简洁的摘要。

## 3. 早期客户数据库模板

### 核心字段

| 字段 | 说明 | 为什么重要 |
|:---|:---|:---|
| User ID | 系统唯一标识 | 去重与追踪 |
| 姓名 | 全名 | 个性化沟通 |
| 邮箱 | 主要联系方式 | 日常沟通 |
| 电话 | 可选 | 高接触销售跟进 |
| 注册日期 | 注册时间 | 队列分析 |
| 激活日期 | 完成 Onboarding / 到达 Aha Moment 的日期 | 区分注册用户和真正的活跃用户 |
| 最后活跃时间 | 最近一次使用产品的时间 | 流失风险预警 |
| 活跃天数（近 30 天） | 近 30 天内活跃天数 | 参与深度 |
| LinkedIn URL | 职业社交主页 | 调研、外联、社会证明 |
| 公司 | 公司名 | 账户维度分析 |
| 公司规模 | 员工数或营收区间 | 按规模区分定价和销售策略 |
| 行业 | 所属行业 | 识别最佳匹配的市场细分 |
| 职位 | 角色 | 用户画像映射 |
| 核心使用场景 | 他们主要用产品做什么 | 功能优先级排序 |
| 获客渠道 | 怎么找到你的（Product Hunt、Google、推荐等） | 渠道 ROI |
| 标签 | 标记：`KOL`、`Enterprise`、`Potential Ambassador`、`Power User`、`At Risk` | 过滤与触发后续动作 |

### 参与度与营收字段

| 字段 | 说明 | 怎么用 |
|:---|:---|:---|
| Lead Score | 基于参与度信号的 0-100 分 | 优先分配销售资源 |
| 套餐/层级 | Free、Starter、Pro、Enterprise | 营收维度分群 |
| MRR | 该账户的月度经常性收入 | 追踪扩展/收缩 |
| NPS 评分 | 最近一次 NPS 分值（0-10） | 识别推荐者（9-10）和贬损者（0-6） |
| NPS 开放式反馈 | 在开放题中写了什么 | 定性信号 |
| Sean Ellis 调研结果 | "非常失望"/"有点失望"/"不会失望" | 按群体追踪 PMF |
| 推荐人数 | 推荐了多少新用户 | 识别天然的口碑传播者 |
| 工单数（近 90 天） | 提交的工单数量 | 高工单 + 低 NPS = 流失风险 |
| 功能需求 | 他们提过什么需求 | 产品路线图输入 |
| 扩展信号 | 新增团队成员、达到用量上限、询问企业版功能 | 追加销售触发点 |
| 流失风险评分 | 高/中/低，基于使用量下降、负面 NPS、工单增加 | 主动留存 |
| 合同续约日期 | 年付/月付何时到期 | 续约外联时机 |

### 如何对数据进行分群并采取行动

**按参与度分层：**
- **超级用户**（活跃度前 10%）：邀请他们提供客户证言、案例研究、推荐。纳入大使计划。这些人就是你的免费推销员。
- **健康用户**（活跃，NPS 正面）：基本不用打扰。推送产品更新，邀请填写 PMF 问卷。
- **高风险用户**（活跃度下降、NPS 负面、工单增多）：主动外联。"嗨，我注意到你最近遇到了一些问题——需要我帮忙吗？"别等他们流失。
- **沉睡用户**（注册后从未激活或已停止使用）：唤醒策略（见第 8 节）。

**按价值分层：**
- **高 MRR 账户：** 配备 CSM，每季度做业务回顾，白手套服务。
- **中等账户但有扩展信号：** 销售辅助追加销售。"我看到你的团队从 3 人增长到了 12 人——需要我介绍一下团队版吗？"
- **高参与度的免费用户：** 转化策略。他们已经在获取价值——帮他们看到付费为什么值得。

**按画像/场景分层：**
- 按职位 + 使用场景分组。这能告诉你哪类画像 PMF 最强，以及市场和销售应该往哪里集中火力。

数据库至少每周更新一次。过时的数据比没有数据更糟，因为它制造虚假的信心。

## 4. 产品发布策略

### 三阶段节奏

**阶段一：Private Beta（封闭测试，4-8 周）**

目标：发现并修复关键问题。用真实使用验证核心价值主张。

- 向精心筛选的 20-50 名用户开放。选择符合理想客户画像、并且会真正给反馈的人（不是只挂个名不理你的大佬）。
- 明确设定预期："这是早期版本，会有问题。作为回报，你将得到我们全部的关注和路线图的话语权。"
- 高强度使用和反馈。每周与测试用户沟通。观看会话录像（Hotjar、FullStory、PostHog）。追踪激活率——如果低于 30% 的测试用户完成了 Onboarding，说明产品还没准备好。
- 为测试用户建一个私密的 Slack 或 Discord 频道。这就是你的第一个社区。
- 每周修复前 3 个阻塞问题。别试图修所有东西——专注于阻止用户到达 Aha Moment 的问题。

**阶段二：Early Access / 等候名单（2-4 周）**

目标：制造期待感和社会证明，测试可扩展性。

- 通过邀请码或等候名单扩大范围。每个测试用户拿到 3-5 个邀请码。这既能驱动自然增长，又让每个邀请显得珍贵。
- 等候名单制造稀缺感和期待。但别让它停滞——每周分批放号，并定期给等候名单用户发产品更新邮件，让他们别忘了你。
- 等候名单规模参考：Superhuman 在发布前积累了 180,000+ 人。Linear 约 10,000 人。你不需要海量数字——对 B2B 产品来说，500-2,000 个有参与度的等候用户就算不错了。
- 在等候注册时收集数据：角色、公司规模、使用场景、目前在用什么工具。这是免费的调研。
- 开始发布内容：关于你解决的问题的博客、产品幕后构建日志、早期用户故事。这些为正式发布蓄能。

**阶段三：Public Launch（公开发布）**

目标：在集中的时间窗口内获得最大曝光，将关注度转化为注册。

**Product Hunt 发布指南：**

时机：
- 选在周二、周三或周四发布。周二对 B2B 产品通常表现最好。
- 太平洋时间 12:01 AM 发布（Product Hunt 日期重置时间），这样你有最长的时间积累 upvote。
- 避免和大型产品发布撞车（Apple 发布会、科技大厂公告等）。提前查看 Product Hunt 的 Upcoming 页面。

准备工作（提前 2-3 周开始）：
- 写好 Tagline（60 字符以内，突出价值，不用行话）。用测试用户测试 5 个版本。
- 制作 1-2 分钟演示视频。展示产品解决真实问题的过程。别加无聊的片头或纯讲述——前 5 秒就进入演示。
- 准备 4-6 张高质量截图/GIF，展示核心功能。
- 写 Product Hunt 描述：问题 → 方案 → 核心功能 → 社会证明（测试数据、用户评价）。控制在 300 词以内。
- 找好你的"Hunter"——最好是在 Product Hunt 上有大量关注者的人。顶级 Hunter 如 Chris Messina 或 Kevin William David 可以让你的曝光提升 2-3 倍。
- 组建发布团队：50-100 人在发布日 upvote 并留下真实评论。亲自联系测试用户、朋友、顾问、Twitter 互关的人。不要用刷票服务——Product Hunt 能检测到并会处罚。

发布当天：
- 发布后立即发一条"Maker's Comment"。讲故事：为什么做这个，解决什么问题，你们的做法有什么不同。真诚一点。
- 30 分钟内回复每条评论。Product Hunt 奖励互动。
- 在所有渠道分享：Twitter/X、LinkedIn、邮件列表、社区频道。给出 Product Hunt 页面的直接链接。
- 全天发布进度更新（"我们目前在 Product Hunt 排名第 3——谢谢大家！"）。
- 不要直接请求 upvote——而是请别人"来看看我们的发布，告诉我们你的想法"。

B2B 产品成功发布的参考数据：
- 当日产品 Top 5：约 300-500 upvotes
- 当日产品 #1：约 600-1000+ upvotes
- Top 5 Product Hunt 发布带来的注册量：通常 1,000-5,000，取决于产品类型
- 从 Product Hunt 访客到注册的转化率：10-20% 算不错

**发布前邮件序列：**

如果你有等候名单，发这套序列：

- **T-7 天：** "我们将在下周 [某天] 发布。这是我们做了什么以及为什么。" 附上预览。
- **T-1 天：** "明天就是那一天了。以下是你可以如何帮忙——这是你的提前访问链接。"
- **发布当天（早上）：** "我们已经在 Product Hunt 上线了。[直接链接]。如果你一直在用并且喜欢，一条关于你使用体验的评论比一个 upvote 更有价值。"
- **T+1 天：** "谢谢大家。以下是发生了什么。" 分享结果，感谢支持者，宣布发布日的成果。

**其他同步启动的发布渠道：**
- Hacker News "Show HN" 帖子（写成技术故事，不是推销）
- Twitter/X 长推文：问题 → 你尝试了什么 → 你做了什么 → 测试阶段的结果
- 创始人的 LinkedIn 帖子（个人号 > 公司号）
- 相关 Subreddit（严格遵守每个 sub 的自我推广规则，否则会被封号）
- 行业相关的 Slack 群和 Discord 服务器
- Indie Hackers、dev.to 或其他相关社区

**应对发布后的流量回落：**

每次发布都会有一个高峰然后回落。这是正常的。第 1-3 天的流量是平时的 10-50 倍。到第 7 天，流量下降 80-90%。真正的工作从这时候开始。

怎么做：
- 别慌。回落不是失败——只是发布兴奋期结束了。
- 专注于激活已注册的用户。如果 3,000 人注册了，有多少完成了 Onboarding？有多少到达了 Aha Moment？
- 启动你的内容引擎（见第 5 节）。发布给你一次性流量峰值；内容给你复利增长。
- 亲自跟进发布期间注册的、符合 ICP 的每一个用户。创始人的个人邮件转化率是自动化邮件的 5-10 倍。
- 追踪发布队列用户的第 7 天和第 30 天留存率。这能告诉你发布是否吸引到了对的人。

**实际发布案例参考：**
- **Notion：** 多次在 Product Hunt 发布（2016、2018）。2018 年的重新发布获得了当日产品。他们将 Product Hunt 与 Freemium 模式和社区模板结合。关键启示：你可以重新发布。
- **Linear：** 公开发布前邀请制运营了 2 年。通过口碑和精美的等候页面积累了大量需求。Product Hunt 发布是高潮，而不是起点。
- **Arc Browser：** 长达一年的等候名单，使用"会员"的定位。每个用户只有有限的邀请码。让人觉得是在加入一个俱乐部，而不是注册一个软件。面向 100 万+ 等候用户发布。

## 5. 社区驱动增长（CLG）

### 为什么社区对 B2B 有效

社区具有复利效应。付费广告停投就没量了，内容需要数月才能获得排名。但一个健康的社区能同时产生口碑传播、用户生成内容、产品反馈和客户留存。最好的 B2B 社区（dbt、Figma、Notion、Airbyte）会成为竞争对手无法复制的护城河。

### 三阶段路径

**第一阶段：冷启动（0-100 成员）**

这是最难的阶段。一个空荡荡的社区就是一个死社区。

策略：
- **创始人驱动。** 你（创始人或社区负责人）亲自在 2 小时内回答每一个问题。这个阶段没人会比你更上心。
- **用已有关系做种子。** 邀请测试用户、你访谈过的人、Twitter/X 上的朋友。用个人邀请，不是群发邮件。"嗨，我们正在为研究 [某问题] 的人建一个社区。你很合适——链接在这里。"
- **平台选择：** 开发者/技术产品用 **Discord**，商业/企业产品用 **Slack**。别在这上面纠结太久——后面可以迁移。
- **核心频道：** `#introductions`、`#general`、`#feedback`、`#showcase`（用户展示他们做的东西）、`#announcements`、`#help`
- **每天发帖。** 分享产品更新、有趣的链接、问问题。如果创始人自己不发帖，就没人会发。
- **突出早期成员。** "本周向 @user 致敬，他用我们的产品做了 [某个东西]。" 公开认可是最便宜也最有效的激励。
- **在现有频道活跃之前不要新增频道。** 15 个空频道比 4 个活跃频道更糟。

第一阶段的成功指标：每天至少 5 条非团队成员的自发消息。

**第二阶段：内容与文化塑造（100-1,000 成员）**

你已经过了冷启动。现在要建立习惯和文化。

固定内容节奏：
- **每周更新日志：** 每周五发布本周上线了什么。保持简短，附上截图/GIF。
- **每两周教程：** "如何用 [产品] 做 [某件事]"。可以是文字、视频或直播演示。
- **每月 AMA：** 创始人或产品负责人在线回答问题 30-60 分钟。为错过的人发布回顾。
- **每季度路线图回顾：** 分享即将推出的功能，征求意见。这让成员觉得自己是共建者，而不只是用户。

鼓励 UGC：
- **Showcase 频道：** 主动邀请大家分享自己做的东西。回复每一条帖子。最好的转发到你的 Twitter/博客。
- **模板/资源分享：** 建一个频道让用户分享模板、脚本、工作流。
- **"我是怎么用 [产品] 的" 故事：** 邀请高活跃用户写下他们的工作流。如果他们没时间，提出帮他们一起写。
- **用户互助：** 当有人问了一个你知道另一个成员能回答的问题，@那个成员而不是自己回答。这能建立横向连接。

尽早建立社区文化规范：
- 提问要具体（不是"不能用"而是"我在做 Y 的时候遇到了错误 X"）
- 庆祝成果（上线的功能、里程碑、用户故事）
- 不允许垃圾推广
- 欢迎建设性反馈，但不容忍敌意

**内容日历模板：**

| 日期 | 内容类型 | 负责人 | 频道 |
|:---|:---|:---|:---|
| 周一 | "这周你在做什么？"讨论帖 | 社区负责人 | #general |
| 周二 | 教程或小贴士 | 产品/社区 | #tutorials |
| 周三 | 用户聚焦或 Showcase 转发 | 社区负责人 | #showcase |
| 周四 | 产品更新或剧透 | 产品团队 | #announcements |
| 周五 | 每周更新日志 + "本周成果"帖 | 产品团队/社区 | #announcements, #general |

月度：
- 第 1 周：创始人/产品负责人 AMA
- 第 2 周：用户案例或嘉宾分享
- 第 3 周：社区挑战（"用我们的产品做 X，分享成果"）
- 第 4 周：反馈汇总——最热门的需求以及我们的应对

**第三阶段：规模化与大使计划（1,000+ 成员）**

到了这个规模，你不可能事事亲力亲为了。你需要结构化运营。

- **启动大使计划**（见第 6 节）。大使成为社区的免疫系统——他们回答问题、管理秩序、定义社区氛围。
- **增加角色或主题频道。** `#enterprise`、`#developers`、`#marketing-teams` 等。
- **社区活动：** 线上聚会、工作坊、Hackathon。从每月一次开始，有需求就提频到每两周。
- **专职社区经理。** 成员超过 1,000 时这就是一个全职岗位。招有社区建设经验的人，不是做社交媒体的。
- **与其他社区互通。** 和互补产品合作搞联合活动或内容。

### 社区参与度指标

| 指标 | 衡量什么 | 健康基准 |
|:---|:---|:---|
| 日活跃成员（DAM） | 每天发帖或互动的成员 | 总成员的 5-10% |
| 周活跃成员（WAM） | 每周至少活跃一次的成员 | 总成员的 20-30% |
| 每日消息数 | 整体活跃度 | 1,000 成员时 50+ |
| 4 小时内得到回复的问题比例 | 响应速度 | >80% |
| 成员间互相回复 | 横向参与度（不只是团队→成员） | 回复中占 >50% |
| 每周新增成员 | 增长速度 | 每周增长 5-10% 算强劲 |
| 流失率 | 连续 30 天不活跃的成员比例 | 月流失 <10% |
| 社区成员 vs 非成员的 NPS | 社区对产品满意度的影响 | 社区成员应高出 15-20 分以上 |

### 社区管理手册

**前置管理：**
- 在置顶的 `#rules` 或 `#code-of-conduct` 频道设置规则。简短就好：善意、具体、不发垃圾信息、不骚扰。
- 用机器人自动标记新成员发的含链接帖子（垃圾过滤）。
- 24 小时内欢迎每一个新成员。机器人欢迎可以，但人工跟进（"是什么让你来这里的？"）效果好得多。

**被动管理：**
- 垃圾信息：删除并封禁。不需要警告。
- 敌意或人身攻击：先私信对方。"我理解你的沮丧，但这种说法在这里不合适。能换个方式表达吗？"删除消息。如果再犯，封禁。
- 跑题或自我推广：引导到正确频道或私信："嗨，自我推广请发到 #showcase 频道。麻烦移一下？"
- 钓鱼或恶意争论：私信警告一次，然后封禁。不要在公开场合喂养喷子。

记录每一次管理操作。万一有人质疑时你需要这些记录。

### 社区转化为销售管线

社区不应该让人觉得是一个销售渠道，但如果做得好，它确实能带来收入。

- **追踪哪些社区成员转化为付费。** 在 CRM/数据库中标记社区成员。对比转化率：社区成员 vs 非社区成员。
- **在社区中识别扩展信号。** 当有人问"[产品] 能给我 50 人的团队用吗？"——这就是一个等待发生的销售对话。转给销售。
- **从社区产出案例研究。** 你最好的客户已经在社区里讨论他们怎么用你的产品了。问他们是否愿意做正式的案例研究。
- **社区专属提前体验。** 让社区成员优先使用新功能。这既奖励了参与度，又能获得反馈。

典型参考数据：社区成员的付费转化率是非社区用户的 2-5 倍。社区成员的流失率比平均水平低 20-40%。

## 6. 大使计划

### 为什么要大使，而不只是"粉丝"

大使计划将你与最活跃用户的关系正式化。粉丝和大使的区别：粉丝在别人提到时会推荐你的产品。大使会主动创作内容、回答问题、组织活动、招募新用户。结构化的计划给他们认可、资源和更多贡献的激励。

### 申请表

1. 全名
2. 邮箱
3. 所在地（城市、国家）
4. LinkedIn 或 Twitter/X 主页
5. 自我介绍：你是谁，做什么的
6. 你使用 [产品] 多久了？
7. 你最喜欢产品的哪些方面？
8. 你感兴趣的贡献方向（可多选）：
   - 内容创作（博客、视频、教程）
   - 社区支持（回答问题、帮助新用户）
   - 开发者布道（代码示例、集成、技术内容）
   - 活动组织（组织或参与 Meetup、大会演讲）
   - 本地化（翻译内容、管理非英语频道）
9. 你已经创作过的关于 [产品] 或相关领域的内容链接
10. 你能实际投入的时间（最低：每月 3-5 小时）
11. 你希望从这个计划中获得什么？（坦诚说——职业成长、人脉、周边等都可以）
12. 行为准则协议

**筛选标准：** 不要来者不拒。要看：
- 在社区已经活跃（不是来白拿好处的陌生人）
- 有内容创作经历（博客、视频、推文）——即使不是关于你的产品
- 参与目的明确
- 时间投入预期合理

目标录取率 20-40%。有选择性才能让计划显得有价值。

### 报酬与激励结构

报酬应与贡献挂钩。别不管贡献多少都给一样的东西。

**第一层：新大使（前 1-3 个月）**
- 周边礼包（T 恤、贴纸、笔记本）——预算每人 30-50 美元
- 付费版产品免费使用
- 大使专属 Slack/Discord 频道
- 个人资料和社区中的"大使"徽章

**第二层：活跃大使（3 个月以上，持续贡献）**
- 第一层全部权益
- 每月 50-100 美元产品额度或礼品卡
- 新功能和 Beta 版的提前体验
- 联合推广：在官方渠道分享他们的内容，在博客中推荐他们
- 行业会议门票赞助（每年 1-2 次）
- 产品团队直通车（每季度反馈专场）

**第三层：顶级大使/社区领袖（6 个月以上，贡献突出）**
- 一、二层全部权益
- 收入分成：对他们带来的推荐用户给予 15-25% 的持续佣金（使用可追踪的推荐链接）
- 年度津贴：500-2,000 美元/年，取决于你的阶段和他们的影响力
- 顾问委员会参与，附带期权（0.01-0.05%，仅限表现极为突出的长期大使——非常罕见，只给真正的共建者）
- 在你的活动中的演讲机会
- 对产品路线图和功能优先级的话语权

**其他报酬模式：**
- **按内容付费：** 每篇博客 50-200 美元，每个视频 100-300 美元。适合以内容为核心的大使计划。
- **悬赏系统：** 发布具体任务（写一个关于 X 的教程、本月回答 20 个社区问题）并设定奖励。让贡献方向集中。
- **积分系统：** 为不同活动赋分（发帖 = 5 分、回答问题 = 10 分、博客文章 = 50 分）。每月兑换奖励。可以用 Orbit 或 Common Room 等工具来追踪。

### 大使 KPI

按每位大使追踪，每月回顾：

| KPI | 衡量方式 | 最低标准 |
|:---|:---|:---|
| 内容产出 | 博客、视频、推文、教程 | 每月 2 篇 |
| 社区问题回复 | 通过 Discord/Slack 追踪 | 每月 10 条 |
| 活动组织/参与 | Meetup、演讲、工作坊 | 每季度 1 次 |
| 推荐注册 | 通过推荐链接追踪 | 每月 3 个注册 |
| 内容互动量 | 他们内容的浏览、点赞、评论 | 环比上升 |

### 清退不活跃大使

这是大家不愿意谈的部分。一个不再贡献但继续享受权益的大使会损害整个计划。

**流程：**
1. 如果某位大使连续 2 个月未达到最低标准，私下关怀："嗨，我注意到你最近不太活跃了。一切还好吗？有什么我们能帮忙的？"
2. 如果他们回复并承诺重新参与，再给一个月。
3. 如果没有回复，或者第三个月仍未达标，发一条善意但明确的离场消息："我们先暂停你的大使身份。你随时欢迎继续待在社区，如果情况变了，随时可以重新申请。感谢你之前的所有贡献。"
4. 移除权益（产品额度、周边发放、私密频道权限）。别搞得太正式。

**永远不要公开宣布谁被清退了。** 安静地处理就好。如果其他大使问起，就说"他们因为其他事务暂时退出了"。

### 从 10 人到 100+ 人的规模化

**10 位大使时：** 你亲自管理。每周群组通话或异步 Slack 沟通。你了解每个人的优势和兴趣。

**25 位大使时：** 设立区域或职能负责人。"Maria 负责拉美地区大使，Raj 负责开发者布道。"负责人处理日常协调；你负责策略和顶级贡献者关系。

**50 位大使时：** 你需要工具了。用 Orbit、Common Room 或自定义看板来追踪贡献、自动化奖励分发、识别高风险大使。

**100+ 位大使时：**
- 招一个全职大使计划经理。
- 按批次运营——每季度接收新大使，批量 Onboarding，为他们分配老大使作为导师。
- 年度大使峰会（线上或线下）。预算 5,000-20,000 美元，取决于形式。这是你最高杠杆率的投入——它能建立深度忠诚并产出内容。
- 正式化层级和晋升标准。大使应该清楚地知道从第一层升到第二层需要做什么。

## 7. 定价策略

### B2B SaaS 的定价模式

**Freemium：** 提供可用的免费版；对高级功能、更高用量或团队协作收费。
- 适合：有病毒传播或网络效应的产品、边际成本低、市场空间大的产品。
- 案例：Slack（小团队免费，付费解锁消息历史和功能）、Notion（个人免费，团队付费）、Figma（3 个项目免费）。
- 目标：2-5% 的免费转付费率。低于 2%，说明免费版太慷慨或付费版不够吸引人。高于 5%，说明免费版可能限制得太多。
- 关键风险：永远不转化的免费用户会消耗你的成本（基础设施、客服）。设置的限制应该让用户感受到价值，但在认真使用时碰壁。

**限时免费试用：** 7-14 天全功能使用，到期后付费或失去访问权。
- 适合：价值能快速显现但需要真实部署/集成的产品（BI 工具、CRM、项目管理）。
- 试用时长：简单产品 7 天，需要集成的产品 14 天，企业级 30 天（但 30 天试用的转化率通常更低，因为缺乏紧迫感）。
- 目标：15-25% 的试用转付费率。低于 15%，说明你的 Onboarding 没能让用户快速到达 Aha Moment。高于 25%，说明你也许可以缩短试用期。
- 如果你的产品有学习曲线，不要要求预先绑定信用卡——这会减少 50-70% 的试用开始量。如果你的产品能立即交付价值且你想筛选认真买家，那就要求。

**按用量计费：** 基于消耗量收费（API 调用、数据处理量、消息数、席位数）。
- 适合：使用量与获得的价值成正比的产品（基础设施、API、数据工具）。
- 案例：Twilio（按消息/通话）、Snowflake（按计算量）、AWS（按一切）。
- 优势：入门门槛低，收入随客户成功同步增长。客户喜欢，因为他们不用为闲置的软件付费。
- 风险：收入可预测性差。客户可能自行优化以减少用量。你需要出色的计量和计费基础设施。
- 混合方案：基础平台费 + 用量组件。这兼顾了收入可预测性和用量增长带来的上升空间。

### 如何定价

**Van Westendorp 价格敏感度模型**

向目标客户样本（至少 30 个回复）问四个问题：
1. "价格低到什么程度你会怀疑它的质量？"（太便宜）
2. "什么价格让你觉得很划算？"（便宜/高性价比）
3. "什么价格开始觉得贵了，但你还是会考虑购买？"（贵但可接受）
4. "什么价格让你觉得太贵而不会考虑购买？"（太贵）

在图表上绘制累积分布曲线。交叉点告诉你：
- 边际便宜点（太便宜 × 贵）：你的价格下限
- 边际昂贵点（太贵 × 便宜）：你的价格上限
- 最优价格点（太便宜 × 太贵）：认为太便宜和太贵的人数相等的点
- 无差异价格点（便宜 × 贵）：最常见的"合理"价格

这给你一个价格区间，而不是单一数字。从最优价格点开始测试。

**Gabor-Granger 方法**

更简单的方式。给出一个价格，问："你会以 X 美元/月购买吗？"从高价开始。如果说不，降低。如果说是，提高。找到意愿骤降的那个价格点。

在理解需求曲线和价格弹性方面比 Van Westendorp 更有用，但细腻度不如后者。

**竞品锚定**

参考客户实际在对比的替代方案来定价：
- 如果你在替代手工流程（电子表格、实习生手动做）：定价为你替代的人力成本的 30-50%。
- 如果你在替代现有工具：定在该工具价格的 20% 以内，然后通过价值差异化，而不是价格竞争。或者大幅低于（50%+），如果你是新进入者想抢份额。
- 如果你在创造一个新品类：你有更大的定价自由度，但需要锚定问题的成本，而不是其他工具的成本。

### 套餐设计

标准 B2B 套餐结构：

| | Free / Starter | Pro | Enterprise |
|:---|:---|:---|:---|
| 目标用户 | 个人用户、评估使用 | 团队、成长期公司 | 大型组织、合规需求 |
| 定价 | $0 或 $10-30/席位/月 | $30-100/席位/月 | 定制（通常 $50-200/席位/月） |
| 功能 | 核心功能、有限用量 | 全部功能、更高上限、集成 | 全部 + SSO、SAML、审计日志、SLA、专属支持 |
| 支持 | 社区/文档 | 邮件、在线客服 | 优先级支持、专属 CSM |
| 计费 | 月付 | 月付或年付（年付折扣 10-20%） | 年度合同、发票结算 |

经验法则：
- 设 3-4 个层级刚好。太少变成"买或不买"的二选一。太多造成决策瘫痪。
- 每个层级都应该有清晰的"为谁设计"——如果你不能用一句话描述某个层级的目标买家，就删掉它。
- 从免费到第一个付费层级的跳跃是最难的。让它感觉像不需要犹豫的决定（SMB 的 10-30 美元区间，中型企业的 50-100 美元区间）。
- 企业版应该显示"联系我们"而不是价格。如果你标出价格，你就在浪费钱，还失去了按公司规模差异化定价的能力。

### 何时以及如何涨价

**何时涨价：**
- 产品已经大幅改进但定价 12 个月以上没变
- 免费试用转化率超过 25%（说明定价太低）
- 对竞品胜率超过 70%（说明你是最便宜的选项，而不是最有价值的）
- 客户从不对价格有异议（听起来不错但意味着你定价过低）
- 你正在增加重大新能力

**如何涨价：**
- 给现有客户 6-12 个月的老价格过渡期。这能防止反弹并奖励忠诚度。
- 提前 30-60 天通知。"在 [日期]，我们的 Pro 套餐将从 X 美元调整为 Y 美元。以下是我们上次调价后发布的内容：[列表]。"
- 新客户立即执行新价格。
- 如果涨幅超过 30%，考虑新增一个层级而不是直接提价。这让人感觉是"更多选择"而不是"花更多钱买同样的东西"。
- 追踪涨价后 90 天的流失率。如果流失率比基线高 5 个百分点以上，说明涨得太猛了。

## 8. 留存与激活

### 定义你的 Aha Moment

Aha Moment 是用户第一次体验到你的核心价值的时刻，而不仅仅是了解到它。这是他们心想"哦，原来大家为什么用这个"的那个瞬间。

案例：
- **Slack：** 和队友发送并收到一条消息（不只是注册）
- **Dropbox：** 在一个地方保存文件，在另一台设备上访问
- **Zoom：** 毫无技术障碍地完成第一通视频通话
- **HubSpot：** 看到第一个带有自己网站真实数据的仪表板

**如何找到你的 Aha Moment：**

1. **队列分析：** 对比 30 天后留存的用户和流失的用户。留存用户在第一次会话/第一周做了哪些流失用户没做的动作？
2. **先找相关性，因果性后面再说。** 如果 80% 的留存用户在前 3 天完成了动作 X，而只有 15% 的流失用户做了，那么动作 X 很可能就是你的 Aha Moment。
3. **B2B 中常见的 Aha Moment 模式：**
   - 创建了第一个 [东西]（项目、报告、工作流、仪表板）
   - 邀请了队友（协作类产品）
   - 连接了一个集成（数据类产品）
   - 看到了基于自己数据的真实结果（分析类产品）
   - 自动化了一个之前手动完成的任务（自动化类产品）

4. **通过用户访谈验证。** 问留存用户："你是从什么时候开始觉得这个产品'对了'的？当时在做什么？"他们的回答应该和你的数据分析一致。

### 有效的 Onboarding 流程

你的 Onboarding 只有一个任务：让用户尽快到达 Aha Moment。其他一切都是干扰。

**原则：**
- **缩短价值感知时间。** 如果你的 Aha Moment 需要 5 个步骤，别让用户先做 12 个步骤。砍掉非必要的设置，推迟配置，预填示例数据。
- **展示，而不是讲述。** 交互式引导优于视频教程。视频教程优于文档。文档优于什么都没有。
- **每屏只有一个 CTA。** 不要用所有功能一次性轰炸新用户。引导他们走关键路径。
- **衡量每个步骤的完成率。** 如果 70% 的用户完成了第 1 步但只有 20% 完成了第 2 步，第 2 步就是你的瓶颈。修第 2 步。

**Onboarding 清单模式：**

给新用户展示一个 3-5 项的简单清单：
1. 完善你的资料
2. [达到 Aha Moment 的核心操作]
3. 邀请一个队友
4. [第二个核心操作]
5. 庆祝："你已经准备好了！"

展示进度条。完成偏好是真实存在的心理效应——开始了清单的人会有冲动完成它。目标清单完成率 60% 以上。

**Onboarding 邮件：**

| 时机 | 邮件 | 目的 |
|:---|:---|:---|
| 立即 | 欢迎 + 快速入门指南 | 让他们进入产品 |
| 第 1 天（如未激活） | "需要帮忙开始吗？" | 消除摩擦 |
| 第 3 天 | "[类似公司] 是这样用 [产品] 的" | 社会证明 + 场景启发 |
| 第 5 天（如未激活） | 来自真人的"我能帮忙吗？"（创始人或客服） | 给卡住的用户人情味 |
| 第 7 天 | "看看你错过了什么"——展示关键功能 | 扩展认知 |
| 第 10 天 | 案例研究或 ROI 示例 | 证明购买决策的合理性 |
| 第 14 天（试用到期） | "你的试用明天结束——以下是你将失去的" | 紧迫感 + 损失厌恶 |

**个性化很重要。** 如果你在注册时问了用户的使用场景，就据此定制 Onboarding 路径。同一个产品的市场人员和开发者需要不同的首次体验。

### 缩短价值感知时间

| 策略 | 做什么 | 影响 |
|:---|:---|:---|
| 预填模板 | 用户在动手之前就能看到价值 | 高——消除空白画布焦虑 |
| 示例数据 | 产品立刻看起来功能完整 | 中高——展示"完成时"的样子 |
| 一键集成 | 几秒钟内连接到他们已有的工具 | 对数据/工作流产品很高 |
| 跳过设置，后续再提示 | 让用户先探索再配置 | 中——减少设置界面的流失 |
| 应用内提示 | 引导但不打断 | 中——对复杂 UI 有效 |
| 管家式 Onboarding | 为高价值客户提供一对一设置帮助 | 非常高——企业激活率提升 2-3 倍 |

基准：顶级 B2B 产品能让用户在注册后 5-10 分钟内到达 Aha Moment。如果你的价值感知时间以天为单位，就有问题了。

### 留存指标

| 指标 | 定义 | 健康基准（B2B SaaS） |
|:---|:---|:---|
| 次日留存 | 注册后次日返回的比例 | 40-60% |
| 7 日留存 | 注册后第 7 天活跃的比例 | 25-40% |
| 30 日留存 | 注册后第 30 天活跃的比例 | 15-25% |
| 周环比留存 | 上周活跃用户本周仍活跃的比例 | 80%+ |
| 净收入留存率（NRR） | 现有客户收入含扩展减去流失 | 100-130%（顶级水平：130%+） |
| Logo 流失率 | 每月取消的客户比例 | SMB 月流失 <3%，企业 <1% |
| 激活率 | 注册用户到达 Aha Moment 的比例 | 30-50% |

### 流失用户唤醒策略

不是所有流失用户都一去不返了。有些人离开是因为某个缺失的功能你已经做了，某个 Bug 你已经修了，或者只是时机不对。

**何时做唤醒：**
- 上线了某个被高频需求的重大功能后
- 修复了一个已知痛点后
- 每季度，针对 30-90 天前流失的用户（超过 90 天，响应率急剧下降）

**唤醒邮件模板：**

标题："[产品] 在你离开后发生了不少变化"

> 嗨 [名字]，
>
> 我注意到你大约 [X 周] 前停止使用 [产品] 了。完全理解——当时我们确实还不够好。
>
> 从那以后，我们做了 [针对常见流失原因的具体改进]：
> - [功能 1]：[一句话说明价值]
> - [功能 2]：[一句话说明价值]
> - [功能 3]：[一句话说明价值]
>
> 如果其中有解决了你当时遗憾的地方，希望你能再试试看：[链接]
>
> 无论如何都没关系。但如果你有 30 秒，我真的很想知道：你当时停用的主要原因是什么？这能帮我们变得更好。
>
> [你的名字]

**预期效果：** 唤醒邮件的打开率通常在 15-25%，重新激活率在 2-5%。绝对值看起来不高，但这些用户你已经花过获客成本了——CAC 为零。

**按流失原因分层唤醒**（如果你有退出调研数据）：
- 因缺少功能流失 → 该功能上线时发邮件
- 因价格流失 → 促销活动或新增更便宜层级时发邮件
- 因转用竞品流失 → 你上线了竞品没有的功能时发邮件
- 因"用得不够多"流失 → 发送使用场景指南或同类公司的成功案例

## 9. 竞争情报

### 为什么需要系统化方法

大多数创业公司追踪竞品都是被动的——有人在销售电话里提到竞品，然后大家手忙脚乱。应该建立一个系统。好的竞争情报意味着：你永远不会被竞品的动作打个措手不及，你的销售团队对"为什么不选 [竞品]？"有现成的答案，你的产品路线图是基于市场空白来制定的。

### 如何追踪竞品

**设置监控（1-2 小时搞定，之后基本自动化）：**

1. 为每个竞品的公司名、产品名和 CEO 名字设置 **Google Alerts**。
2. 用专门的列表或 TweetDeck 之类的工具**在社交媒体上关注竞品**（Twitter/X、LinkedIn）。
3. 用非公司邮箱**订阅他们的邮件列表**。你要看他们的产品更新、定价变动和营销信息。
4. **监控他们的更新日志/博客。** 用 RSS 阅读器或 Feedly 之类的工具。每周查看。
5. **追踪他们的招聘信息。** 招什么人意味着他们在投资什么方向。突然招 5 个 ML 工程师？他们在做 AI 功能。发了企业销售 VP 的 JD？他们在向上走。
6. **评测网站：** 在 G2、Capterra、TrustRadius 上为你的品类设置提醒。读竞品的差评——他们的 2 星评价就是你的销售弹药。
7. **专利和融资公告。** 用 Crunchbase 看融资，Google Patents 看专利。大额融资轮意味着他们未来 12-18 个月会激进地花钱。

**竞品追踪数据库：**

| 竞品 | 最后更新 | 定价 | 核心功能 | 目标市场 | 近期变化 | 优势 | 劣势 | 我们的差异化 |
|:---|:---|:---|:---|:---|:---|:---|:---|:---|

至少每季度更新一次。指定一个人负责。

### 赢单/输单分析框架

每一笔成交（赢或输）后，收集以下信息：

**赢单：**
- 客户评估了哪些替代方案？
- 在评估的哪个阶段决定选我们？
- 选择我们的第一大原因是什么？
- 价格是决定因素吗？我们的定价和竞品比怎么样？
- 内部推动者是谁？他们在内部怎么说服别人的？

**输单：**
- 他们最终选了什么？（竞品、自建、什么都不做）
- 在哪个阶段丢单的？（初期评估、POC、最终决策、采购流程）
- 不选我们的第一大原因是什么？
- 价格是决定性因素，还是功能/能力差距？
- 如果有什么变化，他们会重新考虑吗？

**如何收集数据：**
- 销售团队在每笔成交后填写结构化表格（5 分钟，不可选择不填）。
- 对重要输单：和客户做 15 分钟"输单访谈"。"不会介意的——我们只是想学习。你们为什么选了别的方向？"如果你态度诚恳不带防御心，大多数客户会如实告诉你。
- 对赢单：在 Onboarding 时问。"你们还考虑了哪些工具？为什么选了我们？"

**每季度分析：**
- 按竞品统计胜率：如果对某个竞品输了 70% 以上，说明你在定位或产品上有差距需要补。
- 最常见的赢单原因：在市场和销售信息中加倍强化这些点。
- 最常见的输单原因：输入产品路线图和销售培训。
- 按客户类型统计胜率：你可能在中型市场对竞品 A 赢了 80%，但在企业市场只赢了 30%。这告诉你该集中力量在哪里。

### 竞品作战卡

竞品作战卡是一页纸的快速参考，帮助销售团队在竞争场景中实时应对。每个在成交中重叠超过 10% 的竞品都应该有一张。

**作战卡模板：**

**[竞品名称] 作战卡**
*最后更新：[日期]*

**概述**
- 他们是做什么的（2-3 句话）
- 成立于 [年份]，[融资阶段/金额]，[员工数]
- 目标市场：[卖给谁]

**他们的定价**
- [层级 1]：$X——包含 [功能]
- [层级 2]：$Y——包含 [功能]
- [Enterprise]：$Z——包含 [功能]

**他们的强项**
- [强项 1——对他们做得好的地方要诚实]
- [强项 2]
- [强项 3]

**他们的弱项（我们的优势）**
- [我们的优势 1 + 证据：客户原话、基准数据或他们缺少的能力]
- [我们的优势 2 + 证据]
- [我们的优势 3 + 证据]

**常见异议与应对**

| 异议 | 回应 |
|:---|:---|
| "[竞品] 有功能 X 而你们没有" | "确实——我们选择把资源投在 [Y] 上，因为 [原因]。对你们的场景来说，这更重要，原因是：[解释]" |
| "[竞品] 更便宜" | "他们的 [层级] 定价 $X，但不包含 [关键功能]。如果对比同等能力的套餐，我们相差不到 10%——而以下是你在我们这里能得到而在那里得不到的：[列表]" |
| "我们已经在用 [竞品] 了" | "可以理解——我们很多客户之前也用 [竞品]。他们告诉我们的主要原因是：[1、2、3]。这些和你的体验有相似的地方吗？" |

**埋地雷**
你的销售团队应该向客户提的问题，用来暴露竞品的弱点：
- "你了解过 [竞品] 在 [他们做得差的方面] 是怎么处理的吗？"
- "你在 [竞品的][已知薄弱领域——客服、稳定性、定价透明度] 方面的体验怎么样？"

**作战卡使用规则：**
- 每季度更新一次，或竞品有重大变动时更新。
- 永远不要在客户面前贬低竞品。陈述事实，不带情绪。"对于 [场景] 来说他们是个好产品。我们不同的地方在于 [具体能力]。"
- 每季度对销售团队进行作战卡培训。模拟竞争场景。
- 把它放在销售团队真正会用的地方（Notion、Guru、Highspot——不是一个他们永远找不到的 Google Doc）。
